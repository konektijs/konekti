# Chapter 2. Creating Your First Project with the CLI

Chapter 1에서 fluo가 어떤 생각으로 만들어졌는지 봤다면, 이제 그 철학을 실제 디렉터리와 명령어로 옮길 차례입니다. 이 장을 마치면 실행 가능한 FluoBlog 스타터가 준비되고, 생성된 파일들이 왜 그 자리에 있는지도 분명해집니다. 이후 아키텍처를 쌓기 전에, 안정적으로 돌아갈 기준선을 먼저 확보하는 셈이죠.

## Learning Objectives
- fluo CLI를 전역 설치하거나 패키지 러너로 실행하는 방법을 익힙니다.
- `fluo new` 명령으로 새 프로젝트를 스캐폴딩합니다.
- 생성된 프로젝트 구조와 각 디렉터리의 역할을 분석합니다.
- 로컬 개발에 사용되는 `package.json` 스크립트를 이해합니다.
- 첫 번째 FluoBlog 설정을 실행하며 확인합니다.
- 다음 장으로 넘어가기 전에 초기 문제 해결 습관을 익힙니다.

## Prerequisites
- Node.js 18 이상.
- npm, pnpm, yarn 중 하나의 패키지 매니저.
- Chapter 1 완료.
- 실습 중 계속 열어 둘 수 있는 터미널 세션.

## 2.1 Installing @fluojs/cli

첫 단계는 단순해 보이지만 중요합니다. 앞으로 나오는 예제들이 기대하는 작업 환경을 CLI가 먼저 잡아 주기 때문입니다.

fluo CLI는 처음 프로젝트를 만들 때의 진입점입니다. 모든 설정 파일을 손으로 하나씩 맞추는 대신, 프레임워크의 관례를 이미 알고 있는 도구에서 출발할 수 있습니다.

선호하는 패키지 매니저로 설치합니다.

```bash
# Using npm
npm install -g @fluojs/cli

# Using pnpm
pnpm add -g @fluojs/cli

# Using yarn
yarn global add @fluojs/cli
```

설치가 끝나면 명령이 실제로 인식되는지 확인해 봅니다.

```bash
fluo --version
```

버전 문자열이 출력되면 셸이 실행 파일을 제대로 찾고 있다는 뜻입니다.

### Global vs Local CLI

처음 시작할 때는 보통 가장 간단한 방법인 전역 설치를 선택합니다.

실무에서도 흔한 선택입니다. 다만 차이를 알고 있으면 나중에 팀 작업이나 CI를 만날 때 판단이 빨라집니다.

- **전역 설치**는 개인 개발 환경에서 편리합니다.
- `npx`, `pnpx` 같은 **로컬 실행**은 CI에서 버전을 고정하기 쉽습니다.
- 팀 단위 작업에서는 모두가 같은 버전을 쓰도록 로컬 실행을 선호하기도 합니다.

셋 다 사용할 수 있습니다. 핵심은 습관적으로 고르기보다, 지금 상황에 맞게 선택하는 것입니다.

### Verifying Your PATH

설치 후에도 `fluo --version`이 실패한다면 가장 흔한 원인은 셸 PATH입니다.

이럴 때는 아래 순서대로 확인하면 됩니다.

1. 패키지 매니저 설치가 정상 종료되었는지 확인합니다.
2. 새 터미널 창을 엽니다.
3. 버전 명령을 다시 실행합니다.
4. 여전히 실패하면 글로벌 바이너리 경로를 확인합니다.

이 문제는 fluo만의 특수한 문제가 아닙니다. CLI 도구를 처음 설치할 때 자주 만나는 기본 환경 이슈입니다.

### Why the Book Starts Here

이 책이 CLI부터 시작하는 이유는 설정 잡음을 줄여 주기 위해서입니다.

CLI가 없다면 처음 프로젝트를 만드는 시점부터 너무 많은 결정을 한꺼번에 내려야 합니다.

- 패키지 이름,
- TypeScript 설정,
- 스타터 소스 구조,
- 개발 스크립트,
- 런타임 부트스트랩 연결.

CLI는 신뢰할 수 있는 시작점을 제공합니다. 덕분에 사소한 설정에 발목 잡히기보다, fluo가 어떤 방식으로 앱을 구성하는지에 집중할 수 있죠.

### Troubleshooting global installs

전역 설치가 기대대로 작동하지 않는다면 보통 권한 문제나 Node.js 버전 불일치 때문입니다.

일부 시스템에서는 `sudo`를 써야 할 수도 있지만(npm에서는 권장하지 않음), 가능하면 `nvm`이나 `fnm` 같은 버전 매니저로 루트 권한 없이 Node.js 설치를 관리하는 편이 안전합니다.

pnpm을 사용한다면 `pnpm setup` 명령을 실행해서 글로벌 바이너리 디렉터리가 PATH에 추가되었는지 확인하세요.

### CLI as an Education Tool

CLI는 단순히 빠르게 시작하게 해 주는 도구만은 아닙니다. 구조를 보여 주는 학습용 기준점이기도 합니다. CLI가 어떤 의존성을 포함하는지, TypeScript를 어떻게 설정하는지, 파일을 어디에 두는지 살펴보는 것만으로도 프레임워크의 베스트 프랙티스를 파악할 수 있습니다.

`fluo new`를 실행할 때 설치되는 패키지들을 유심히 보세요. `@fluojs/core`, `@fluojs/http` 같은 모듈형 컴포넌트들을 보게 될 것입니다. 여기서 중요한 점은 fluo가 거대한 단일체(monolith) 프레임워크가 아니라, 필요한 만큼 조합해서 쓰는 특화된 도구들의 집합이라는 사실입니다.

### Staying Updated with the CLI

fluo 생태계는 빠르게 발전합니다. 새로운 기능, 보안 패치, 성능 개선 사항이 정기적으로 릴리스됩니다.

최신 기능과 버그 수정을 계속 활용하려면 주기적으로 CLI 업데이트를 확인하세요.

```bash
# Using npm
npm update -g @fluojs/cli

# Using pnpm
pnpm update -g @fluojs/cli
```

잘 관리된 CLI는 새 프로젝트를 시작할 때마다 최신 커뮤니티 지식과 프레임워크의 이점을 함께 가져오게 해 줍니다.

## 2.2 fluo new: Interactive Scaffolding

이제 명령을 사용할 수 있으니, 파일을 하나씩 손으로 만들 필요는 없습니다. 생성기가 깨끗한 시작점을 먼저 세우게 합니다.

CLI 설치가 끝났다면 책의 학습 프로젝트를 만듭니다.

```bash
fluo new fluo-blog
```

이 명령은 프로젝트 정보를 묻는 대화형 위저드를 시작합니다.

일반적인 질문은 다음과 같습니다.

1. **Description**: 프로젝트의 짧은 설명.
2. **Version**: 새 앱이라면 보통 `1.0.0`.
3. **Author**: 작성자 이름 또는 팀 이름.
4. **Package Manager**: 생성된 스크립트가 전제로 할 도구.

### What happens under the hood?

이 생성기는 단순히 폴더를 복사하는 도구가 아닙니다.

- 스타터 템플릿을 준비합니다.
- `package.json`에 프로젝트 메타데이터를 기록합니다.
- fluo 관례에 맞는 소스 트리를 만듭니다.
- 앱 실행에 필요한 패키지를 설치합니다.
- 곧바로 읽고 실행할 수 있는 구조를 남깁니다.

처음 fluo를 접하는 개발자에게 이 과정이 유용한 이유는 분명합니다. “프레임워크를 공부하고 싶다”는 의도를 “이제 실제로 열어 보고 실행할 수 있는 기준점이 있다”는 상태로 바꿔 주기 때문입니다.

### Naming the Project Carefully

프로젝트 디렉터리 이름은 생각보다 오래 따라다닙니다. 이후 개발 경험의 일부가 되죠.

`fluo-blog`는 학습용 이름으로 좋습니다.

- 읽기 쉽고,
- 입력하기 쉽고,
- 책의 흐름과도 바로 연결되기 때문입니다.

실제 프로젝트에서도 너무 길지 않으면서 목적이 보이는 이름을 고르는 편이 좋습니다.

### Interactive Flow vs Non-Interactive Flow

많은 스캐폴딩 도구는 대화형 흐름과 플래그 기반 자동화 흐름을 모두 지원합니다.

이 장에서는 대화형 흐름이 더 유익합니다. 각 설정 선택지가 눈앞에 드러나기 때문입니다. 나중에는 팀 템플릿이나 CI 재현성을 위해 자동화가 중요해지지만, 지금은 질문을 하나씩 보며 선택의 의미를 파악하는 편이 좋습니다.

### A Healthy Beginner Habit

생성기 출력 메시지를 그냥 지나치지 마세요.

처음 실행하면 마지막에 짧은 터미널 요약이 보입니다. 그 부분을 꼭 확인하세요.

보통 다음 정보를 알려 줍니다.

- 어떤 폴더가 생성되었는지,
- 어떤 의존성이 설치되었는지,
- 다음에 어떤 명령을 실행해야 하는지,
- 실행 전 주의할 점이 있는지.

이 확인 습관이 나중에 문제를 빨리 찾는 힘이 됩니다.

## 2.3 Analyzing the Project Structure

생성이 끝난 직후에는 바로 수정하지 않고 먼저 구조를 읽어 보는 편이 좋습니다.

먼저 새 디렉터리로 이동합니다.

```bash
cd fluo-blog
```

생성된 구조는 의도적으로 작습니다. 새로운 기능을 추가하기 전에 전체를 이해할 수 있도록 설계되어 있습니다.

```text
fluo-blog/
├── package.json              # Scripts and dependencies
├── pnpm-lock.yaml            # Locked dependency versions
├── tsconfig.json             # TypeScript configuration
└── src/                      # Application source
    ├── app.ts                # App assembly
    ├── hello.controller.ts   # Default HTTP route
    ├── hello.service.ts      # Route logic
    └── main.ts               # Bootstrap entrypoint
```

이 단계에서 각 파일은 처음 프로젝트를 열었을 때 자연스럽게 생기는 질문에 답해 줍니다.

- 앱은 어디서 시작되는가?
- 기본 앱은 어디서 조립되는가?
- 첫 HTTP 응답은 어떤 파일이 담당하는가?
- 가장 자주 실행할 스크립트는 무엇인가?

### src/main.ts

`src/main.ts`는 가장 작은 파일처럼 보여도 책임은 매우 큽니다.

이 파일이 애플리케이션을 부트스트랩합니다.

다시 말해, 프레임워크 설정이 실제 서버 시작으로 바뀌는 지점입니다. 이후 환경 설정, 로깅, 플랫폼 옵션이 들어오면 이 파일은 더 중요한 진입점이 됩니다.

### src/app.ts

`src/app.ts`는 기본 스타터의 조립 지점입니다.

초기에는 작아 보여도 실제 구조를 파악하는 데 도움이 됩니다. 이 파일을 보면 스타터가 어떤 컨트롤러와 서비스를 기본으로 묶는지 바로 읽을 수 있습니다.

FluoBlog가 커질수록 이 조립 지점은 이후 장에서 더 많은 구성을 담는 출발점이 됩니다.

### package.json

처음에는 `package.json`을 의존성 목록으로만 보기 쉽습니다. 하지만 실제로는 프로젝트의 명령 표면이기도 합니다.

이 파일을 통해 다음을 알 수 있습니다.

- 로컬 개발을 어떻게 시작하는지,
- 프로덕션 빌드는 어떻게 하는지,
- 컴파일된 출력을 어떻게 실행하는지,
- 그리고 린트나 테스트를 어떻게 연결하는지.

### src/hello.controller.ts와 src/hello.service.ts

이 두 파일은 기본 스타터가 실제로 응답 가능한 앱이라는 사실을 가장 빨리 보여 줍니다.

컨트롤러는 기본 라우트를 노출하고, 서비스는 그 라우트가 반환할 값을 제공합니다. 처음에는 이 한 쌍만 읽어도 요청이 어디로 들어와 어디서 값이 만들어지는지 바로 추적할 수 있습니다.

### Reading Before Editing

바로 수정에 들어가기 전에 생성된 파일을 몇 분만 읽어 보세요.

그 짧은 멈춤이 다음 세 가지를 분리해 줍니다.

1. 프레임워크가 기본으로 주는 것.
2. 이후 장에서 직접 바꿀 것.
3. 어떤 파일이 어떤 책임을 가지는지.

### Why the starter includes hello files

`hello.controller.ts`와 `hello.service.ts`는 첫 실행 확인을 위한 가장 작은 예제입니다.

복잡한 도메인 코드를 넣기 전에, 기본 라우트와 응답이 어디서 오는지 눈으로 확인할 수 있게 해 줍니다. 이 출발점 덕분에 이후 장에서 코드를 바꿨을 때 무엇이 새로 추가된 것인지 더 쉽게 구분할 수 있습니다.

### Exploring the `node_modules` Folder (Briefly)

처음에는 보통 `node_modules`를 무시하라고 말하지만, 가끔 들여다보는 것도 구조 이해에 도움이 됩니다.

`@fluojs` 네임스페이스 아래에 핵심 프레임워크 로직들이 있는 것을 볼 수 있습니다. 이 패키지들이 얼마나 작은지 주목해 보세요. 방대한 외부 라이브러리에 기대기보다 네이티브 언어 기능을 활용하는 fluo의 "표준 우선(Standard-First)" 철학이 여기에도 드러납니다.

프레임워크 자체가 잘 조직된 TypeScript 코드의 집합일 뿐이라는 사실을 이해하면, 처음 느꼈던 "마법" 같은 느낌이 걷히고 코드가 더 친숙하게 다가옵니다.

### Why TypeScript?

fluo는 TypeScript를 기반으로 구축되었으며, 생성된 프로젝트도 이를 반영합니다.

왜 단순한 JavaScript를 쓰지 않는지 궁금할 수 있습니다. TypeScript는 다음을 제공합니다.

- **자동 완성**: VS Code 같은 IDE가 적절한 데코레이터와 매개변수를 찾도록 도와줍니다.
- **조기 오류 발견**: 많은 버그가 앱을 실행하기 전, 코드를 작성하는 동안 발견됩니다.
- **코드로 표현된 문서**: 타입 자체가 최신 상태를 유지하는 문서 역할을 합니다.

CLI는 fluo에 최적화된 설정으로 `tsconfig.json`을 대신 구성해 줍니다. 프로젝트가 커질수록 컴파일러가 뒤에서 일관성을 지켜 준다는 점이 큰 장점으로 드러납니다.

## 2.4 Understanding package.json Scripts

디렉터리 구조를 살펴봤다면, 이제는 이 프로젝트 안에서 매일 어떤 명령으로 작업하게 되는지 이해할 차례입니다.

생성된 프로젝트에는 초기 개발 워크플로 전체를 받쳐 주는 소수의 스크립트가 보통 포함됩니다.

- **`dev`**: 빠른 피드백을 위한 개발 모드 실행.
- **`build`**: TypeScript를 프로덕션용 출력으로 컴파일.
- **`start`**: 빌드된 결과를 실행.
- **`lint`**: 스타터가 제공하는 경우 코드 품질 점검.

### Why `dev` matters most right now

Part 0에서는 `dev`가 가장 자주 쓰이는 명령입니다.

피드백 루프를 짧게 만들어 주기 때문입니다.

- 코드를 수정하고,
- 파일을 저장하고,
- 결과를 확인하고,
- 빠르게 반복합니다.

모듈, 데코레이터, 라우팅을 익히는 단계에서는 이 속도가 정말 중요합니다.

### Why `build` still matters early

배포를 아직 다루지 않더라도 `build`는 중요합니다. 소스가 프로덕션 가능한 상태로 깨끗하게 컴파일되는지를 검증해 주기 때문입니다.

권장 루틴은 다음과 같습니다.

1. 코드를 바꿀 때는 `dev`를 사용합니다.
2. 컴파일 가능 상태를 확인하고 싶을 때는 `build`를 실행합니다.

### `start` is not the same as `dev`

이 차이는 지금 배워 두는 편이 좋습니다.

- `dev`는 반복 작업에 최적화됩니다.
- `start`는 빌드 결과를 실행합니다.

`dev`에서는 되는데 `start`에서 안 된다면, 보통 빌드나 환경 가정 중 하나가 잘못된 것입니다.

### Scripts as Team Language

공유 스크립트는 팀의 혼란도 줄여 줍니다.

모두가 같은 명령 이름을 쓰면 문서는 짧아지고 온보딩은 쉬워집니다. 좋은 스타터 템플릿은 첫 장부터 그 공통 언어를 제공합니다.

## 2.5 FluoBlog: Initial Scaffolding and First Run

파일을 읽어 봤다면 이제 이 스캐폴드가 단지 정리만 잘된 상태가 아니라 실제로 살아 있는 프로젝트인지 확인할 차례입니다.

프로젝트를 실행합니다.

```bash
pnpm dev
```

설치 단계에서 npm이나 yarn을 골랐다면 그에 맞는 동등 명령을 사용하세요.

처음 실행하면 이런 흐름으로 보입니다. 다음과 비슷한 시작 로그를 기대할 수 있습니다.

```text
[Fluo] Starting application...
[Fluo] AppModule initialized.
[Fluo] HTTP Server listening on port 3000.
```

이 로그는 중요한 사실을 알려 줍니다. 프레임워크가 루트 모듈을 로드했고, 부트스트랩을 마쳤으며, HTTP 서버를 바인딩했다는 뜻입니다.

### Verifying the Response

서버가 뜬 뒤에는 실제로 한 번 요청을 보내 봅니다.

```bash
curl http://localhost:3000/health
curl http://localhost:3000/hello
```

`{"status":"ok"}`와 `{"message":"Hello, World!"}`가 보이면 기본 헬스 체크와 스타터 라우트가 현재 계약대로 함께 정상 동작하고 있다는 의미입니다.

### What You Actually Verified

처음 실행에 성공하면 그 요청이 무엇을 증명하는지 과소평가하기 쉽습니다.

- CLI가 유효한 프로젝트를 생성했습니다.
- 의존성이 정상적으로 설치되었습니다.
- TypeScript가 개발 모드에서 기대한 방식으로 컴파일 또는 변환되었습니다.
- 런타임 어댑터가 정상적으로 시작되었습니다.
- 기본 검증 라우트인 `/health`와 `/hello`에 접근할 수 있습니다.

이 정도만 확인해도 이후 장을 위한 강한 기준선이 마련됩니다.

### Common Beginner Issues

앱이 시작되지 않는다면 다음 원인을 먼저 확인하세요.

1. 의존성 설치가 중간에 실패했는가?
2. 이미 다른 프로세스가 `3000` 포트를 사용 중인가?
3. 잘못된 패키지 매니저 명령을 사용했는가?
4. Node.js 버전이 스타터 기대보다 낮은가?

대부분의 경우 터미널 출력만 읽어도 해결 방향이 보입니다.

### Why We Start with the Default App

일부 학습자는 스타터를 바로 커스터마이즈하고 싶어 합니다.

그 흐름도 가능하지만, 기본 앱을 먼저 확인한 뒤 수정하는 편이 안전합니다.

기본 앱은 “정상 동작하는 기준 상태”를 제공하기 때문에 가치가 큽니다. 이후 모듈이나 컨트롤러를 추가했을 때 새 오류가 생기면, 그 문제가 초기 스캐폴드가 아니라 자신의 변경에서 왔다는 사실을 더 쉽게 알 수 있습니다.

### A Short Reflection Before Chapter 3

이 장이 끝났을 때의 목표는 단순히 “생성기를 돌려 봤다”가 아닙니다.

더 강한 목표는 다음입니다.

- 프로젝트가 어떻게 시작되었는지 안다.
- 핵심 파일의 역할을 안다.
- 앱을 어떻게 실행하는지 안다.
- 건강한 스타터 상태가 어떤 모습인지 안다.

이 맥락이 있으면 다음 장의 모듈, 프로바이더, 컨트롤러가 훨씬 쉽게 이해됩니다.

### Developing with a Plan

처음 프로젝트를 다룰 때도 계획을 가지고 시작하는 편이 좋습니다. 코드로 뛰어들기 전에 무엇을 달성하고 싶은지 먼저 정리하세요.

1. **목표 이해**: 어떤 문제를 해결하려고 하는가?
2. **단계 나누기**: 그 목표에 도달하기 위해 필요한 작은 단계들은 무엇인가?
3. **기록하기**: 지금 여러분이 읽고 있는 것과 같은 간단한 체크리스트만으로도 집중력을 유지할 수 있습니다.

이 책의 FluoBlog 프로젝트도 이 패턴을 따를 것입니다. 각 장마다 명확한 목표를 정하고, 그 목표를 작은 단계로 나눠 구현합니다. 이런 훈련된 접근은 단순히 "되는 대로 코딩하는" 방식보다 훨씬 빠르고 안정적으로 fluo를 익히게 해 줍니다.

### The Joy of the First Green Log

터미널에 "Application started successfully"라는 첫 로그가 찍힐 때는 분명한 성취감이 있습니다.

그 순간은 추상적인 개념이 실제로 동작하는 프로그램이 되는 순간입니다. 그냥 지나치기보다, 현대적이고 고성능인 백엔드 환경을 성공적으로 세웠다는 사실을 확인해 두는 편이 좋습니다.

이제 표준, 성능, 그리고 깨끗한 아키텍처를 중요하게 여기는 fluo의 기본 흐름을 직접 다룰 준비가 되었습니다.

### Learning from the Logs

fluo가 출력하는 로그는 단순한 상태 업데이트가 아니라 진단 도구입니다.

오류가 발생하면 스택 트레이스(stack trace)를 먼저 읽어 보세요. 보통 어떤 파일의 몇 번째 줄에서 문제가 생겼는지 정확히 알려 줍니다. 로그를 읽고 이해하는 습관은 백엔드 개발자로서 오래 도움이 되는 기본기입니다.

CLI와 로그를 통해 프레임워크와 더 많이 상호작용할수록 fluo는 더 직관적으로 느껴질 것입니다. 새로운 언어를 배우는 것과 비슷합니다. 처음에는 프레임워크가 제시하는 패턴을 따라가지만, 곧 프로젝트 상황에 맞게 응용할 수 있게 됩니다.

## Summary
- fluo CLI는 처음 fluo를 접하는 개발자에게 일관된 출발점을 제공합니다.
- `fluo new`는 단순한 폴더가 아니라 파일과 관례를 함께 스캐폴딩합니다.
- 생성된 소스 트리는 부트스트랩, 모듈 구성, 프로젝트 메타데이터의 위치를 알려 줍니다.
- `dev`, `build`, `start`는 개발 생명주기의 서로 다른 단계를 담당합니다.
- 첫 성공 요청은 스캐폴드가 실제로 동작한다는 강한 증거입니다.

이 장의 진짜 성과는 단순히 생성기를 한 번 실행했다는 데 있지 않습니다. 이제 여러분은 다음 장의 아키텍처 개념을 올릴 수 있는, 정상 동작하는 출발 상태를 손에 넣었습니다.
