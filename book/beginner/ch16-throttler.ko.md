# Chapter 16. Rate Limiting and Throttling

이 장은 FluoBlog API를 남용과 과도한 트래픽으로부터 보호하기 위한 속도 제한 전략을 설명합니다. Chapter 15가 인증과 인가로 접근을 제어했다면, 이 장은 요청 빈도 자체를 통제해 서비스 안정성을 높입니다.

## Learning Objectives
- API 보안과 가용성을 위한 속도 제한(Rate Limiting)의 중요성을 이해합니다.
- 전역 및 라우트별 보호를 위해 `ThrottlerModule`을 설정합니다.
- 인메모리와 Redis 저장소 전략을 비교합니다.
- 사용자 ID나 IP 주소를 기준으로 제한 규칙을 조정합니다.
- `429 Too Many Requests` 응답을 처리하는 방법을 익힙니다.
- 버스트 제어와 슬라이딩 윈도우 같은 패턴을 살펴봅니다.
- 분산 환경에 맞는 속도 제한 아키텍처를 설계합니다.

## Prerequisites
- Chapter 5와 Chapter 15 완료.
- HTTP 요청 흐름과 상태 코드에 대한 기초 이해.
- 프록시 또는 로드 밸런서 뒤에서 동작하는 API 환경에 대한 기본 개념 이해.

## 16.1 Why Throttle Your API?
공개된 API에서는 모든 트래픽을 선의로 볼 수 없습니다. 적절한 보호 조치가 없다면 서버는 무차별 대입 로그인 시도, 악의적인 스크래퍼, 혹은 클라이언트 측 애플리케이션의 잘못 작성된 루프 때문에 빠르게 마비될 수 있습니다. 이는 **서비스 거부(DoS)** 상태로 이어져, 서버가 불필요한 요청을 처리하느라 정당한 사용자가 사이트에 접근하지 못하게 만듭니다.

속도 제한, 즉 **스로틀링(Throttling)**은 네트워크 인터페이스에서 송수신되는 트래픽의 속도를 제어하는 기법입니다. 이를 통해 단일 사용자나 특정 IP 주소가 시스템의 리소스를 독점하지 못하도록 보장합니다. 스로틀링을 구현하면 모든 사용자에게 공정한 사용 기회를 보장할 수 있으며, 악의적이거나 실수로 인한 트래픽 급증으로 인해 인프라 비용이 통제 불능 상태로 치솟는 것을 방지할 수 있습니다. 이는 상용 수준의 API 아키텍처에서 필수적인 구성 요소입니다. 스로틀링은 예상치 못한 급증 시 데이터베이스나 내부 마이크로서비스와 같은 다운스트림 서비스가 과부하되는 것을 방지하는 중요한 안전 밸브 역할을 합니다.

### 16.1.1 The Security Aspect
스로틀링은 무차별 대입 공격(Brute-force)에 대한 일차적인 방어선입니다. 공격자가 사용자의 비밀번호를 알아내려 할 때 초당 수천 개의 조합을 시도할 수 있습니다. IP당 분당 5회의 로그인 시도와 같은 간단한 제한만으로도 공격 비용은 크게 올라갑니다. 마찬가지로, API 스크래퍼가 단 몇 분 만에 데이터베이스 전체를 긁어가는 것을 제한해 지적 재산과 데이터 프라이버시를 보호할 수 있습니다. 상호작용의 속도를 제한하면 공격의 경제성이 바뀌고, 많은 공격은 지속하기에 너무 느리고 비싸집니다.

### 16.1.2 Preventing "Friendly" DoS
항상 악의적인 행위자만이 문제를 일으키는 것은 아닙니다. 때로는 정당한 클라이언트 개발자의 실수로 인해 "우호적인 DoS(Friendly DoS)"가 발생하기도 합니다. 예를 들어, 적절한 의존성 배열 없이 `useEffect` 훅 안에 API 호출을 배치하여 무한 루프 요청이 발생하는 경우입니다. 스로틀링은 이러한 정직한 실수로부터 백엔드를 보호하여, 특정 클라이언트 버전을 수동으로 개입하거나 차단하지 않고도 서비스 안정성을 유지할 수 있게 해줍니다. 스로틀링된 시스템의 이러한 자가 치유 특성은 복원력 있는 백엔드 설계의 초석입니다.

이 차이는 초기 프로젝트에서도 바로 중요해집니다. 로그인 경로는 반복적인 비밀번호 추측 시도를 받을 수 있고, 평범한 엔드포인트도 버그가 있는 클라이언트 스크립트 때문에 과도하게 호출될 수 있습니다. 이때 **속도 제한(Rate Limiting)** 또는 Throttling이 필요합니다. 이는 클라이언트가 특정 시간 내에 보낼 수 있는 요청 수를 제한하여 그런 압력을 실용적으로 줄여 줍니다.

### 16.1.3 Cost Management in the Cloud
현대적인 클라우드 환경에서 많은 서비스는 요청 수나 소비된 리소스를 기준으로 요금이 청구됩니다. 스로틀링되지 않은 API는 공격자(또는 버그)가 회사의 은행 계좌를 고갈시키도록 내버려 두는 백지 수표와 같습니다. 엄격한 제한을 적용함으로써 인프라에 대한 예측 가능한 비용 구조를 만들 수 있습니다. 이는 API 호출이 타사 AI 모델 추론, 복잡한 암호화 서명 또는 집중적인 데이터 처리 작업과 같이 비용이 많이 드는 다운스트림 작업을 트리거할 때 특히 중요합니다.

## 16.2 Introduction to @fluojs/throttler
Fluo는 `fluo` 요청 생명주기와 통합되는 `@fluojs/throttler` 패키지를 제공합니다. 이 패키지는 비동기 설계를 사용해 요청 오버헤드를 낮추면서 필요한 보호 기능을 제공합니다. 다양한 런타임에서 작동하도록 설계되었으므로, 속도 제한 로직도 다른 `fluo` 애플리케이션 코드처럼 이식성을 유지합니다.

`fluo`는 데코레이터 기반의 쉬운 속도 제한을 위해 `@fluojs/throttler` 패키지를 제공합니다. 덕분에 애플리케이션 구조를 바꾸지 않고도 필요한 경로 가까이에 제한 규칙을 둘 수 있습니다.

### 16.2.1 Key Concepts: Limit, TTL, and Tracker
- **Limit**: 특정 시간 범위 내에 허용되는 최대 요청 수입니다.
- **TTL (Time To Live)**: 시간 범위의 지속 시간(초 단위)입니다.
- **Tracker**: 고유한 요청자를 식별하는 로직입니다. 기본값은 IP 주소이지만, 사용자 ID나 API 키가 될 수도 있습니다.

이 세 가지 개념을 결합하면 애플리케이션의 각 부분에 맞는 정교한 트래픽 제어 정책을 만들 수 있습니다. 예를 들어, 공개 데이터 조회에는 분당 100회를 허용하고 비밀번호 변경과 같은 민감한 작업에는 분당 5회만 허용할 수 있습니다.

### 16.2.2 The Throttler Guard
`@fluojs/throttler` 패키지는 전역, 컨트롤러 또는 개별 라우트에 적용할 수 있는 `ThrottlerGuard`를 제공합니다. 이 가드는 저장소를 확인하고, 요청 횟수를 증가시키며, 제한 초과 여부를 결정하는 로직을 관리합니다. Fluo의 DI 시스템에 통합되어 있으므로, 이 가드를 확장해 복잡한 추적 로직에 필요한 다른 서비스를 주입할 수도 있습니다. 가드는 정책을 실제 요청 경계에서 집행하는 장치이며, 애플리케이션 로직의 가장 앞단에서 요청을 가로챕니다.

Fluo에서 가드는 인터셉터 체인보다 먼저 실행되고, 그 다음에 파이프와 핸들러가 이어집니다. 즉, 요청이 스로틀링되면 데이터 유효성 검사나 비즈니스 로직 처리에 필요한 리소스를 전혀 소비하지 않습니다. 이러한 "조기 실패(Fail Fast)" 접근 방식은 DDoS 공격 동안 높은 가용성을 유지하는 데 필수적입니다. 입구에서 악의적인 트래픽을 거부함으로써 소중한 CPU 사이클과 메모리가 정당한 사용자를 위해 예약되도록 보장합니다.

### 16.2.3 Response Headers
전문적인 API는 항상 클라이언트에게 현재의 속도 제한 상태를 알려야 합니다. 현재 계약에서 속도 제한 응답은 사용자가 얼마나 기다려야 하는지 알려주는 `Retry-After` 헤더를 중심으로 전달됩니다.

이 헤더를 제공하면 잘 설계된 클라이언트가 재시도 시점을 맞춰 스스로 속도를 조절할 수 있어, 사용자가 429 에러를 반복해서 겪는 일을 줄이고 전반적으로 더 나은 개발자 생태계를 조성할 수 있습니다. 클라이언트와 서버 사이의 이러한 협력적 동작은 확장 가능한 분산 시스템의 기초입니다.

### 16.2.4 Asynchronous Throttling Logic
데이터베이스 확인을 기다리는 동안 메인 실행 스레드를 차단할 수 있는 전통적인 미들웨어와 달리, `@fluojs/throttler`는 Fluo 고유의 비동기 실행 모델 위에 구축되었습니다. 단순한 메모리 저장소를 사용하든 고성능 Redis 클러스터를 사용하든, 스로틀러는 절대 이벤트 루프를 차단하지 않습니다. 이를 통해 수백 개의 서로 다른 트래커 키에 대해 수백 개의 동시 요청을 처리할 때도 API의 응답성이 유지됩니다.

스로틀러는 또한 "경쟁 상태(Race Conditions)"의 위험을 최소화하기 위해 고급 동시성 패턴을 활용합니다. 트래픽이 많은 환경에서는 동일한 사용자의 여러 요청이 서로 다른 서버 인스턴스에 동시에 도달할 수 있습니다. Fluo의 Redis 저장소 프로바이더는 원자적 증가(Atomic Increments)와 Lua 스크립트를 사용하여 가장 격렬한 트래픽 상황에서도 모든 요청이 정확하게 계산되도록 보장합니다. 이러한 정밀함이 Fluo 스로틀러를 금융 서비스나 기타 높은 신뢰성이 요구되는 환경에 적합하게 만듭니다.

## 16.3 Basic Configuration

첫 단계는 애플리케이션 전체에 적용할 기본 제한을 정하는 것입니다. 루트 모듈에 `ThrottlerModule`을 등록합니다.

```typescript
import { Module } from '@fluojs/core';
import { ThrottlerModule } from '@fluojs/throttler';

@Module({
  imports: [
    ThrottlerModule.forRoot({
      // 기본적으로 60초마다 10회의 요청 허용
      limit: 10,
      ttl: 60,
    }),
  ],
})
export class AppModule {}
```

### 16.3.1 Global Throttling
루트 수준에서 모듈을 설정하면 애플리케이션 전체에 대한 기본 보호 수준이 수립됩니다. 이것이 첫 번째 방어선입니다. 모든 들어오는 요청은 별도의 오버라이드가 없는 한 전역 제한을 기준으로 추적됩니다. 이러한 "기본 보안 적용(secure by default)" 태세는 Fluo의 철학이 반영된 것으로, 명시적으로 보호하는 것을 잊은 라우트도 일정 수준의 남용으로부터 보호합니다.

이 설정은 `ThrottlerGuard`를 연결한 라우트에 대해 60초마다 10회 요청이라는 기본 제한을 정의합니다. 이후 민감한 엔드포인트에 더 엄격한 규칙을 얹기 전에, FluoBlog 전체에 기본 방어선을 세우는 셈입니다.

전역 스로틀링은 **로드 밸런서 통합**과 결합될 때 특히 효과적입니다. 애플리케이션이 Nginx, HAProxy 또는 클라우드 네이티브 로드 밸런서(예: AWS ALB)와 같은 프록시 뒤에서 실행되는 경우, 실제 클라이언트 IP를 식별하기 위해 `X-Forwarded-For` 헤더가 올바르게 파싱되도록 해야 합니다. 이 설정이 없으면 전역 스로틀러가 모든 트래픽이 프록시 자체에서 발생하는 것으로 간주하여 모든 사용자를 실수로 "전역 블랙리스트"에 추가하는 결과를 초래할 수 있습니다. Fluo에서는 플랫폼 설정에서 `trustProxyHeaders: true`를 활성화하면 `ThrottlerGuard`가 추적 로직에 필요한 정확한 IP 주소를 전달받을 수 있습니다.

단순한 IP 추적을 넘어, 전역 스로틀링은 **시스템 총계 제한(Aggregate System Limits)**을 강제하는 데 사용될 수 있습니다. 예를 들어, 어떤 사용자가 요청을 보내든 상관없이 전체 API 클러스터에 대해 분당 10,000개의 글로벌 제한을 설정하여 데이터베이스 고갈을 방지할 수 있습니다. 이러한 상위 수준의 리소스 관리는 예상치 못한 트래픽 급증이나 폭발적인 성장에 직면했을 때 인프라 안정성을 유지하는 데 필수적입니다. 이러한 "가드레일(Guardrails)"을 설정함으로써 시스템이 압박 속에서 무너지지 않고 우아하게 실패하도록 보장할 수 있습니다.

전역 규칙이 생기면 이제 경로 성격에 맞게 조정할 수 있습니다. 글로벌 설정을 재정의하거나 특정 컨트롤러나 메서드에 대해 속도 제한을 건너뛸 수 있습니다.

### 16.3.2 Multiple Throttling Definitions
현대적인 애플리케이션은 종종 여러 계층의 스로틀링을 필요로 합니다. 예를 들어 "버스트(burst)" 제한(예: 초당 10회)과 "지속(sustained)" 제한(예: 시간당 1000회)을 동시에 두고 싶을 수 있습니다. 현재 `@fluojs/throttler` 계약은 하나의 모듈 수준 기본 정책과 클래스/메서드 수준 `@Throttle({ ttl, limit })` 오버라이드를 제공합니다. 여러 시간 범위를 조합해야 한다면 한 계층은 애플리케이션 수준 HTTP rate-limit middleware로 두고 handler별 제한은 `ThrottlerGuard`로 적용하거나, 추가 정책을 명시적으로 인코딩하는 커스텀 `ThrottlerStore`/guard wrapper를 제공하십시오.

```typescript
ThrottlerModule.forRoot({
  ttl: 60,
  limit: 100,
})
```

`ThrottlerGuard`는 라우트 매칭 이후 guard 단계에서 이 활성 정책을 강제합니다. 같은 handler에 버스트 제한과 지속 제한을 모두 적용해야 한다면, 명명된 정의가 내장되어 있다고 가정하지 말고 명시적인 store 또는 guard 확장으로 모델링하십시오.

### 16.3.3 Throttling by Request Type
시간 기반 제한 외에도 HTTP 메서드나 다른 요청 속성을 기반으로 스로틀링을 수행할 수 있습니다. 예를 들어, 캐시에서 제공될 수 있는 `GET` 요청에 비해 데이터베이스 쓰기나 고비용 처리가 수반되는 `POST` 및 `PUT` 요청에는 훨씬 낮은 제한을 적용할 수 있습니다. 각 요청 유형에 route-specific `@Throttle()` 오버라이드를 적용하면, 각 상호작용과 관련된 특정 비용에 맞춰 보안 태세를 조정할 수 있습니다.

이러한 세밀한 제어는 `@Throttle()` 데코레이터로도 확장되어, 특정 라우트에 사용할 `{ ttl, limit }` 정책을 지정할 수 있습니다. 이를 통해 가장 일반적인 기본값을 모듈에 중앙 집중화하는 동시에, 필요한 곳에는 특화된 규칙을 적용할 수 있는 유연성을 유지할 수 있습니다. 중앙 거버넌스와 로컬 조정을 함께 가져가는 방식입니다.

### 16.3.4 The Throttler Decorator Structure
`@Throttle()` 데코레이터를 사용할 때는 `ThrottlerGuard`가 실행 중에 참조할 메타데이터를 전달합니다. 이 메타데이터는 해당 스코프에 대해 모듈 수준의 기본값을 재정의합니다. Fluo의 데코레이터 시스템은 이러한 재정의가 타입 안전하고 시작 시 검증되도록 보장하여, 음수 제한이나 0초 TTL과 같은 일반적인 설정 오류를 방지합니다.

데코레이터는 클래스(컨트롤러)와 메서드 모두에 적용할 수 있습니다. 클래스에 적용하면 해당 클래스 내의 모든 메서드에 영향을 미칩니다. 이는 특정 속도 제한 정책을 공유해야 하는 관련 엔드포인트들을 그룹화할 때 유용합니다. 만약 메서드에도 `@Throttle()` 데코레이터가 있다면, 클래스 수준의 데코레이터보다 우선순위를 가집니다. 이러한 계층적 재정의를 통해 트래픽 제어 전략을 매우 정밀하게 구성할 수 있습니다.

## 16.4 Storage Providers: Memory vs. Redis
스로틀러는 각 트래커의 요청 횟수를 저장할 공간이 필요합니다. 적절한 저장소 프로바이더를 선택하는 것은 속도 제한의 성능과 정확성 모두에 영향을 미치는 중요한 결정입니다.

- **인메모리 (기본값)**: 빠르고 설정이 필요 없습니다. 로컬 개발, 테스트 및 단일 서버 인스턴스에서 실행되는 소규모 애플리케이션에 이상적입니다. 하지만 서버가 재시작되면 데이터가 사라지며, 여러 서버 인스턴스를 사용하는 환경(로드 밸런싱)에서는 작동하지 않습니다. 인스턴스가 두 개라면 사용자는 이론적으로 제한의 두 배를 사용할 수 있습니다.
- **Redis**: 분산 시스템을 위한 프로덕션 표준입니다. 서버 재시작 후에도 횟수를 유지하며, 여러 서버 인스턴스가 동일한 스로틀링 상태를 공유할 수 있게 해줍니다. 키 만료 및 원자적 연산에 대한 Redis의 네이티브 지원은 속도 제한 구현에 잘 맞습니다.

### 16.4.1 The Role of the Storage Provider Interface
Fluo은 모든 store adapter가 구현해야 하는 표준 `ThrottlerStore` 인터페이스를 정의합니다. 이러한 추상화를 통해 가드나 데코레이터 로직을 변경하지 않고도 저장소 백엔드를 교체할 수 있습니다. Redis에서 Memcached나 DynamoDB와 같은 다른 분산 저장소로 이동하기로 결정했다면, 동일한 `consume(...)` 계약을 가진 새 store를 제공하면 됩니다.

```typescript
export interface ThrottlerStore {
  consume(key: string, input: ThrottlerConsumeInput): ThrottlerStoreEntry | Promise<ThrottlerStoreEntry>;
}
```

이 인터페이스 기반 설계는 Fluo의 "표준 우선(Standard-First)" 접근 방식의 핵심으로, 특정 벤더나 기술에 종속되지 않도록 보장합니다. 또한 다양한 속도 제한 시나리오(예: 저장소 가득 참, 느린 연결)를 시뮬레이션하기 위해 저장소 프로바이더를 쉽게 모킹(mock)할 수 있어 유닛 테스트 작성이 쉬워집니다.

### 16.4.2 Configuring Redis for High Availability
프로덕션 환경에서 Redis를 사용할 때는 관리형 서비스(AWS ElastiCache 등)나 고가용성 클러스터 설정을 사용하는 것이 강력히 권장됩니다. 스로틀러는 매 요청마다 Redis에 의존하기 때문에 Redis 클러스터의 가용성이 API의 가용성에 직접적인 영향을 미칩니다.

```typescript
import { ThrottlerModule, RedisThrottlerStore } from '@fluojs/throttler';

const redisStore = new RedisThrottlerStore(redisClient);

ThrottlerModule.forRoot({
  limit: 100,
  ttl: 60,
  store: redisStore,
});
```

구성된 store가 reject하거나 잘못된 counter를 반환하면 `ThrottlerGuard`는 이를 `429`로 변환하지 않고 요청 실패로 전파합니다. `Retry-After`는 실제 제한 초과에만 기록됩니다. bootstrap 중 비핵심 Redis readiness를 degraded로 보고하려면 platform status helper를 사용하고, fail closed·memory fallback·비핵심 throttling layer 임시 비활성화 중 어떤 정책을 사용할지는 애플리케이션 경계에서 결정하십시오.

## 16.5 Route-Specific Throttling
전역 제한이 전반적인 보호에는 좋지만, 특정 라우트는 더 엄격하거나 느슨한 제약이 필요할 때가 많습니다. `@Throttle()` 데코레이터를 사용하여 전역 설정을 재정의하십시오.

```typescript
import { Controller, Post } from '@fluojs/http';
import { Throttle } from '@fluojs/throttler';

@Controller('auth')
export class AuthController {
  @Post('login')
  @Throttle({ limit: 5, ttl: 60 }) // 로그인에는 엄격한 제한 적용
  async login() {
    // ...
  }

  @Post('signup')
  @Throttle({ limit: 3, ttl: 3600 }) // 매우 엄격: 시간당 3회 가입
  async signup() {
    // ...
  }
}
```

### 16.5.1 Overriding for Higher Performance
반대로 실시간 분석 하트비트나 입력 시 실시간 검색과 같이 빈번한 호출이 발생하는 라우트의 경우 제한을 늘리고 싶을 수도 있습니다. `@Throttle()` 데코레이터는 민감한 엔드포인트의 보안을 타협하지 않으면서도 애플리케이션의 응답성을 조정할 수 있는 유연성을 제공합니다. 이러한 세밀한 제어 덕분에 Fluo 애플리케이션은 빠르면서도 안전할 수 있습니다. 이는 모든 API 엔드포인트가 동일하지 않으며 보안은 위험에 비례해야 함을 인정하는 것입니다.

### 16.5.2 Skipping Throttling
어떤 경우에는 특정 라우트나 컨트롤러를 스로틀링에서 완전히 제외하고 싶을 수도 있습니다. 예를 들어 내부 로드 밸런서가 사용하는 상태 확인 엔드포인트입니다. `@SkipThrottle()` 데코레이터는 신뢰할 수 있는 내부 트래픽에 대해 전역 스로틀링 로직에서 쉽게 제외할 수 있는 방법을 제공합니다. 이는 내부 인프라가 일상적인 작업 중에 실수로 보안 블록을 트리거하는 것을 방지합니다.

또한 `@SkipThrottle()`을 컨트롤러 수준에서 적용하여 라우트 그룹 전체를 제외하면서 다른 컨트롤러는 계속 보호할 수 있습니다. 이러한 정밀한 제어를 통해 Fluo는 서로 다른 클라이언트(예: 모바일 앱 vs 서버 측 통합)가 매우 다른 트래픽 패턴과 신뢰 수준을 갖는 복잡하고 이질적인 환경을 지원할 수 있습니다.

### 16.5.3 Dynamic Throttle Limits
더욱 진보된 시나리오에서는 현재 시스템 부하나 사용자 상태에 따라 스로틀 제한을 동적으로 조정해야 할 수도 있습니다. 지원되는 확장 지점은 configuration입니다. 커스텀 요청자 식별에는 `keyGenerator`를 제공하고, 정책별 카운팅에는 커스텀 `ThrottlerStore`를 제공하거나, 자체 정책 입력을 해석한 뒤 `ThrottlerGuard`를 호출하는 애플리케이션 guard wrapper를 작성하십시오.

예를 들어, "시스템 상태" 서비스를 확인하여 데이터베이스 지연 시간이 특정 임계값을 초과하는 경우 모든 제한을 강화할 수 있습니다. 또는 사용자의 구독 상태를 실시간으로 확인하여 "프리미엄" 회원에게 더 높은 제한을 부여할 수도 있습니다. 이러한 "반응형 스로틀링(Reactive Throttling)" 접근 방식은 시스템이 안전하면서도 공정하게 유지되도록 보장하며, 프로덕션 환경의 끊임없이 변화하는 상황에 방어 체계를 자동으로 적응시킵니다.

동적 제한의 또 다른 구현은 **적응형 속도 제한(Adaptive Rate Limiting)**입니다. 하드코딩된 숫자 대신 스로틀러가 인프라 모니터링 도구의 피드백 루프를 사용할 수 있습니다. 클러스터 전체의 CPU 사용률이 80%에 도달하면 스로틀러가 필수적이지 않은 모든 라우트에 대해 `limit`를 자동으로 50% 줄일 수 있습니다. 부하가 가라앉으면 원래의 제한으로 복구합니다. 이는 시스템 가용성을 무엇보다 우선시하는 자율 규제 생태계를 구축하여, 극심한 부하 상황에서도 중요한 작업(예: 결제 또는 인증)이 계속 작동하도록 보장합니다.

## 16.6 Advanced: Custom Trackers
때로는 IP 기반 스로틀링만으로는 충분하지 않습니다. 예를 들어, 사무실 건물 내에서는 수백 명의 정당한 사용자가 동일한 공인 IP를 공유할 수 있습니다. 이 경우 `JwtPrincipal`의 subject(사용자 ID)를 기준으로 스로틀링을 수행해야 합니다.

### 16.6.1 The Benefits of Identity-Based Throttling
식별자 기반 스로틀링은 악의적인 사용자 한 명이 공용 IP의 제한을 모두 소진하여 동료들의 접근을 차단하는 일을 방지합니다. 이는 사용자별로 더 공정한 경험을 제공하며 보안 로직을 더 정밀하게 만듭니다. 느슨한 IP 기반 제한과 엄격한 사용자 기반 제한을 결합하여 다차원적인 방어 전략을 구축할 수도 있습니다. 이 접근 방식은 신원이 일시적인 네트워크 주소보다 행동에 대한 더 안정적인 신호임을 전제로 합니다.

기본적으로 throttler는 클라이언트를 IP 주소로 식별합니다. 좋은 출발점이지만, 실제 배포 경로를 함께 생각해야 합니다. 애플리케이션이 프록시(Nginx, Cloudflare 또는 로드 밸런서 등) 뒤에 있는 경우 모든 사용자의 IP가 동일하게 보일 수 있기 때문입니다. 따라서 이미 인증된 사용자라면 사용자 ID를, API 키 기반 클라이언트라면 해당 키를 기준으로 제한을 거는 편이 원시 IP보다 더 공정할 수 있습니다.

### 16.6.2 Throttling by API Key
B2B 애플리케이션의 경우 클라이언트의 API 키를 기반으로 스로틀링을 수행하고 싶을 수 있습니다. `keyGenerator`를 설정하여 요청 헤더에서 API 키를 추출하고, 트래픽의 출처와 관계없이 해당 클라이언트에 특정 제한을 적용할 수 있습니다. 이는 사용량에 따른 요금제 기반 API 서비스에서 흔히 쓰이는 패턴입니다. 이를 통해 인프라 레이어에서 직접 비즈니스 계약을 강제하여 고객이 지불한 만큼의 리소스만 사용하도록 보장할 수 있습니다.

### 16.6.3 Implementing Geo-Aware Throttling
글로벌 애플리케이션에서는 사용자의 지리적 위치에 따라 서로 다른 제한을 적용하고 싶을 수 있습니다. 예를 들어, 의심스러운 활동이 많이 발생하는 지역에 대해서는 제한을 강화할 수 있습니다. 커스텀 `ThrottlerGuard`에 GeoIP 서비스를 통합하면 요청에서 국가 코드를 추출하여 추적 키의 일부로 사용할 수 있습니다. 이러한 "지리적 방어"는 전 세계 사용자 기반 전체에 영향을 주지 않으면서 지역적 위협에 동적으로 대응하기 위한 선택지입니다.

## 16.7 Handling the "Too Many Requests" Error
사용자가 제한을 초과하면 Fluo는 HTTP 패키지의 `TooManyRequestsException`을 던지며, 이는 `429 Too Many Requests` HTTP 상태 코드로 나타납니다. 응답에는 사용자가 얼마나 기다려야 하는지 알려주는 `Retry-After` 헤더가 포함됩니다.

### 16.7.1 Client-Side Responsibility
잘 설계된 클라이언트 측 애플리케이션은 이 429 상태를 감지하여 "제출" 버튼을 비활성화하거나 카운트다운 타이머를 표시해야 합니다. 이는 사용자가 불필요하게 같은 요청을 반복하지 않게 하고, 서버가 의미 없는 재시도를 계속 처리하는 상황을 줄입니다. 좋은 에러 처리는 백엔드와 프론트엔드의 협력으로 완성되며, Fluo는 그 협력에 필요한 메타데이터를 제공합니다. 투명한 제한 정보는 정상적인 연동 코드가 시스템 경계를 지키도록 돕습니다.

### 16.7.2 Customizing the Exception Response
기본 에러 메시지가 API 스타일과 맞지 않는 경우 runtime bootstrap에서 `ExceptionFilterHandler`를 등록하고 `TooManyRequestsException`을 처리할 수 있습니다. 이를 통해 추가 안내, 지원 링크 또는 브랜딩이 포함된 커스텀 JSON 본문을 반환할 수 있습니다. 일관된 에러 형식을 유지하는 것은 고품질의 개발자 경험(DX)을 위해 필수적입니다.

응답을 커스터마이징할 때 **현지화(L10n)**를 포함하는 것도 좋은 관행입니다. 클라이언트가 선호하는 언어(예: `Accept-Language` 헤더 기준)에 따라 사용자가 왜 차단되었는지 이해하는 데 도움이 되는 현지화된 메시지를 제공할 수 있습니다. 이는 기술적인 "429 Too Many Requests" 메시지가 혼란스러울 수 있는 일반 사용자 대상 애플리케이션에서 특히 중요합니다. 현지화된 429 안내는 제한 초과 사실과 재시도 가능 시점을 중립적으로 전달하는 데 초점을 두는 편이 좋습니다.

또한 예외 필터를 사용하여 **외부 보안 알림**을 트리거할 수도 있습니다. 특정 IP나 사용자 ID가 짧은 기간 내에 반복적으로 `TooManyRequestsException`을 발생시키는 경우, 필터는 보안 팀에 알림을 보내거나 방화벽의 임시 "강력 차단(Hard Block)" 목록을 자동으로 업데이트할 수 있습니다. 이러한 능동적인 응답 변환은 스로틀링 레이어를 수동적인 필터에서 시스템 전체 보안 태세의 적극적인 참여자로 변화시킵니다.

```typescript
import { TooManyRequestsException } from '@fluojs/http';
import type { ExceptionFilterHandler } from '@fluojs/runtime';

export class TooManyRequestsFilter implements ExceptionFilterHandler {
  async catch(error, { response, requestId }) {
    if (!(error instanceof TooManyRequestsException)) {
      return undefined;
    }

    const retryAfter = response.headers['Retry-After'];

    response.setStatus(429);
    await response.send({
      error: 'Too Many Requests',
      message: '잠시 숨을 고르세요! 속도 제한을 초과했습니다.',
      requestId,
      retryAfter,
      statusCode: 429,
    });

    return true;
  }
}
```

## 16.8 Best Practices for Throttling
1. **계층형 제한(Tiered Limits)**: 유료 사용자와 무료 사용자에게 서로 다른 제한을 제공하여 API를 수익화하십시오.
2. **주요 IP 화이트리스트**: 내부 도구나 헬스 체크 도구가 절대 차단되지 않도록 하십시오.
3. **프로덕션에서 Redis 사용**: 로드 밸런싱된 환경에서 제한 우회를 방지하기 위해 항상 공유 저장소를 사용하십시오.
4. **정보성 헤더**: 잘 설계된 클라이언트를 돕기 위해 `Retry-After` 헤더를 기준으로 재시도 시점을 안내하십시오.
5. **스로틀러 모니터링**: Fluo의 메트릭 모듈을 사용하여 사용자들이 얼마나 자주 차단되는지 추적하고 제한 수치를 적절히 조정하십시오.
6. **IP보다 식별자 우선**: 공정성과 정밀도를 높이기 위해 가능한 한 인증된 주체(subjects)를 트래커로 사용하십시오.
7. **제한 테스트**: 부하 테스트를 실행하여 스로틀링이 예상한 시점에 정확히 작동하고 과도한 지연 시간을 유발하지 않는지 확인하십시오.
8. **우아한 성능 저하(Graceful Degradation)**: Redis와 같은 스로틀링 레이어에 일시적인 문제가 발생하더라도 시스템이 계속 작동하도록 설계하십시오.

## 16.9 Summary
Rate limiting은 전문적인 API 설계의 초석입니다. 이는 예측 불가능한 네트워크 환경에서 보안, 비용 및 공정성의 균형을 맞춥니다.

인메모리 기본 저장소는 학습용이나 단일 인스턴스 배포에는 잘 맞습니다. 하지만 FluoBlog의 인스턴스를 여러 개 실행하는 경우에는 각 인스턴스가 자체 로컬 카운트를 가지기 때문에 올바르게 작동하지 않습니다.

- **스로틀링**은 DoS 및 무차별 대입 공격을 방지합니다.
- **Redis 저장소**는 분산된 프로덕션 환경에 필수적입니다.
- **@Throttle()**은 민감하거나 트래픽이 많은 라우트에 대해 세밀한 제어를 가능하게 합니다.
- **커스텀 트래커**는 식별자 기반 보호를 가능하게 하여 모든 사용자에 대한 공정성을 보장합니다.
- 계층형 제한 및 정보성 헤더를 포함하는 **전략적 구현**은 보안과 개발자 경험을 모두 개선합니다.

## 16.10 Deep Dive: Throttling in Microservices
마이크로서비스 아키텍처에서 스로틀링은 더욱 복잡해집니다. 전체 클러스터를 보호하기 위해 API 게이트웨이 수준에서 스로틀링을 수행해야 할 수도 있고, 단일 다운스트림 서비스가 내부 요청으로 인해 마비되는 것을 방지하기 위해 개별 서비스 수준에서도 수행해야 할 수도 있습니다. Fluo의 이식성 있는 스로틀러 로직은 이 두 수준 모두에 배포될 수 있어 전체 인프라에 걸쳐 일관된 보안 모델을 제공합니다.

### 16.10.1 Global vs. Local Rate Limiting
전역 속도 제한(보통 게이트웨이에서 수행)은 시스템 전체를 보호하는 반면, 로컬 속도 제한(서비스 수준에서 수행)은 개별 리소스를 보호합니다. 예를 들어, 게이트웨이는 전체적으로 분당 10,000개의 요청을 허용할 수 있지만, `EmailService`는 SMTP 버퍼 오버플로를 방지하기 위해 분당 100개의 요청만 허용할 수 있습니다. 이 두 계층의 균형을 맞추는 것은 건강한 분산 시스템을 위해 매우 중요합니다.

로컬 속도 제한과 함께 **서킷 브레이커(Circuit Breakers)**를 구현하는 것은 강력한 복원력 패턴입니다. 서비스가 로컬 스로틀링으로 인해 429 에러를 반환하기 시작하면, 호출자는 짧은 기간 동안 요청을 완전히 중단해야 합니다. 이는 "천둥 치는 무리(Thundering Herd)"가 어려움을 겪는 서비스에 계속 부하를 주는 것을 방지하여 서비스가 더 빨리 복구되도록 돕습니다. Fluo의 DI 시스템을 사용하면 스로틀러와 다른 복원력 서비스 간에 속도 제한 상태를 쉽게 공유할 수 있어, 응집력 있고 자가 치유가 가능한 아키텍처를 구축할 수 있습니다.

보호 외에도 로컬 속도 제한은 마이크로서비스에서 **테넌트 인식 격리(Tenant-Aware Isolation)**를 위해 사용될 수 있습니다. 다중 테넌트 환경에서는 한 고객의 `Service A` 과도한 사용이 다른 고객의 `Service B` 경험에 영향을 미치는 것을 원치 않을 것입니다. 테넌트 ID를 기반으로 로컬 제한을 적용함으로써 모든 고객이 기본 인프라 리소스를 공정하게 사용하도록 보장하고, 전체 플랫폼에서 엄격한 서비스 수준 협약(SLA)을 유지할 수 있습니다.

기본적으로 throttler는 클라이언트를 IP 주소로 식별합니다. 좋은 출발점이지만, 실제 배포 경로를 함께 생각해야 합니다. 애플리케이션이 프록시(Nginx, Cloudflare 또는 로드 밸런서 등) 뒤에 있는 경우 모든 사용자의 IP가 동일하게 보일 수 있기 때문입니다. 그래서 마이크로서비스 환경에서도 원시 IP만 고집하기보다, 인증된 사용자라면 사용자 ID를, API 키 기반 클라이언트라면 해당 키를 기준으로 제한을 거는 방식이 더 공정하고 추적 가능할 수 있습니다.

### 16.10.2 Service Mesh Integration
Istio나 Linkerd와 같은 서비스 메시를 사용하는 경우, Fluo의 스로틀러가 어떻게 어울리는지 궁금할 수 있습니다. 서비스 메시는 기본적인 속도 제한을 제공할 수 있지만, Fluo의 스로틀러는 애플리케이션의 도메인 로직에 더 가까이 접근합니다. 특정 사용자 역할, 구독 티어 또는 요청 본문의 내용을 기반으로 스로틀링을 수행할 수 있으며, 이는 일반적인 서비스 메시 프록시가 쉽게 알 수 없는 정보입니다. 서비스 메시의 인프라 수준 보호와 Fluo의 애플리케이션 인식 로직을 결합하면 계층별 책임이 분명한 보안 경계를 만들 수 있습니다.

### 16.10.3 Distributed Counter Strategies
다중 지역 배포(예: US-East 및 EU-West)에서 속도 제한을 위해 단일 전역 Redis를 사용하면 멀리 떨어진 지역의 사용자에게 상당한 지연 시간이 발생할 수 있습니다. 이를 해결하기 위해 "로컬 우선(Local-First)" 스로틀링 전략을 사용할 수 있습니다. 각 지역은 빠른 추적을 위해 자체 로컬 Redis를 보유하고, 백그라운드 프로세스가 주기적으로 카운트를 전역 저장소와 동기화합니다. 이는 낮은 지연 시간과 전역적인 정확성 사이의 좋은 균형을 제공합니다. Fluo의 플러그형 저장소 아키텍처는 엔터프라이즈급 애플리케이션을 위한 이러한 하이브리드 전략을 간단하게 구현할 수 있게 해줍니다.

### 16.10.4 Resilience and the "Thundering Herd"
또한, 서비스 중단 후 많은 클라이언트가 동시에 재시도할 때 발생하는 "Thundering Herd" 문제를 고려하십시오. 클라이언트 측에서는 **지터(Jitter)를 포함한 지수 백오프(Exponential Backoff)**와 같은 고급 재시도 패턴을 구현해야 하며, 서버는 스로틀링을 사용하여 결과적으로 발생하는 트래픽 급증을 완만하게 만들어야 합니다. 클라이언트의 첫 시도부터 최종 서비스의 응답까지의 전체 요청 생명주기에 대한 이러한 총체적인 관점이 기초적인 앱과 전문적이고 복원력 있는 시스템을 구분하는 차이입니다. 서버 측 스로틀링과 클라이언트 측 지능을 결합하면 가장 치명적인 장애 상황에서도 시스템이 우아하게 복구되도록 보장할 수 있습니다.

## 16.11 Implementing a "Token Bucket" Algorithm
기본 스로틀러는 고정 윈도우 카운터를 사용하지만, 일부 고성능 시나리오에서는 **토큰 버킷(Token Bucket)** 알고리즘이 필요합니다. 이 모델에서는 토큰이 일정한 속도로 "버킷"에 추가됩니다. 각 요청은 하나의 토큰을 소비합니다. 버킷이 비어 있으면 요청이 차단됩니다. 이를 통해 평균 속도를 유지하면서도 제어된 버스트 트래픽을 허용할 수 있습니다.

Fluo에서는 커스텀 `ThrottlerStore`를 제공하여 이를 구현할 수 있습니다. `consume` 메서드에서 `lastUpdated` 타임스탬프를 추적하고 마지막 요청 이후 추가되었어야 할 토큰 수를 계산합니다. 이 접근 방식은 일시적인 트래픽 급증을 단순히 모두 거부하지 않고 부드럽게 처리하는 데 매우 효과적입니다.

### 16.11.1 Bucket Capacity vs. Refill Rate
버킷 용량은 최대 버스트 크기를 결정하고, 리필 속도는 지속적인 처리량을 결정합니다. 예를 들어 용량이 50이고 초당 10개씩 리필되는 버킷은 사용자가 즉시 50개의 요청을 보낼 수 있게 하지만, 초기 버스트가 소진되면 초당 10개로 제한합니다. 이러한 유연성은 사용자가 빠른 연속 동작을 수행한 후 비활동 기간을 갖는 애플리케이션에 이상적입니다.

### 16.11.2 Precision with Lua Scripts in Redis
분산 환경에서 정확성을 보장하려면 토큰 버킷 로직을 Redis 내의 Lua 스크립트로 구현해야 합니다. 이는 "읽기-계산-쓰기" 시퀀스가 원자적으로 이루어지도록 보장하여, 여러 서버 인스턴스가 사용 가능한 토큰을 중복 계산하는 경쟁 상태를 방지합니다. Fluo의 Redis 프로바이더는 이러한 커스텀 스크립트를 효율적으로 실행하도록 설계되어 있어 고급 속도 제한 로직을 프레임워크 코어만큼 견고하게 만듭니다.

## 16.12 Monitoring and Observability
스로틀링 시스템은 가시성만큼만 강력합니다. 누가 얼마나 자주 제한에 걸리고 있는지 모니터링해야 합니다.

Part 3를 마무리하며, 지금까지 쌓아 온 실천 항목을 한곳에 모아 보면 전체 그림이 더 선명해집니다. 프로덕션 준비가 된 FluoBlog를 위한 체크리스트를 검토해 보겠습니다:

### 16.12.1 Logging Throttled Requests
throttler는 `Retry-After`가 포함된 결정적인 429 응답을 내보냅니다. 보안 분석이 필요한 애플리케이션은 이 예외 주변에 구조화된 로깅을 추가해야 합니다. 트래커 ID(IP 또는 사용자 ID), 특정 라우트 및 타임스탬프를 포함하십시오. 이 데이터는 봇넷 패턴을 식별하고 버그가 있는 클라이언트와 의도적인 공격을 구분하는 데 매우 중요합니다. 이러한 로그를 ELK나 Datadog 같은 도구와 통합하여 실시간 보안 대시보드를 구축할 수 있습니다.

구조화된 로깅을 구현할 때 **상관관계 ID(Correlation IDs)**를 포함하는 것이 유익합니다. 요청이 차단된 경우, 해당 상관관계 ID를 사용하여 마이크로서비스 전체에서 해당 트랜잭션의 전체 이력을 추적할 수 있습니다. 이는 스로틀링 이벤트의 영향을 이해하는 데 도움이 됩니다. 예를 들어, 해당 이벤트가 다운스트림 데이터베이스의 연쇄 장애를 방지했나요? 아니면 재시도 폭풍(retry storm) 중에 업스트림 로드 밸런서에 의해 트리거되었나요? 이러한 종단 간 가시성을 확보하는 것은 복잡한 분산 환경에서 근본 원인 분석(root-cause analysis)을 수행하는 데 필수적입니다.

나아가, **설정 변경에 대한 감사 로깅(Audit Logging)**을 고려해야 합니다. 개발자가 스로틀 제한을 변경하거나 화이트리스트를 업데이트할 때마다 "누가, 언제, 왜" 했는지에 대한 기록이 남아야 합니다. 이는 책임 소재를 명확히 하고, 새로운 제한이 실수로 중요한 연동을 깨뜨렸을 때 "롤백 시나리오"에서 큰 도움이 됩니다. Fluo의 DI system을 사용하면 `AuditService`를 설정 프로바이더에 쉽게 주입하여 이러한 기록 프로세스를 자동화할 수 있으며, 높은 수준의 운영 우수성을 유지할 수 있습니다.

### 16.12.2 Metrics and Alerting
`@fluojs/metrics` 패키지를 사용하여 429 에러 발생률을 추적하십시오. 차단된 요청이 갑자기 급증한다면 이는 보안 사고나 클라이언트 애플리케이션의 심각한 결함을 나타내는 강력한 신호입니다. 이러한 지표에 대한 알림을 설정하면 SRE 팀이 시스템 안정성에 영향을 주기 전에 선제적으로 대응할 수 있습니다.

### 16.12.3 Visualizing Traffic Windows
정말 전문적인 설정을 원한다면 상위 사용자들의 "남은" 용량을 시각화하는 대시보드를 구축할 수 있습니다. 이는 사용자의 최근 트래픽 이력을 명확하게 보여줌으로써 고객 지원 팀이 "왜 429 에러가 발생하나요?"와 같은 질문에 답변하는 데 도움을 줍니다. 커스텀 `ThrottlerStore`를 사용하면 store counter를 읽는 보안 관리자 엔드포인트를 노출해 이러한 투명성을 제공할 수 있습니다.

## 16.13 Throttling Beyond HTTP
Rate limiting은 REST API만을 위한 것이 아닙니다. 현대적인 Fluo 애플리케이션에서는 이러한 원칙을 다른 통신 채널에도 적용해야 할 수 있습니다.

### 16.13.1 WebSockets and Real-Time Data
WebSocket 연결의 경우, 악의적인 클라이언트가 메시지 버스를 플러딩하는 것을 방지하기 위해 들어오는 메시지에 스로틀링을 적용할 수 있습니다. WebSocket 연결은 수명이 길기 때문에 소켓 ID별 메시지 속도를 추적할 수 있습니다. 이를 통해 하나의 연결이 오작동하더라도 실시간 기능의 응답성을 유지할 수 있습니다.

WebSocket을 스로틀링할 때 **인바운드 바이트 속도(Inbound Byte Rates)**를 모니터링하는 것도 좋은 관행입니다. 공격자는 서버의 메모리를 고갈시키기 위해 많은 수의 작은 메시지 대신 몇 개의 매우 큰 메시지를 보낼 수 있습니다. `WebSocketGuard`에서 메시지 수 제한과 바이트 속도 제한을 결합하면 더욱 종합적인 방어가 가능합니다. 클라이언트가 이러한 제한을 초과하면 메시지를 조용히 드랍하거나 시스템 리소스를 회수하기 위해 강제로 연결을 종료할 수 있습니다.

또한 연결 유지 시간에 따른 **동적 WebSocket 제한**을 고려하십시오. 새로 수립된 연결은 시간이 지나면서 "신뢰성"을 증명할 때까지 더 엄격한 제한을 가질 수 있습니다. 이는 자동화된 연결 폭주 공격을 완화하는 데 도움이 됩니다. Fluo에서는 각 소켓 이벤트를 가로챌 수 있으므로, 메인 비즈니스 로직을 복잡하게 만들지 않고도 이러한 정교한 실시간 보안 정책을 구현하는 데 필요한 세밀한 제어가 가능합니다.

### 16.13.2 Message Queues and Background Jobs
큐(`@fluojs/queue`)에서 작업을 처리할 때, 타사 서비스(이메일 제공업체나 결제 게이트웨이 등)를 압도하지 않도록 처리 속도를 조절해야 할 수 있습니다. 작업 프로세서 내에서 `ThrottlerGuard` 로직을 사용하면 워커가 외부 시스템의 한계를 존중하도록 보장하고, 제한에 도달하면 나중에 자동으로 작업을 재시도하게 할 수 있습니다.

**큐 컨슈머의 백프레셔(Backpressure)** 구현은 이 전략의 핵심입니다. 가능한 한 많은 작업을 가져온 다음 속도 제한으로 인해 실패시키는 대신, 컨슈머가 스로틀러 상태를 모니터링하고 폴링 속도를 선제적으로 늦출 수 있습니다. 이는 큐가 "실패했지만 재시도 가능한" 작업으로 가득 차는 것을 방지하여 높은 지연 시간과 데이터베이스 비대화를 막아줍니다. Fluo에서는 큐 프로세서를 `ThrottlerModule`과 쉽게 통합하여 이러한 균형 잡힌 처리량을 달성할 수 있습니다.

나아가 **작업별 속도 제한**을 고려해야 합니다. 모든 백그라운드 작업이 동일하지는 않습니다. "긴급 알림" 작업은 "월간 보고서 생성" 작업보다 더 높은 우선순위와 더 느슨한 속도 제한을 가질 수 있습니다. 서로 다른 작업 카테고리에 대해 특정 스로틀러를 정의함으로써, 중요도가 낮은 백그라운드 태스크로 인해 가장 중요한 비즈니스 프로세스가 지연되지 않도록 보장할 수 있습니다. 이러한 지능적인 우선순위 지정은 단순한 태스크 러너와 프로덕션급 작업 처리 시스템을 구분하는 차이점입니다.

### 16.13.3 GraphQL Complexity Throttling
GraphQL API에서는 단일 쿼리가 매우 비쌀 수 있기 때문에 단순한 요청 카운팅만으로는 부족합니다. 대신 "쿼리 복잡도(Query Complexity)"를 기준으로 스로틀링해야 합니다. Fluo의 GraphQL 통합을 사용하면 각 필드에 비용을 할당하고 총 복잡도 예산을 초과하는 쿼리를 거부할 수 있습니다. 이러한 "비용 기반 스로틀링"은 유연하고 풍부한 데이터를 제공하는 API에서 중요한 보호책입니다.

**복잡도 기반 스로틀링(Complexity-Based Throttling)**을 구현하려면 일반적으로 쿼리가 실행되기 전에 "비용(Cost)"을 계산하는 유효성 검사 규칙을 사용합니다. 예를 들어 사용자 목록을 가져오는 데는 사용자당 1포인트가 소요되지만, 각 사용자의 관련 게시물을 가져오는 데는 추가로 5포인트가 소요될 수 있습니다. 이러한 비용을 정의함으로써 악의적이거나 잘못 작성된 쿼리로 인해 서버를 마비시킬 수 있는 "N+1" 성능 문제나 깊은 재귀 공격을 방지할 수 있습니다. Fluo에서 이러한 복잡도 분석은 `@fluojs/graphql` 패키지에 깊이 통합되어 있어 데코레이터를 통해 선언적으로 비용을 할당할 수 있습니다.

나아가 복잡도 분석을 **영구 쿼리 화이트리스트(Persistent Query Whitelisting)**와 결합할 수 있습니다. 프로덕션에서 사전 승인된 명명된 쿼리만 실행할 수 있도록 허용함으로써, 공격자가 임의의 고비용 쿼리를 보낼 위험을 제거합니다. 임의의 쿼리가 필요한 공개 GraphQL API의 경우, 복잡도 기반 스로틀링이 여전히 가장 강력한 방어책입니다. 요청되는 데이터의 "비용"과 "형태"를 모두 분석하는 이러한 이중 계층 접근 방식은 클라이언트의 요구 사항이 아무리 복잡해지더라도 GraphQL 백엔드가 우수한 성능과 보안을 유지하도록 보장합니다.

## 16.14 Common Pitfalls and How to Avoid Them
좋은 도구가 있더라도 속도 제한을 구현할 때 실수하기 쉽습니다.

### 16.14.1 Setting Limits Too Low
제한이 너무 공격적이면 정당한 사용자들을 좌절시키게 됩니다. 항상 느슨한 제한으로 시작하여 관찰된 트래픽 데이터를 바탕으로 점진적으로 조여 나가십시오. 메트릭을 사용하여 일반 사용자의 95%에게는 영향을 주지 않으면서 남용 트래픽의 상위 1%를 차단하는 "스위트 스팟"을 찾으십시오.

### 16.14.2 Forgetting Proxies and Load Balancers
Fluo 앱이 프록시(Nginx나 Cloudflare 등) 뒤에 있는 경우 framework request에 노출되는 raw socket 식별자가 프록시 주소일 수 있습니다. 프록시가 forwarded header를 덮어쓰는 경우에만 `trustProxyHeaders: true`를 활성화하여 throttler가 `Forwarded`, `X-Forwarded-For`, `X-Real-IP`에서 실제 클라이언트 IP를 볼 수 있게 하십시오. 그렇지 않으면 실수로 모든 사용자를 한꺼번에 차단할 수도 있습니다!

**Cloudflare**나 다른 특수한 프록시를 사용하는 경우 클라이언트의 국가 코드(예: `cf-ipcountry` 헤더 기준)를 추출할 수도 있습니다. 이 정보를 추적 로직의 일부로 사용하여 지역별 제한을 적용하거나 고위험 국가의 트래픽을 완전히 차단할 수 있습니다. 이러한 "에지 우선 통합(Edge-First Integration)"은 인프라 공급자가 제공하는 정보를 활용하여 애플리케이션의 복원력을 높입니다. Fluo의 DI 시스템을 사용하면 커스텀 `ThrottlerGuard`에 `CountryService`를 쉽게 주입하여 최소한의 오버헤드로 이 로직을 처리할 수 있습니다.

나아가 **헤더 스푸핑(Header Spoofing) 방지**를 고려해야 합니다. 공격자는 IP 기반 스로틀링을 피하기 위해 가짜 `X-Forwarded-For` 헤더를 보낼 수 있습니다. 프록시 설정은 이러한 헤더가 Fluo 애플리케이션에 도달하기 전에 제거하거나 덮어쓰도록 구성되어야 합니다. 신뢰할 수 있는 내부 프록시 IP에서 온 헤더만 신뢰함으로써 속도 제한 로직이 정확하게 유지되고 외부인에 의해 조작되지 않도록 보장할 수 있습니다. 이러한 세심한 주의는 현대적인 네트워크 환경에서 안전하고 신뢰할 수 있는 API를 유지하는 데 필수적입니다.

### 16.14.3 Ignoring the Client Experience
내용 없는 429 페이지만 반환하지 마십시오. `Retry-After` 헤더와 명확한 에러 메시지를 제공하십시오. 규칙과 제한을 투명하게 공개하면 동료 개발자들이 더 나은 연동 코드를 작성할 수 있고, 보안과 사용성을 함께 확보할 수 있습니다.

## 16.15 Summary
속도 제한(Rate Limiting)은 전문적인 API 설계의 초석입니다. 이는 예측 불가능한 네트워크 환경에서 보안, 비용, 그리고 공정성의 균형을 맞추는 역할을 합니다.

- **심층 방어(Defense in Depth)**: 다중 계층(단기/장기)과 다양한 유형(IP/식별자)의 스로틀링을 사용하십시오.
- **분산 상태 관리**: 서버 클러스 전체에서 정확한 카운팅을 위해 Redis를 사용하십시오.
- **개발자 경험(DX)**: 클라이언트에게 명확한 헤더와 도움이 되는 에러 응답을 제공하십시오.
- **관측 가능성(Observability)**: 스로틀러 메트릭을 모니터링하여 공격을 탐지하고 정책을 조정하십시오.
- **전체론적 보호**: HTTP뿐만 아니라 WebSocket 및 백그라운드 작업에도 스로틀링 원칙을 적용하십시오.

이제 FluoBlog는 사용자를 인증하고, 요청을 인가하고, 민감한 엔드포인트에 적절한 제한까지 둘 수 있게 되었습니다. 초기 단계에서 갖추기 좋은 보안 기본선이 마련된 셈이며, 다음 Part도 이 기반 위에서 자연스럽게 이어집니다. Part 4에서는 HTTP를 넘어 WebSockets를 이용한 실시간 통신을 살펴보겠습니다.
