## 14.1 Introduction to JWT

JSON Web Token(JWT)은 당사자 간에 정보를 JSON 객체로 안전하게 전송하기 위한 간결하고 독립적인 방식을 정의하는 개방형 표준(RFC 7519)입니다. 전통적인 세션 기반 인증과 달리, JWT를 사용하면 서버가 데이터베이스나 세션 저장소를 조회하지 않고도 요청을 검증할 수 있으므로 분산 시스템 및 서버리스 환경에 이상적입니다. 이러한 상태 유지(stateful) 방식에서 상태 비저장(stateless) 인증으로의 전환은 현대 백엔드 개발의 초석이며, 애플리케이션이 매우 동적인 환경에서 효율적으로 운영될 수 있게 해줍니다.

현대적인 웹 애플리케이션에서 JWT는 상태 비저장 인증의 사실상 표준입니다. 서버는 데이터베이스에 세션 ID를 저장하는 대신, 암호화되어 서명된 토큰을 클라이언트에 발급합니다. 클라이언트는 모든 요청마다 이 토큰을 주로 `Authorization: Bearer <token>` 헤더에 담아 다시 보냅니다. 이 접근 방식은 서버 측에서 메모리 집약적인 세션 저장소를 필요로 하지 않으므로, 동기화 문제 없이 여러 인스턴스나 심지어 서로 다른 클라우드 지역에 걸쳐 애플리케이션을 수평적으로 확장할 수 있게 해줍니다. 산업계가 전 세계적으로 분산된 시스템으로 이동함에 따라, 중앙 상태 없이 인증할 수 있는 능력은 거대한 경쟁 우위가 됩니다.

FluoBlog에서는 이 표준이 왜 중요한지부터 이해하면 뒤의 설정이 더 자연스럽습니다. JWT를 사용하면 서버가 세션 상태를 계속 조회하지 않아도 요청마다 사용자 신원을 전달할 수 있습니다. 세션 ID를 데이터베이스에 저장하고 모든 요청마다 확인하는 대신, 서버는 클라이언트에 서명된 토큰을 발급합니다. 그러면 클라이언트는 모든 요청과 함께 이 토큰을 다시 보내고, 서버는 토큰을 확인하는 것만으로 사용자의 신원을 검증할 수 있습니다.

### Structure of a JWT
JWT는 점(`.`)으로 구분된 세 부분으로 구성됩니다.
1. **Header**: 서명 알고리즘(예: `HS256` 또는 `RS256`)을 포함한 토큰에 대한 메타데이터입니다.
2. **Payload**: "클레임(claims)"이라고 불리는 사용자 ID(`sub`), 만료 시간(`exp`), 역할 등 실제 데이터입니다.
3. **Signature**: 인코딩된 헤더와 페이로드를 비밀 키와 결합하여 생성된 해시입니다. 이는 토큰이 변조되지 않았음을 보장합니다.

JWT를 밀랍 인장으로 봉인된 봉투라고 생각하면 이해하기 쉽습니다. 헤더는 누가 보냈는지, 어떻게 봉인되었는지 알려줍니다. 페이로드는 봉투 안의 편지입니다. 서명은 밀랍 인장 그 자체입니다. 만약 인장이 부서졌거나 교체되었다면, 수신자는 즉시 내용물이 변조되었음을 알 수 있습니다. 이러한 자기 검증 특성이 바로 JWT를 현대 웹 아키텍처에서 매우 강력하게 만드는 요소입니다. 공격자가 토큰을 가로채더라도 비밀 키가 안전하게 유지되는 한 서명을 무효화하지 않고는 페이로드를 변경할 수 없습니다. 서명의 암호학적 강도는 전체 시스템의 신뢰 기반을 제공합니다.

### Why Statelessness Matters
JWT의 가장 큰 장점은 상태 비저장 특성입니다. 전통적인 세션 기반 시스템에서 서버는 모든 요청마다 데이터베이스나 Redis와 같은 공유 캐시에서 세션 ID를 조회해야 합니다. 이는 병목 현상과 단일 장애 지점(Single Point of Failure)을 만듭니다. 만약 세션 저장소가 다운되면 아무도 로그인할 수 없으며, 현재 접속 중인 모든 사용자도 사실상 로그아웃됩니다.

JWT를 사용하면 서버는 네트워크 호출 없이 단 몇 마이크로초 만에 끝나는 암호학적 체크만 수행합니다. 이러한 효율성은 지연 시간(latency)이 중요한 지표인 트래픽이 많은 API에서 매우 중요합니다. 상태를 클라이언트에 위임함으로써 인프라를 단순화하고 전반적인 시스템 복원력을 향상시킬 수 있습니다. 또한 상태 비저장성은 애플리케이션을 기본적으로 "클라우드 네이티브"하게 만듭니다. 모든 서버 인스턴스가 동일한 서명 비밀 키나 공개 키를 공유하는 한 어떤 인스턴스가 요청을 처리하든 상관없기 때문입니다. 서버 상태와 식별의 이러한 분리는 현대의 컨테이너화 및 자동 확장(auto-scaling) 환경에서 필수적입니다.

## 14.2 The @fluojs/jwt Package
Fluo는 전송 방식에 구애받지 않고 "표준 우선(Standard-First)" 시대를 위해 구축된 `@fluojs/jwt` 패키지를 제공합니다. 이 패키지는 표준 Web Crypto API를 준수하면서 토큰의 서명, 검증 및 데이터 추출과 한 무거운 작업을 처리합니다. 표준 Web Crypto API를 사용함으로써 Fluo는 여러분의 인증 로직이 Node.js, Bun, Deno와 같은 서로 다른 런타임 간에 이식 가능하도록 보장하여 벤더 종속성을 방지하고 코드베이스의 미래를 보장합니다.

`fluo`는 전용 패키지인 `@fluojs/jwt`를 제공하며, 이는 전송 계층에 독립적(transport-agnostic)입니다. 즉, FluoBlog에서 HTTP를 먼저 다루더라도 같은 토큰 모델을 WebSockets나 RPC 호출에도 이어서 사용할 수 있습니다.

### Core Philosophy: Principal Normalization
Fluo의 가장 강력한 기능 중 하나는 **Principal 정규화(Principal Normalization)**입니다. 실제 프로젝트에서 서로 다른 시스템은 클레임에 대해 서로 다른 명명 규칙을 사용할 수 있습니다(예: 어떤 시스템은 `uid`, 다른 시스템은 `sub`, 레거시 시스템은 `userId` 사용). 이러한 불일치는 단지 사용자의 식별 정보를 추출하기 위해 수많은 조건부 로직으로 가득 찬 지저분한 코드를 유발하곤 합니다.

`@fluojs/jwt`는 이러한 변형들을 자동으로 통일된 `JwtPrincipal` 객체로 매핑합니다.
- `subject`: 사용자의 고유 ID (`sub`에서 매핑).
- `roles`: RBAC를 위한 문자열 배열 (`roles`, `groups`, 또는 `permissions`에서 매핑).
- `scopes`: 특정 권한 마커 (`scope` 또는 `scp`에서 매핑).
- `claims`: 페이로드의 추가 커스텀 데이터를 담는 가공되지 않은 바구니.

이러한 정규화 레이어 덕분에 ID 공급자를 변경하거나 토큰 구조를 리팩토링하더라도 비즈니스 로직을 변경할 필요가 없습니다. 가드와 서비스는 단순히 `JwtPrincipal`과 상호작용하므로 코드를 유지 관리하고 이해하기가 훨씬 쉬워집니다. 이는 파편화된 클레임 세트를 신뢰할 수 있고 타입이 지정된 인터페이스로 바꿔줍니다. 이러한 추상화는 여러 OAuth2 또는 OpenID Connect 공급자와 통합되는 시스템에서 매우 중요한데, 소스에 관계없이 사용자 식별을 위한 단일 진입점을 제공하기 때문입니다.

## 14.3 Configuring JwtModule
FluoBlog에서 JWT를 사용하려면 `JwtModule`을 등록하십시오. 빠른 실험을 위해 `forRoot`를 사용할 수도 있지만, 프로덕션 환경에서는 `ConfigService`를 사용하는 `forRootAsync`가 표준입니다. 이와 같이 설정을 중앙 집중화하면 전체 애플리케이션에서 일관성을 보장할 수 있으며, 비밀 키나 토큰 수명과 같은 환경별 설정을 관리하기가 훨씬 쉬워집니다.

토큰의 모양과 역할을 이해했다면, 이제 그 규칙을 애플리케이션 설정으로 옮겨야 합니다. FluoBlog에서 JWT를 사용하기 시작하려면 `JwtModule`을 등록해야 합니다.

### Static Registration
단순한 설정의 경우 `forRoot`를 사용할 수 있습니다:

```typescript
import { Module } from '@fluojs/core';
import { JwtModule } from '@fluojs/jwt';

@Module({
  imports: [
    JwtModule.forRoot({
      secret: 'your-very-secure-secret',
      issuer: 'fluoblog-api',
      audience: 'fluoblog-client',
      accessTokenTtlSeconds: 3600, // 1시간
    }),
  ],
})
export class AuthModule {}
```

### Dynamic Registration with ConfigService
하드코딩된 예시는 설정의 형태를 빠르게 보여 주지만, 실제 운영 방식으로 이어지면 안 됩니다. 프로덕션 환경에서는 비밀 키를 절대 하드코딩해서는 안 됩니다. 대신 Chapter 11에서 배운 `ConfigService`를 사용하세요.

```typescript
import { Module } from '@fluojs/core';
import { JwtModule } from '@fluojs/jwt';
import { ConfigService } from '@fluojs/config';

@Module({
  imports: [
    JwtModule.forRootAsync({
      inject: [ConfigService],
      useFactory: (config: ConfigService) => ({
        // 강력하고 환경별로 고유한 비밀 키를 사용하세요.
        secret: config.get('JWT_SECRET'),
        issuer: 'fluoblog-api',
        audience: 'fluoblog-client',
        // 보안을 위해 액세스 토큰은 수명이 짧아야 합니다.
        accessTokenTtlSeconds: 900, // 15분
      }),
    }),
  ],
})
export class AuthModule {}
```

### Advanced Configuration Options
`JwtModule`은 단순한 비밀 키 외에도 광범위한 설정 옵션을 지원합니다. 여러 발급자(issuers)를 정의하거나, 검증을 위한 클록 스큐(clock skew)를 커스터마이징하거나, 특정 사용 사례를 위한 커스텀 서명자(signers)를 제공할 수 있습니다. 예를 들어, 시계가 동기화되지 않았을 가능성이 있는 모바일 기기에서 오는 토큰에 대해 더 긴 클록 스큐를 허용하여 사용자에게 불필요한 "인증되지 않음" 에러가 발생하는 것을 방지할 수 있습니다.

Fluo는 기본 경험을 단순하고 안전하게 유지하면서도 이러한 실제 세계의 예외 상황을 처리하는 데 필요한 세밀한 제어 권한을 제공합니다. 또한 로컬 개발에는 `HS256`을 사용하고 프로덕션에는 인증서와 함께 `RS256`을 사용하는 등 환경마다 다른 알고리즘을 설정할 수도 있습니다. 이러한 유연성은 모듈의 핵심 아키텍처에 내장되어 있습니다. 더 나아가 일회성 작업을 위한 극히 짧은 수명의 토큰이나 신뢰할 수 있는 내부 서비스를 위한 더 긴 수명의 액세스 토큰 등 토큰 유형별로 다른 TTL(Time To Live) 설정을 정의할 수 있습니다.

## 14.4 Signing Tokens
설정이 완료되면 `DefaultJwtSigner`를 주입하여 로그인 과정에서 토큰을 생성할 수 있습니다. 서명자는 복잡한 인코딩 및 서명 로직을 처리하므로, 여러분은 사용자의 신원을 나타내는 페이로드에만 집중할 수 있습니다.

잘 설계된 페이로드는 효율적인 인증의 핵심입니다. 사용자 ID나 역할과 같은 충분한 정보를 포함함으로써, 다운스트림 서비스가 반복적인 데이터베이스 조회 없이 인가 결정을 내릴 수 있도록 지원합니다. 이러한 데이터와 보안의 균형이 `@fluojs/jwt`가 진정으로 빛을 발하는 지점입니다. 토큰은 이를 소유한 사람이라면 누구나 쉽게 디코딩할 수 있으므로, 페이로드에 비밀번호나 개인정보(PII)와 같은 민감한 정보를 넣어서는 안 됩니다. 목표는 토큰을 사용자가 누구인지, 무엇을 할 수 있는지 증명하는 데 필요한 최소한의 정보만 담은 "여권"으로 만드는 것입니다.

모듈이 서명과 검증 규칙을 알게 되면, 이제 서비스 계층에서 실제 토큰을 발급할 수 있습니다. 이때 `DefaultJwtSigner`를 주입하면 컨트롤러가 반환할 토큰을 만들 수 있습니다.

```typescript
import { Inject } from '@fluojs/core';
import { DefaultJwtSigner } from '@fluojs/jwt';

@Inject(DefaultJwtSigner)
export class AuthService {
  constructor(private readonly signer: DefaultJwtSigner) {}

  async generateToken(user: User) {
    const payload = {
      sub: user.id.toString(),
      roles: user.roles,
      // 커스텀 비즈니스 로직 클레임
      scopes: ['posts:write', 'comments:read'],
    };

    // 최종적으로 base64 인코딩된 문자열을 생성합니다.
    const accessToken = await this.signer.signAccessToken(payload);
    return { accessToken };
  }
}
```

### Managing Token Claims Effectively
토큰 클레임을 설계할 때 애플리케이션이 "에지(edge)"에서 알아야 할 정보가 무엇인지 생각하십시오. API 게이트웨이가 요청을 라우팅하기 위해 사용자가 'admin'인지 알아야 한다면 토큰에 'admin' 역할을 포함하십시오. 하지만 사용자의 전체 프로필을 포함하지는 마십시오. 특히 모바일 클라이언트를 위해 대역폭 사용을 최소화하도록 토큰을 작게 유지하십시오. 작은 토큰은 암호학적 검증의 오버헤드도 줄여줍니다. Fluo의 `DefaultJwtSigner`를 사용하면 애플리케이션 요구 사항이 변함에 따라 클레임을 쉽게 추가하거나 제거할 수 있으며, 토큰 생성을 위한 깔끔한 API를 제공합니다. 효과적인 클레임 관리에는 타사 도구 및 라이브러리와의 호환성을 보장하기 위해 가능한 한 표준 클레임 이름(`iat`, `exp`, `nbf` 등)을 사용하는 것도 포함됩니다.

## 14.5 Refresh Token Rotation
액세스 토큰은 탈취될 경우의 피해를 최소화하기 위해 의도적으로 수명을 짧게 설정합니다. 하지만 사용자가 15분마다 로그인하게 할 수는 없습니다. 바로 여기서 **리프레시 토큰(Refresh Tokens)**이 등장합니다.

1. **Access Token**: 짧은 수명 (15분). API 접근에 사용됩니다.
2. **Refresh Token**: 긴 수명 (7일). 오직 *새로운* 액세스 토큰을 요청할 때만 사용됩니다.

토큰을 발급하는 것만으로 인증 흐름이 끝나지는 않습니다. 사용자는 다시 로그인하지 않고 세션을 이어 가야 하고, 동시에 액세스 토큰은 너무 오래 살아 있지 않아야 합니다. 그래서 보안에 민감한 애플리케이션은 "이중 토큰(Dual Token)" 패턴을 사용합니다:
1. **Access Token**: 수명이 짧음(예: 15분). 모든 요청에 사용됨.
2. **Refresh Token**: 수명이 김(예: 7일). 새로운 액세스 토큰을 얻는 데만 사용됨.

### Rotation Strategy
Fluo는 **리프레시 토큰 로테이션(Refresh Token Rotation)**을 구현합니다. 클라이언트가 새로운 액세스 토큰을 받기 위해 리프레시 토큰을 사용할 때마다, 서버는 해당 리프레시 토큰을 무효화하고 *새로운* 리프레시 토큰을 발급합니다. 만약 공격자와 정당한 사용자가 동일한 리프레시 토큰을 사용하려 한다면, Fluo는 재사용을 감지하고 해당 토큰 제품군(family) 전체를 무효화하여 관련된 모든 사람에게 재로그인을 강제합니다.

이러한 선제적인 보안 접근 방식은 리프레시 토큰이 유출되더라도 악용될 수 있는 기간을 극도로 좁혀줍니다. 재사용 감지 시 토큰 제품군 전체를 무효화함으로써 사용자를 지속적인 무단 접근으로부터 보호합니다. 이는 민감한 사용자 데이터를 다루는 모든 애플리케이션에서 매우 중요한 심층 방어(defense-in-depth) 조치입니다. 로테이션을 수동으로 구현하는 것은 매우 어렵고 오류가 발생하기 쉽지만, Fluo의 내장 지원을 통해 표준적이고 걱정 없는 인증 흐름의 일부가 됩니다. 이는 복잡한 보안 프로토콜을 간단한 설정 하나로 바꿔줍니다.

### Securing Refresh Tokens
리프레시 토큰은 수명이 길기 때문에 각별히 주의해서 저장해야 합니다. 웹에서는 `httpOnly`, `secure`, `sameSite: 'strict'` 쿠키에 저장하는 것이 표준입니다. 이는 크로스 사이트 스크립팅(XSS) 공격을 통해 JavaScript로 토큰에 접근하는 것을 방지합니다. Fluo의 인증 패턴은 쿠키 기반과 헤더 기반 토큰 전달 방식 모두와 원활하게 작동하도록 설계되어, 브라우저, 네이티브 모바일 앱, 또는 다른 서버 등 특정 클라이언트 유형에 가장 적합한 보안 모델을 선택할 수 있는 유연성을 제공합니다. 또한 모바일 앱의 경우 보안 인클레이브(secure enclaves)나 키체인 저장소를 활용하여 이러한 영구적인 자격 증명을 무단 추출로부터 보호하는 것이 권장됩니다.
## 14.6 Implementing FluoBlog Auth Endpoints
이제 설정과 토큰 수명 주기를 실제 엔드포인트 흐름으로 연결해 보겠습니다. FluoBlog를 위한 실제 `AuthController`를 만들어 보겠습니다.

```typescript
// src/auth/auth.controller.ts
import { Controller, Post, RequestDto } from '@fluojs/http';
import { AuthService } from './auth.service';
import { LoginDto } from './dto/login.dto';

@Controller('auth')
export class AuthController {
  constructor(private readonly authService: AuthService) {}

  @Post('login')
  @RequestDto(LoginDto)
  async login(dto: LoginDto) {
    // 1. AuthService를 통해 자격 증명 확인
    // 2. signAccessToken 및 signRefreshToken 수행
    return this.authService.signIn(dto.email, dto.password);
  }
}
```

### The Authentication Lifecycle
Fluo에서의 인증 생명주기는 `login` 엔드포인트에 대한 요청으로 시작됩니다. 자격 증명을 확인한 후(주로 데이터베이스에서 해시된 비밀번호 확인), 서비스는 `JwtSigner`를 사용하여 토큰을 생성합니다. 이 토큰들은 응답 바디나 보안 쿠키를 통해 클라이언트에게 반환됩니다.

그 시점부터 클라이언트는 모든 요청의 `Authorization` 헤더에 액세스 토큰을 포함합니다. 액세스 토큰이 만료되면 클라이언트는 새로운 토큰 쌍을 얻기 위해 리프레시 토큰과 함께 `refresh` 엔드포인트를 호출합니다. 이 사이클은 상태 비저장성의 성능 이점을 유지하면서 지속적이고 안전한 사용자 세션을 보장합니다. 이는 애플리케이션의 정문을 안전하면서도 환영받는 상태로 유지하는 엔진입니다. 또한 이러한 생명주기를 통해 약간 만료된 액세스 토큰이 특정 저위험 작업에는 여전히 허용되면서도 다른 작업에는 강제 갱신을 트리거하는 "유예 기간"을 둘 수도 있습니다.

## 14.7 Verifying Tokens Manually
대부분의 라우트는 가드(Chapter 15)를 사용하겠지만, `DefaultJwtVerifier`를 사용하여 토큰을 수동으로 검증할 수도 있습니다. 이는 이메일로 전송된 비밀번호 재설정 토큰 확인, 일회용 비밀번호(OTP) 토큰 체크, 또는 HTTP 요청 컨텍스트 외부에서 작동하는 백그라운드 작업에서의 토큰 검증과 같은 일회성 작업에 유용합니다.

실제 애플리케이션에서는 보통 Chapter 15의 가드가 이 검증을 대신 처리합니다. 그래도 한 번 직접 확인해 보면 다음 장에서 가드가 무엇을 대신해 주는지 더 분명해집니다. `DefaultJwtVerifier`를 주입하여 수동으로 검증할 수도 있습니다.

```typescript
import { Inject } from '@fluojs/core';
import { UnauthorizedException } from '@fluojs/http';
import { DefaultJwtVerifier } from '@fluojs/jwt';

@Inject(DefaultJwtVerifier)
export class TokenService {
  constructor(private readonly verifier: DefaultJwtVerifier) {}

  async check(token: string) {
    try {
      const principal = await this.verifier.verifyAccessToken(token);
      return principal;
    } catch (e) {
      // ExpiredTokenError 또는 InvalidSignatureError의 자동 처리
      throw new UnauthorizedException('토큰이 만료되었거나 위조되었습니다');
    }
  }
}
```

### Handling Token Errors Gracefully
검증이 실패하면 `DefaultJwtVerifier`는 적절하게 반응할 수 있도록 구체적인 에러 타입을 던집니다. `ExpiredTokenError`는 토큰이 유효했지만 시간이 초과되었음을 알려주며, `InvalidSignatureError`는 잠재적인 변조 시도나 서명 키 불일치를 나타냅니다.

이러한 구체적인 에러를 잡음으로써 단순히 "접근 거부"라고 말하는 대신 사용자에게 세션을 갱신하라고 알려주는 등 더 나은 피드백을 제공하거나, 모니터링 시스템에서 보안 경보를 트리거할 수 있습니다. 예를 들어 특정 IP 주소에서 `InvalidSignatureError`가 빈번하게 발생하면 방화벽에서 자동으로 차단하도록 할 수 있습니다. Fluo의 명시적인 에러 처리는 모호한 에러 메시지와 씨름하지 않고도 이러한 고급 보안 기능을 구축할 수 있게 해줍니다. 또한 클라이언트 측의 버그(예: 빈 토큰 전송)와 악의적인 활동을 구분할 수 있게 해줍니다.

## 14.8 Best Practices for JWT in Fluo
- **페이로드에 민감한 데이터를 저장하지 마세요**: JWT는 인코딩될 뿐 암호화되는 것이 아닙니다. 누구나 내용을 볼 수 있습니다.
- **확장성을 위해 비대칭 서명(RS256)을 사용하세요**: 여러 서비스가 있는 경우, 비밀 키로 서명하고 공개 키로 검증하십시오. 이는 팀 간에 비밀 키를 공유할 필요를 없애고 전반적인 보안 태세를 향상시킵니다.
- **토큰 만료 모니터링**: `exp` 클레임을 사용하여 로그아웃을 강제하고, Chapter 19의 메트릭을 사용하여 무차별 대입이나 자격 증명 스터핑 공격의 징후일 수 있는 높은 인증 실패율을 추적하십시오.
- **토큰 폐기 검증**: 중요한 애플리케이션의 경우, 사용자가 로그아웃하거나 토큰이 만료되기 전에 계정이 침해된 경우를 처리하기 위해 폐기된 토큰의 "거부 목록(denylist)"(예: Redis에 저장)을 유지하십시오.
- **JTI (JWT ID) 구현**: 개별 토큰을 추적하고 세밀한 폐기를 가능하게 하려면 모든 토큰에 고유 식별자를 사용하십시오.
- **토큰 발급 감사**: 사고 후 분석에 도움이 되도록 토큰이 누구에게 언제 발급되었는지 로그를 남기십시오.

## 14.9 Summary
JWT는 FluoBlog의 보안 중추입니다. `@fluojs/jwt`를 활용하면 표준을 준수하고 정규화되었으며 로테이션을 지원하는 인증 시스템을 즉시 사용할 수 있습니다. 신뢰할 수 있는 인증은 단지 비밀번호를 확인하는 것 이상입니다. 애플리케이션의 성장에 따라 확장되고 전체 인터넷에서 사용자의 데이터를 보호하는 강력한 신원 레이어를 구축하는 것입니다.

- `JwtModule`은 보안 설정을 중앙 집중화하고 환경 간 관리를 용이하게 합니다.
- `JwtPrincipal`은 다양한 신원 형식을 정규화하여 비즈니스 로직을 단순화하고 버그를 줄입니다.
- 이중 토큰 패턴과 로테이션은 현대의 자동화된 위협에 대해 보안 태세를 크게 강화합니다.
- 상태 비저장성은 성능과 인프라 확장성을 향상시킵니다.
- 토큰 생명주기와 로테이션의 선제적 관리는 사용자와의 신뢰를 구축하는 데 필수적입니다.

다음 장에서는 **Passport 전략**과 **가드**를 사용하여 이러한 토큰을 실제 HTTP 요청 생명주기에 연결하고, 서명된 토큰을 FluoBlog의 모든 행동을 제어하는 강력한 액세스 제어 시스템으로 바꿀 것입니다.

### Token Revocation and Whitelisting
JWT는 상태 비저장 방식이지만, 사용자가 비밀번호를 변경하거나 기기 분실을 신고하는 등 특정 보안 요구 사항에 따라 토큰이 자연적으로 만료되기 전에 폐기해야 할 수도 있습니다. Fluo는 하이브리드 접근 방식을 통해 이러한 시나리오를 지원합니다. Redis와 같은 빠른 인메모리 저장소에 **폐기 목록(Revocation List)**(또는 거부 목록)을 구현할 수 있습니다. 토큰을 수락하기 전에 `JwtVerifier`는 해당 목록에서 토큰의 고유 ID(`jti`)를 확인합니다.

또는 극도로 민감한 시스템의 경우, 저장소에 명시적으로 존재하는 토큰만 유효한 것으로 간주하는 **화이트리스팅(Whitelisting)** 전략을 사용할 수도 있습니다. 이는 특정 라우트에 대해 JWT를 사실상 상태 유지 메커니즘으로 전환하면서도 다른 라우트에 대해서는 상태 비저장성을 유지하게 합니다. Fluo의 모듈형 아키텍처를 사용하면 서비스별 또는 요청별로 이 동작을 전환할 수 있어, 기본적으로는 고성능 상태 비저장성을 활용하고 보안 수준이 높아야 할 때는 완벽한 제어력을 확보하는 등 두 세계의 장점을 모두 누릴 수 있습니다.

### Scaling Auth with Multi-Tenancy
단일 Fluo 애플리케이션이 여러 조직에 서비스를 제공하는 멀티테넌트(multi-tenant) 환경에서는 JWT 설정이 더욱 유연해야 합니다. 각 테넌트는 자신만의 서명 비밀 키를 가지거나 외부 ID 공급자를 사용할 수 있습니다. `JwtModule`은 동적 설정 프로바이더를 지원하여, 요청 컨텍스트(예: 커스텀 헤더의 테넌트 ID)를 기반으로 올바른 서명 및 검증 설정을 찾아낼 수 있게 해줍니다.

이러한 정교함이 Fluo를 SaaS 백엔드를 위한 전문적인 선택지로 만듭니다. 단일 전역 비밀 키로 간단하게 시작하여, Fluo 생태계를 벗어나지 않고도 복잡한 다중 공급자, 다중 테넌트 인증 시스템으로 확장할 수 있습니다. 앞서 논의한 `JwtPrincipal` 정규화는 여기서 특히 강력한데, 통합하는 ID 소스가 얼마나 많든 상관없이 멀티테넌트 비즈니스 로직에 안정적인 인터페이스를 제공하기 때문입니다.

### Cryptographic Agility
암호화 표준이 진화함에 따라 애플리케이션도 함께 진화해야 합니다. Fluo의 `@fluojs/jwt`는 **암호화 민첩성(Cryptographic Agility)**을 고려하여 설계되어, 가동 중단 없이 서명 알고리즘과 키를 교체할 수 있습니다. `JwtModule`이 동시에 여러 개의 활성 검증 키를 지원하도록 설정할 수 있습니다. 이를 통해 전환 기간 동안 이전 키로 서명된 유효한 기존 토큰을 여전히 수용하면서, 새로운 더 강력한 알고리즘(예: `HS256`에서 `EdDSA`로 이동)으로 새로운 토큰을 발급할 수 있습니다.

이러한 "무중단" 키 교체는 복원력 있는 엔터프라이즈급 시스템의 특징입니다. 이는 보안 업그레이드가 사용자의 불만 요인이 되지 않도록 보장합니다. 아키텍처 수준에서 변화의 필요성을 예측함으로써, Fluo는 여러분의 장기적인 유지보수 비용과 사용자의 보안을 보호합니다.

### Auth in a Serverless World
AWS Lambda나 Cloudflare Workers와 같은 서버리스 환경에서는 콜드 스타트(cold start)와 실행 시간이 매우 중요합니다. 무거운 네이티브 의존성이나 동기식 파일 I/O에 의존하는 전통적인 인증 라이브러리는 성능을 크게 저하시킬 수 있습니다. Fluo의 JWT 패키지는 가볍고 네이티브한 Web Crypto API를 기반으로 구축되어 이러한 환경에 최적화되어 있습니다. 콜드 스타트 오버헤드를 최소화하고 레거시 라이브러리에 필요한 시간의 아주 일부분만으로 검증을 실행합니다.

이러한 성능 우위는 규모가 커질수록 더욱 두드러집니다. 수백만 개의 요청을 처리할 때 모든 인증 체크에서 절약된 매 밀리초는 상당한 비용 절감과 사용자에게 더 쾌적한 경험으로 이어집니다. 대규모 Kubernetes 클러스터에서 실행하든 작은 에지 워커에서 실행하든, Fluo의 인증 로직은 가볍고 빠르며 안전하게 유지됩니다.

### Implementing a Unified Auth Flow
Chapter 15에서 가드에 대해 자세히 다루겠지만, Fluo 애플리케이션에서 JWT 인증이 일반적으로 어떻게 적용되는지 이해하는 것이 도움이 됩니다. 대부분의 프로젝트는 보호가 필요한 라우트에 `@UseAuth('jwt')`를 붙이고, 검증이 끝난 principal을 `RequestContext.principal`로 읽는 흐름을 사용합니다.

이러한 principal 중심 방식은 특정 라우트는 공개(사용자 정보가 선택 사항)로 유지하면서 다른 라우트는 엄격하게 보호할 수 있게 해줍니다. 이 로직을 중앙 집중화함으로써 전체 API 표면에서 인증이 일관되게 처리되도록 보장할 수 있고, 컨트롤러는 raw request 객체 대신 검증된 principal만 신뢰하면 되므로 더 단순해집니다. 이 패턴은 "기본 보안 적용(Secure by Default)" 원칙을 장려하여, 개발자가 실수로 보호되지 않은 엔드포인트를 통해 민감한 데이터를 노출하는 것을 방지합니다.

### Debugging JWT Issues
문제가 발생했을 때(예: 클라이언트에서 예기치 않은 401 에러가 보고될 때) 적절한 디버깅 도구를 갖추는 것은 필수적입니다. Fluo의 `JwtModule`은 개발 모드에서 실행될 때 상세한 내부 로그를 제공합니다. 만료된 타임스탬프, 서명 불일치, 또는 유효하지 않은 발급자 등 토큰 검증이 실패한 정확한 이유를 확인할 수 있습니다.

클라이언트 측 디버깅을 위해서는 `jwt.io`와 같은 도구를 사용하여 토큰 페이로드를 검사하는 것이 좋습니다(민감한 프로덕션 데이터로 이 작업을 수행하지 마십시오!). 서버 측에서는 Fluo의 미들웨어를 사용하여 들어오는 요청의 `JwtPrincipal`을 로깅함으로써 클레임 매핑이 예상대로 작동하는지 확인할 수 있습니다. 이러한 투명성은 인증과 흔히 연관되는 "마법" 요소를 줄여주고 팀이 문제를 신속하게 해결할 수 있도록 지원합니다.

### Integrating with External Identity Providers
조직에서 Auth0, Okta, 또는 AWS Cognito와 같은 외부 ID 공급자(IdP)를 사용하는 경우, Fluo의 JWT 모듈이 주요 통합 지점이 됩니다. 대부분의 현대적인 IdP는 JSON Web Key Set(JWKS)을 사용하여 검증할 수 있는 표준 JWT를 발행합니다. Fluo는 IdP로부터 공개 키를 자동으로 가져오고 캐싱하는 내장 `JwksVerifier`를 제공하며, 배후에서 키 로테이션을 원활하게 처리합니다.

이러한 정교함이 Fluo를 SaaS 백엔드를 위한 전문적인 선택지로 만듭니다. 단일 전역 비밀 키로 간단하게 시작하여, Fluo 생태계를 벗어나지 않고도 복잡한 다중 공급자, 다중 테넌트 인증 시스템으로 확장할 수 있습니다. 앞서 논의한 `JwtPrincipal` 정규화는 여기서 매우 중요한데, IdP로부터 오는 잠재적으로 복잡하고 독점적인 클레임들을 여러분의 Fluo 서비스가 이해할 수 있는 안정적인 형식으로 매핑할 수 있게 해주기 때문입니다. 주요 ID 공급자의 보안성과 표준 우선 프레임워크의 개발자 경험이라는 두 세계의 장점을 모두 얻을 수 있습니다.

### The Future of Auth: Passkeys and WebAuthn
미래를 내다볼 때, 인증은 비밀번호에서 벗어나 패스키(WebAuthn)와 같이 더 안전한 방식으로 이동하고 있습니다. JWT는 계속해서 세션 메커니즘으로 사용되겠지만, 해당 토큰을 *발급*하는 방식은 변하고 있습니다. Fluo의 모듈형 아키텍처는 이러한 변화에 준비되어 있습니다. 여러분은 `AuthService`에서 WebAuthn 기반의 로그인 흐름을 구현하면서도, 사용자가 검증되면 동일한 `JwtSigner`를 사용하여 토큰을 발급할 수 있습니다.

"신원을 확인하는 방법(인증)"과 "세션을 유지하는 방법(JWT)"을 분리함으로써, Fluo는 하부 기술이 진화하더라도 여러분의 애플리케이션이 현대적이고 안전하게 유지되도록 보장합니다. 이러한 아키텍처적 선견지명은 Fluo 생태계에 대한 여러분의 투자를 보호하고 끊임없이 변화하는 온라인 위협으로부터 사용자를 안전하게 지켜줍니다.

### Token Security: A Multi-Layered Approach
인증은 한 번 설정하고 끝내는 기능이 아닙니다. 지속적인 모니터링과 개선이 필요합니다. JWT를 구현하는 것 외에도 다음과 같은 계층을 추가하는 것을 고려해 보십시오.
- **속도 제한 (Chapter 16)**: 로그인 및 리프레시 엔드포인트에 대한 무차별 대입 공격을 방지합니다.
- **모니터링 (Chapter 19)**: 서로 다른 지리적 지역에서의 인증 패턴 이상 징후를 감지합니다.
- **감사(Auditing)**: 관리자 작업 및 고위험 작업에 대한 기록을 유지합니다.

이러한 모듈들을 결합함으로써 여러분은 우발적인 에러와 의도적인 공격 모두에 복원력이 있는 "요새 API"를 구축하게 됩니다. Fluo는 모든 조각을 제공하며, Chapter 14는 그 토대를 제공합니다.

### Handling Logout and Session Invalidation
상태 비저장 JWT 환경에서의 로그아웃은 전통적인 세션 방식과 다릅니다. 서버가 상태를 저장하지 않기 때문에 단순히 세션을 "삭제"할 수 없습니다. 사용자를 로그아웃시키는 주요 방법은 클라이언트에게 토큰을 삭제하도록 지시하는 것입니다. 하지만 진정한 보안을 위해서는 데이터베이스나 Redis 저장소에서 리프레시 토큰을 무효화해야 합니다.

리프레시 토큰을 제거함으로써, 비록 수명이 짧은 액세스 토큰이 몇 분 동안 더 유효하더라도 현재 토큰이 만료된 후에 사용자가 새로운 토큰을 얻을 수 없도록 보장할 수 있습니다. 로그아웃에 대한 이러한 다층적인 접근 방식은 상태 비저장성의 성능 이점을 희생하지 않으면서도 높은 수준의 보안을 제공합니다. Fluo의 `JwtModule`은 `AuthService` 내에서 이러한 무효화 로직을 쉽게 구현할 수 있는 훅을 제공합니다.

### Token Expiration and User Experience
토큰 만료 관리 보안과 사용자 경험 사이의 섬세한 균형입니다. 토큰이 너무 빨리 만료되면 사용자는 잦은 재로그인으로 인해 좌절감을 느낄 것입니다. 너무 오래 지속되면 오용의 위험이 커집니다. Fluo의 이중 토큰 패턴(액세스 + 리프레시)은 이러한 딜레마에 대한 업계의 해답입니다.

프론트엔드 애플리케이션은 토큰 만료를 조용히(silently) 처리하도록 설계되어야 합니다. 클라이언트 측에서 "인터셉터"를 사용하면 자동으로 401 에러를 감지하고, 리프레시 엔드포인트를 호출하며, 새로운 액세스 토큰으로 원래 요청을 재시도할 수 있습니다. 이 모든 과정이 사용자가 눈치채지 못하는 사이에 진행됩니다. 이러한 원활한 경험은 전문적인 애플리케이션과 아마추어 애플리케이션을 구분하는 차이입니다. Fluo는 이러한 정교한 클라이언트 측 패턴을 지원하는 백엔드 인프라를 제공하여, 보안 조치가 훌륭한 사용자 경험을 방해하지 않도록 보장합니다.

### Secure Token Transmission
JWT의 보안은 그것이 전달되는 채널의 보안만큼만 강력합니다. 항상 HTTPS를 통해 Fluo API를 제공해야 합니다. 전송 계층에서의 암호화가 없다면, 공격자는 클라이언트와 서버 사이에서 전송되는 토큰을 훔치기 위해 "중간자 공격(man-in-the-middle attack)"을 수행할 수 있습니다. 일단 도난당한 JWT는 만료될 때까지 누구나 사용자로 가장하는 데 사용될 수 있습니다.

HTTPS 외에도, 프론트엔드에서 승인되지 않은 스크립트가 실행되어 로컬 저장소에서 토큰을 훔치는 것을 방지하기 위해 "콘텐츠 보안 정책(CSP)" 헤더를 사용하는 것을 고려해 보십시오. 보안은 체인과 같으며 인증 로직은 그중 하나의 고리에 불과합니다. 이 책에서 설명하는 포괄적인 보안 패턴을 따름으로써 여러분은 Fluo 애플리케이션의 모든 고리가 강력하고 신뢰할 수 있도록 보장할 수 있습니다.

### Choosing the Right Algorithm
소규모 프로젝트의 경우 `HS256`(대칭 서명)이 설정하기 쉽지만, 상용 수준의 모듈형 아키텍처를 위해서는 `RS256`(비대칭 서명)이 권장되는 선택입니다. `RS256`을 사용하면 인증 서비스가 토큰 서명을 위한 비밀 키를 보유하는 반면, 다른 마이크로서비스들은 검증을 위한 공개 키만 있으면 됩니다. 이러한 관심사의 분리는 설령 보조 서비스가 침해되더라도 공격자가 비밀 키를 가지고 있지 않기 때문에 새로운 토큰을 발행할 수 없음을 의미합니다.

Fluo의 `JwtModule`을 사용하면 이러한 알고리즘 간의 전환을 간단한 설정만으로 수행할 수 있습니다. 이러한 "알고리즘 민첩성" 덕분에 빠르게 시작하고 애플리케이션의 복잡성이 증가함에 따라 보안 태세를 진화시킬 수 있습니다. 이는 Fluo가 오늘 가치를 제공하면서도 동시에 미래를 위해 구축할 수 있도록 돕는 또 다른 방식입니다.

### Authentication vs. Authorization
**인증(Authentication)**(누구인지 아는 것)과 **인가(Authorization)**(무엇을 할 수 있는지 아는 것)를 구분하는 것이 중요합니다. Chapter 14는 JWT를 통한 인증에 집중합니다. 사용자가 인증되고 그 신원이 `JwtPrincipal`로 표현되면, 다음 단계는 해당 사용자가 특정 작업을 수행할 권한이 있는지 결정하는 것입니다.

Fluo에서 이는 Chapter 15에서 살펴볼 **가드**와 **메타데이터**를 통해 처리됩니다. 이 두 가지 관심사를 분리함으로써 Fluo는 깔끔하고 모듈화된 보안 아키텍처를 제공합니다. 인증 로직은 신원 확인에 집중하고, 인가 로직은 비즈니스 규칙 강제에 집중합니다. 이러한 분리는 보안 코드를 테스트하고 감사하며 유지 관리하기 쉽게 만듭니다.

### Handling Token Replay Attacks
토큰 재전송 공격(token replay attack)은 공격자가 유효한 JWT를 가로채서 정당한 사용자의 세션이 끝난 후에 다시 사용하려고 시도할 때 발생합니다. 짧은 만료 시간으로도 이러한 위험을 완화할 수 있지만, 토큰 페이로드에 **Nonce**(한 번만 사용되는 고유 번호)를 사용하여 보안을 더욱 강화할 수 있습니다. 이러한 Nonce를 빠른 데이터 저장소에서 추적함으로써, 서버는 각 토큰이 의도된 목적으로만 특정 컨텍스트 내에서 한 번만 사용되도록 보장할 수 있습니다.

Fluo의 인증 유틸리티는 Nonce 관리를 위한 내장 지원을 제공하므로, 복잡한 상용구 코드 없이도 이러한 고급 보안 조치를 쉽게 구현할 수 있습니다. 이러한 수준의 보호는 금융 거래나 관리자 설정 변경과 같은 고가치 작업에서 특히 중요합니다. 이를 통해 서명된 완벽하게 유효한 토큰이라 하더라도 공격자에 의해 "재전송"되어 의도하지 않은 부수 효과를 일으키는 일을 방지할 수 있습니다.

### The Role of Encryption (JWE)
표준 JWS(서명된 JWT)가 무결성을 제공한다면, 때로는 토큰을 소유한 사람에게도 내용을 숨겨야 하는 **기밀성(Confidentiality)**이 필요할 때가 있습니다. 바로 여기서 **JSON Web Encryption (JWE)**이 등장합니다. JWE는 페이로드를 암호화하여 올바른 복호화 키를 가진 당사자만 내부 데이터를 볼 수 있도록 합니다.

Fluo의 유연한 아키텍처를 사용하면 매우 민감한 사용 사례를 위해 서명된 JWT를 JWE 봉투 안에 감쌀 수 있습니다. 이러한 "중첩된 JWT" 방식은 (서명을 통한) 무결성과 (암호화를 통한) 기밀성을 동시에 제공합니다. 추가적인 암호화 작업으로 인해 약간의 성능 오버헤드가 발생하지만, 개인정보(PII)를 항상 최고 수준의 엄격함으로 보호해야 하는 의료나 금융과 같은 산업에서는 필수적인 도구입니다.

### Token Size and Performance Trade-offs
JWT에 더 많은 클레임, Nonce, 그리고 잠재적으로 암호화까지 추가할수록 토큰의 크기는 커집니다. 토큰이 커진다는 것은 모든 요청에서 더 많은 대역폭이 소모되고, 인코딩, 디코딩 및 암호학적 검증에 더 많은 CPU 사이클이 사용됨을 의미합니다. 빈번한 호출이 발생하는 API에서 이러한 작은 오버헤드는 상당한 성능 병목 현상으로 축적될 수 있습니다.

고성능 Fluo 애플리케이션을 유지하려면 항상 토큰 구조를 감사하십시오. "이 클레임이 정말 토큰에 있어야 하는가, 아니면 필요할 때 캐시에서 조회할 수 있는가?"라고 스스로 물어보십시오. 토큰을 가볍고 집중된 상태로 유지함으로써, 인증 레이어가 느리고 번거로운 장벽이 아닌 빠르고 효율적인 문지기 역할을 하도록 보장할 수 있습니다. 성능도 하나의 기능이며, 보안의 세계에서 효율적인 코드는 공격 표면을 줄이고 서비스 거부 취약점의 가능성을 최소화하기 때문에 종종 더 안전합니다.

### Identity Federation with OIDC
대규모 기업의 경우, 신원은 여러 부서나 심지어 서로 다른 회사 간에 관리되기도 합니다. **OpenID Connect (OIDC)**는 이러한 "신원 연합(Identity Federation)"을 위한 표준 프로토콜입니다. OIDC는 JWT를 기반으로 구축되어 ID 공급자가 의존 당사자(여러분의 Fluo 앱)에게 인증된 사용자에 대해 알려주는 표준화된 방식을 제공합니다.

Fluo의 인증 생태계는 OIDC와 호환되도록 설계되었습니다. 특히 Principal 정규화 및 JWKS 검증과 관련하여 이 장에서 설명한 모범 사례를 따름으로써, 여러분은 전 세계적인 신원 네트워크에 쉽게 참여할 수 있는 애플리케이션을 구축하게 됩니다. 이는 여러분의 인증 아키텍처의 미래를 보장하며, 최소한의 마찰로 현대적인 싱글 사인온(SSO) 시스템과 통합할 수 있게 해줍니다. 여러분은 전 세계 인터넷이 신뢰하는 표준 위에서 구축하고 있는 것입니다.

### Secure Token Lifecycles in Fluo
토큰의 발급부터 만료, 그리고 최종적인 폐기에 이르기까지 토큰의 생명주기를 관리하는 것은 `AuthService`의 핵심 책임입니다. Fluo의 생명주기 훅을 사용하여 이러한 관리를 자동화하십시오. 예를 들어, `OnModuleInit` 훅을 사용하여 외부 IdP로부터 공개 키를 미리 가져옴으로써 시스템이 시작되자마자 토큰을 검증할 준비를 갖출 수 있습니다.

토큰을 단순히 정적인 문자열이 아니라 시작, 중간, 끝이 있는 동적인 엔티티로 생각함으로써 여러분은 더 견고하고 복원력 있는 인증 시스템을 구축하게 됩니다. 이러한 총체적인 접근 방식이 보안을 사후 고려 사항으로 취급하는 프레임워크와 Fluo를 구분 짓는 차이입니다. 우리는 도구를 제공하고, 여러분은 끊임없이 진화하는 디지털 세계에서 사용자와 데이터를 보호하기 위한 아키텍처적 비전을 제공합니다.

### Handling Large Payloads with Token Compression
앞서 논의했듯이 토큰 크기는 중요한 성능 요소입니다. 하지만 JWT에 대량의 메타데이터를 포함해야만 하는 경우도 있습니다. 이러한 시나리오를 위해 Fluo는 **토큰 압축(Token Compression)**을 지원합니다. 서명 및 인코딩 전에 `zlib`나 `deflate`와 같은 알고리즘을 사용함으로써 최종 JWT 문자열의 크기를 크게 줄일 수 있습니다.

압축은 인코딩과 디코딩에 약간의 CPU 오버헤드를 추가하지만, 특히 느린 모바일 네트워크를 사용하는 사용자의 경우 네트워크 대역폭 절감 효과가 그 비용보다 큰 경우가 많습니다. Fluo의 `JwtModule`을 사용하면 간단한 설정 플래그 하나로 압축을 활성화할 수 있으며, 상태 비저장 아키텍처의 성능을 희생하지 않고도 복잡한 페이로드를 처리할 수 있는 원활한 방법을 제공합니다. 이는 Fluo가 대규모 프로덕션 애플리케이션의 실질적인 요구 사항을 어떻게 예측하는지 보여주는 또 다른 예입니다.

### Security Headers and Token Protection
JWT 기반 인증의 보안은 토큰 자체에서 끝나지 않습니다. 브라우저가 요청을 어떻게 처리하는지도 고려해야 합니다. `Strict-Transport-Security` (HSTS)와 같은 보안 헤더를 사용하면 브라우저가 암호화된 채널을 통해서만 서버와 통신하도록 보장하여 의도치 않은 토큰 유출을 방지할 수 있습니다. 또한 `X-Content-Type-Options: nosniff` 헤더는 브라우저가 API 응답을 잘못 해석하는 것을 방지하여 잠재적인 공격에 대한 또 다른 방어 계층을 추가합니다.

Fluo는 애플리케이션 전체에서 이러한 보안 헤더를 관리할 수 있는 전용 미들웨어를 제공합니다. `AppModule`에서 이러한 설정을 한 번만 구성하면 API의 모든 응답이 일반적인 웹 기반 공격 벡터로부터 강화되도록 보장할 수 있습니다. 보안이 철저한 API란 요청/응답 생명주기를 총체적으로 바라보는 것이며, Fluo makes it easy to follow these industry best practices.

### Token Security in Native Mobile Apps
Fluo를 백엔드로 하여 모바일 앱을 구축할 때 토큰을 저장하는 방식은 웹 브라우저와 근본적으로 다릅니다. 모바일에는 쿠키나 LocalStorage가 동일한 의미로 존재하지 않습니다. 대신 iOS의 **Keychain**이나 Android의 **EncryptedSharedPreferences**와 같은 플랫폼의 네이티브 보안 기능을 사용해야 합니다. 이러한 저장소는 하드웨어 수준에서 보호되며, 무단 접근으로부터 민감한 자격 증명을 보호하도록 특별히 설계되었습니다.

Fluo 측에서는 인증 패턴이 일관되게 유지됩니다. 토큰이 브라우저 쿠키에서 오든 모바일 헤더에서 오든 `JwtVerifier`는 이를 동일하게 처리합니다. 이러한 일관성 덕분에 웹, 모바일, 그리고 IoT 기기에 이르기까지 모든 클라이언트를 지원하는 하나의 안전한 백엔드를 구축할 수 있습니다. 표준 우선(standard-first) 원칙에 집중함으로써 Fluo는 여러분이 어떤 플랫폼에서도 탁월한 성과를 낼 수 있도록 필요한 유연성을 제공합니다.

### Future-Proofing with Standard Decorators
Fluo의 가장 독특한 특징 중 하나는 **TC39 표준 데코레이터(Standard Decorators)**에 의존한다는 점입니다. 실험적인 기능(experimental decorators)을 사용하는 레거시 프레임워크와 달리, Fluo의 인증 데코레이터(`@UseGuards` 또는 `@CurrentUser` 등)는 JavaScript의 미래를 기반으로 구축되었습니다. 이는 여러분의 인증 로직이 더 안정적일 뿐만 아니라 런타임에 무거운 메타데이터 리플렉션에 의존하지 않기 때문에 더 높은 성능을 발휘함을 의미합니다.

`AuthController`와 `AuthService`를 구축할 때 여러분은 오랫동안 유지될 코드를 작성하고 있다는 확신을 가질 수 있습니다. 공식적인 언어 경로에 맞춤으로써 Fluo는 여러분의 보안 투자가 앞으로도 오랫동안 가치를 유지하도록 보장합니다. 여러분은 단순히 블로그를 만드는 것이 아니라, TypeScript 개발의 다음 10년을 위해 설계된 프레임워크를 마스터하고 있는 것입니다.

### Conclusion of Part 3, Chapter 14
인증은 모든 소프트웨어 시스템에서 가장 첫 번째이자 가장 중요한 관문입니다. `@fluojs/jwt`를 통해 JWT를 마스터함으로써 여러분은 전문적인 Fluo 개발자가 되는 큰 걸음을 내디뎠습니다. 상태 비저장 철학, Principal 정규화의 강력함, 그리고 안전한 토큰 생명주기의 필요성에 대해 배웠습니다.

하지만 누군가를 아는 것만으로 충분하지 않습니다. 이제 이 정체성을 사용하여 기능에 대한 접근을 제어하는 방법을 배워야 합니다. 15장에서는 FluoBlog 전체에서 역할 기반 및 정책 기반 권한 부여를 적용하기 위해 **Passport 전략**과 **Custom Guard**를 구현하여 이 기초를 다질 것입니다. 진정으로 안전한 API로 향하는 여정은 계속됩니다.

### Advanced: Handling Token Claims with Zod
Fluo의 `JwtPrincipal`은 일반적인 클레임에 대한 정규화된 뷰를 제공하지만, 토큰 내에서 커스텀 도메인별 데이터를 처리해야 하는 경우가 많습니다. Fluo의 "표준 우선" 및 타입 안전 철학을 유지하려면 검증 프로세스 중에 이러한 커스텀 클레임을 검증하기 위해 **Zod**를 사용하는 것이 좋습니다. 토큰 페이로드에 대한 Zod 스키마를 정의함으로써 애플리케이션이 엄격한 데이터 요구 사항을 충족하는 토큰만 처리하도록 보장할 수 있습니다.

Zod와의 이러한 통합은 잘못 형성되거나 악의적인 토큰에 대한 추가 방어 계층을 제공합니다. 토큰에 유효하지 않은 데이터가 포함되어 있으면 비즈니스 로직이 클레임에 도달하기 전에 스키마 검증이 실패합니다. 이러한 "Validate, Don't Parse" 접근 방식은 현대적인 TypeScript 개발의 특징이며 인증 레이어를 가능한 한 견고하게 보장합니다. Fluo의 `JwtModule`을 사용하면 이러한 커스텀 검증 스키마를 쉽게 연결하여 모든 토큰 관련 작업에 대해 깔끔하고 타입이 지정된 인터페이스를 제공할 수 있습니다.

### Performance Tuning: Caching Public Keys
비대칭 서명(RS256)을 사용할 때 Fluo 애플리케이션은 외부 JWKS 엔드포인트에서 공개 키를 빈번하게 가져와야 합니다. 이것이 성능 병목 현상이나 단일 장애 지점이 되는 것을 방지하기 위해 Fluo는 **자동 JWKS 캐싱**을 구현합니다. `JwksVerifier`는 가져온 키를 고성능 인메모리 캐시에 저장하여 반복적인 네트워크 호출의 필요성을 줄입니다.

애플리케이션의 요구 사항에 맞게 캐시 기간과 백그라운드 갱신 간격을 커스터마이징할 수 있습니다. 예를 들어 매우 민감한 서비스의 경우 캐시 시간을 짧게 설정하거나 보안 위험이 낮은 내부 도구의 경우 캐시 시간을 길게 설정할 수 있습니다. 이 지능형 캐싱은 인터넷을 통해 외부 ID 공급자와 상호 작용할 때도 인증 레이어가 빠르고 반응이 좋게 유지되도록 보장합니다. 이는 Fluo가 여러분을 위해 분산 시스템의 복잡한 현실을 처리하는 또 다른 방식입니다.

### Securing the Login Flow: Rate Limiting
가장 강력한 JWT 구현이라 하더라도 로그인 엔드포인트에 대한 무차별 대입 공격에는 취약할 수 있습니다. FluoBlog 사용자를 보호하려면 항상 인증 로직을 **속도 제한**(16장에서 자세히 살펴봄)과 결합해야 합니다. IP 주소나 사용자 계정당 로그인 시도 횟수를 제한함으로써 자동화된 비밀번호 추측 시도를 효과적으로 저지할 수 있습니다.

Fluo에서 `AuthController`에 속도 제한을 추가하는 것은 데코레이터를 추가하는 것만큼 간단합니다. 이러한 모듈형 접근 방식을 통해 보안 관심을 분리하면서도 통합된 상태로 유지할 수 있습니다. `AuthService`는 신원 확인 로직에 집중하고, `ThrottlerGuard`는 해당 로직을 남용으로부터 보호하는 데 집중합니다. 이들은 함께 전체의 합보다 큰 포괄적인 보안 태세를 구축합니다. 이러한 심층 방어 전략은 모든 프로덕션 API에 필수적입니다.

### Identity Portability and Vendor Lock-in
표준 JWT 및 Web Crypto API를 기반으로 구축함으로써 Fluo는 여러분의 인증 로직이 **이식 가능함**을 보장합니다. 애플리케이션을 한 클라우드 프로바이더에서 다른 프로바이더로, 혹은 한 ID 프로바이더에서 다른 프로바이더로 옮기기로 결정하더라도 핵심 Fluo 코드는 변경되지 않습니다. 독점적인 SDK나 벤더별 인증 메커니즘에 갇히지 않습니다.

이러한 이식성은 Fluo 프레임워크의 핵심 가치입니다. 우리는 여러분이 자신의 코드와 아키텍처적 결정을 소유해야 한다고 믿습니다. 이 장에서 설명한 "표준 우선" 접근 방식을 따름으로써 여러분은 클라우드 산업의 변화하는 환경에 적응할 수 있는 복원력 있는 시스템을 구축하고 있는 것입니다. 여러분은 단지 오늘을 위해 구축하는 것이 아니라 다음 10년을 위해 구축하고 있습니다.

### Auth Auditing and Compliance
많은 산업에서 인증 이벤트의 상세한 감사 로그를 유지하는 것은 법적 요구 사항입니다(예: GDPR, HIPAA 또는 SOC2 준수). Fluo의 `JwtModule`은 조직의 감사 시스템과 쉽게 통합할 수 있는 훅을 제공합니다. 성공적인 로그인, 실패한 시도, 토큰 갱신 등을 IP 주소 및 사용자 에이전트와 같은 메타데이터와 함께 기록할 수 있습니다.

Fluo 애플리케이션 내에서 이러한 로깅을 중앙 집중화함으로써 누가 여러분의 데이터에 언제 접근하고 있는지에 대한 명확하고 감사 가능한 기록을 생성할 수 있습니다. 이러한 투명성은 사용자와의 신뢰를 유지하고 규제 의무를 이행하는 데 필수적입니다. 보안은 단지 기술적인 과제가 아니라 책임의 문제입니다. Fluo는 여러분이 사용자와 커뮤니티에 책임을 다할 수 있도록 필요한 인프라를 제공합니다.

### Best Practices for Token Storage
웹에서 JWT를 어디에 저장할지는 자주 논의되는 주제입니다. LocalStorage는 사용하기 쉽지만 크로스 사이트 스크립팅(XSS)에 취약합니다. 공격자가 여러분의 페이지에서 JavaScript를 실행할 수 있다면 토큰을 훔칠 수 있습니다. 앞서 논의한 것처럼 브라우저를 위한 가장 안전한 옵션은 **httpOnly** 쿠키입니다.

모바일 앱과 같은 비브라우저 환경에서는 이러한 걱정이 없지만 물리적 접근으로부터 토큰을 보호해야 합니다. 플랫폼별 보안 저장소를 사용함으로써 기기를 분실하더라도 토큰이 암호화되어 접근할 수 없도록 보장할 수 있습니다. 이러한 플랫폼별 뉘앙스를 이해하는 것은 시니어 백엔드 엔지니어가 되는 핵심적인 부분입니다. 모든 클라이언트 유형에 대해 최고 수준의 보안을 지원할 수 있도록 Fluo API를 유연하게 설계해야 합니다.

### Token Revocation Strategies: Beyond the Denylist
거부 목록은 효과적이지만 트래픽이 많은 시스템에서는 매우 커질 수 있습니다. 대안은 **버전 관리 토큰(Versioned Token)** 전략입니다. JWT에 `version` 클레임을 추가하고 각 사용자의 현재 유효한 버전을 데이터베이스에 저장하면, 버전 번호를 늘리는 것만으로 해당 사용자의 모든 활성 토큰을 무효화할 수 있습니다.

토큰이 검증될 때 `JwtVerifier`는 클레임의 버전이 데이터베이스(또는 빠른 캐시)의 버전과 일치하는지 확인합니다. 이를 통해 개별 토큰 ID를 모두 추적할 필요 없이 비밀번호 재설정이나 보안 침해 시 사용자의 모든 세션을 거의 즉시 폐기할 수 있습니다. Fluo의 정규화 레이어는 이러한 버전 클레임을 추가하고 확인하는 것을 쉽게 만들어 대규모 세션 관리를 위한 강력한 도구를 제공합니다.

### Designing for Federated Identity
현대 웹에서 사용자들은 또 다른 비밀번호를 만드는 대신 "Google로 로그인"이나 GitHub를 선호하는 경우가 많습니다. 이를 **연합 신원(Federated Identity)**이라고 합니다. OAuth2 프로바이더와 통합함으로써 Fluo 애플리케이션은 초기 인증을 신뢰할 수 있는 제3자에게 위임할 수 있습니다.

Once the third party verifies the user, they provide a token that your `AuthService` can then exchange for your own Fluo-native JWT. This allows you to leverage the security infrastructure of tech giants while still maintaining control over your own session management and role-based access control. Fluo's `JwtPrincipal` is designed exactly for this kind of integration, providing a stable target for identity data regardless of its origin. This approach reduces friction for your users and improves your security by offloading password management to specialists.

### The Role of Refresh Token Rotation in Security
리프레시 토큰 로테이션에 대해 언급했지만 프로덕션 환경에서의 중요성은 아무리 강조해도 지나치지 않습니다. 이는 도난당한 리프레시 토큰이 무기한 접근을 유지하는 데 사용되는 것에 대한 주요 방어책입니다. 모든 액세스 토큰 갱신 시마다 새로운 리프레시 토큰을 발행함으로써 공격자에게 "움직이는 타겟"을 만듭니다.

공격자가 오래된 리프레시 토큰을 사용하려고 시도하면 Fluo의 감지 메커니즘이 재사용을 식별하고 즉시 해당 계정의 전체 세션 제품군을 차단합니다. 이러한 "침해 감지"는 Fluo의 인증 패키지에 기본으로 포함된 고급 보안 기능입니다. 이는 보안의 가장 어려운 측면을 처리하여 여러분이 고민하지 않도록 전문적인 도구를 제공하려는 우리의 약속을 보여줍니다. Fluo가 정문을 지키고 있다는 것을 알고 비즈니스 로직에 집중할 수 있습니다.

### Implementing Secure Default Claims
Fluo 애플리케이션에서 발행하는 모든 JWT에는 표준 보안 클레임 세트가 포함되어야 합니다. 최소한 `iat` (발행 시간), `nbf` (활성 시점), `exp` (만료 시간)가 포함됩니다. 이 클레임들은 토큰의 "시간적 유효성"을 정의하여 발행 전이나 만료 후에 사용되지 않도록 보장합니다.

Fluo's `DefaultJwtSigner` handles these claims automatically, but it's important for you to understand why they exist. They prevent a variety of attacks, including those involving system clock desynchronization or historical token reuse. By following these industry standards, Fluo ensures that your tokens are not only secure within your application but also compatible with the broader ecosystem of security tools and services. Security is about following the rules of the road, and Fluo is your expert guide.
