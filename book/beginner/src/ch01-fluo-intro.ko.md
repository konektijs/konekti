# Chapter 1. Introduction to fluo and Design Philosophy

## Learning Objectives
- 현대적인 TypeScript 생태계에서 fluo가 해결하려는 핵심 문제를 이해합니다.
- 명시적 의존성 주입(DI)의 철학을 살펴봅니다.
- fluo의 런타임 중립적 아키텍처를 배웁니다.
- fluo가 TC39 Stage 3 표준 데코레이터를 채택한 이유를 이해합니다.
- fluo 생태계의 큰 지도를 파악합니다.
- 이 책의 중심 예제인 FluoBlog를 소개받습니다.

## Prerequisites
- TypeScript에 대한 기초 지식.
- 동작하는 Node.js 설치 환경.
- 백엔드 프레임워크가 어떤 방식으로 구성되는지에 대한 호기심.

## 1.1 The Problems fluo Solves

백엔드 프레임워크의 약속은 단순합니다. 개발 속도를 높이되 코드베이스를 취약하게 만들지 않는 것입니다. 하지만 실제로는 많은 프레임워크가 리플렉션, 컴파일러 플래그, 런타임 규칙에 복잡성을 숨기는 방식으로 속도를 얻습니다. 이런 방식은 소스 코드만 읽어서는 동작을 이해하기 어렵게 만듭니다.

fluo는 그 절충안에 대한 반응으로 설계되었습니다. 생산성은 유지하면서도 대규모 TypeScript 애플리케이션을 디버깅하기 어렵게 만드는 숨겨진 동작을 줄이는 것이 목표입니다.

큰 관점에서 fluo는 세 가지 반복적인 문제에 집중합니다.

1. **메타데이터 비대화**: 읽지도 않는 컴파일러 출력 메타데이터 때문에 애플리케이션이 비용을 지불해서는 안 됩니다.
2. **암시적인 마법**: 의존성 연결은 런타임의 간접 추론이 아니라 코드에서 보여야 합니다.
3. **플랫폼 종속성**: 비즈니스 로직은 Node.js, Bun, Deno, 엣지 런타임 사이를 이동해도 유지되어야 합니다.

### Why This Matters for a Beginner

백엔드 개발이 처음이라면 프레임워크의 마법은 처음에는 매우 편리하게 느껴질 수 있습니다. 첫날 작성해야 할 코드의 양을 줄여 주기 때문입니다.

문제는 그 다음에 나타납니다.

- 주입이 실패했을 때 의존성이 어디서 왔는지 알아야 합니다.
- 라우트가 이상하게 동작할 때 어떤 데코레이터가 등록했는지 알아야 합니다.
- 배포 환경이 바뀔 때 어떤 코드가 특정 런타임을 가정하는지 알아야 합니다.

fluo는 이런 개념의 명시적인 버전을 초반부터 가르쳐서 프로젝트가 커져도 머릿속 모델이 무너지지 않게 합니다.

### The Metadata Problem in Detail

레거시 데코레이터 기반 프레임워크는 보통 `emitDecoratorMetadata`와 리플렉션 라이브러리에 의존해 클래스 생성자 타입을 조사합니다.

이 접근에는 실제 비용이 있습니다.

- TypeScript가 많은 클래스에 대해 추가 메타데이터를 출력합니다.
- 번들이 불필요하게 커질 수 있습니다.
- 런타임이 더 많은 메타데이터를 읽어야 하므로 콜드 스타트가 느려질 수 있습니다.
- 의존성 그래프가 컴파일러 출력에 일부 숨겨지므로 디버깅이 어려워집니다.

fluo는 이런 가정을 제거합니다. 프레임워크가 “나중에 컴파일러가 모든 것을 알려주겠지”에서 출발하지 않습니다. 대신 “애플리케이션이 필요한 것을 직접 말해야 한다”에서 시작합니다.

다음과 같이 관점을 비교해 볼 수 있습니다.

| Question | Reflection-heavy approach | fluo approach |
| :--- | :--- | :--- |
| How is a dependency discovered? | By reading emitted metadata | By reading explicit registration and injection |
| What happens if metadata is missing? | Runtime failure or confusing container error | The code itself is incomplete and easier to inspect |
| What does the bundle contain? | Business logic plus metadata helpers | Mostly business logic and explicit framework contracts |

### The Magic Trap

모든 클래스를 자동 발견하고 별도 설명 없이 다 연결해 주는 프레임워크는 작은 데모에서 매우 멋져 보입니다.

하지만 대규모 시스템은 다릅니다.

실무 코드에서 숨겨진 동작은 보통 네 가지 마찰을 만듭니다.

- **추적의 마찰**: 프로바이더가 어디에서 등록되었는지 찾느라 시간을 씁니다.
- **리팩터링의 마찰**: 이름 변경이나 파일 이동이 규칙을 조용히 깨뜨릴 수 있습니다.
- **테스트의 마찰**: 의존성이 추론될 때 모킹이 어색해집니다.
- **온보딩의 마찰**: 새 팀원이 한 모듈씩 읽으며 시스템을 이해하기 어렵습니다.

fluo는 이런 장기 비용을 낮추기 위해 초반에 조금 더 눈에 보이는 구조를 선택합니다.

### A Quick Mental Model

fluo를 코드베이스를 수수께끼가 아니라 지도로 취급하는 프레임워크라고 생각해 보세요.

- 모듈은 경계를 설명합니다.
- 프로바이더는 재사용 가능한 로직을 설명합니다.
- 컨트롤러는 진입점을 설명합니다.
- 데코레이터는 의도를 설명합니다.
- 런타임 어댑터는 앱이 어디서 실행되는지 설명합니다.

소스 트리 안에서 이 요소들을 직접 가리킬 수 있다면 프레임워크는 제 역할을 하고 있는 것입니다.

### The Philosophy of Simplicity

fluo의 가장 중요한 개념 중 하나는 "기교(cleverness)보다 단순함(simplicity)이 낫다"는 것입니다.

많은 프레임워크가 제공하는 자동 파일 기반 라우팅이나 암시적 의존성 발견 같은 "영리한" 기능들은 문제가 생기기 전까지만 마법처럼 느껴집니다. 막상 문제가 생기면 왜 파일이 인식되지 않는지, 왜 특정 의존성이 undefined인지 추측에 의존해야 하죠.

fluo의 단순함은 명시성에서 옵니다. 여러분은 언제나 엔트리 포인트부터 개별 서비스까지의 경로를 추적할 수 있습니다. 이러한 예측 가능성은 큰 팀에서 일하거나 몇 달 만에 코드를 다시 볼 때 가장 강력한 힘이 됩니다.

### Why fluo is "Standard-First"

fluo가 "표준 우선(Standard-First)"이라는 것은 JavaScript와 TypeScript 언어 자체에 내장된 기능을 최우선으로 한다는 뜻입니다.

웹 생태계는 빠르게 움직이며, 독자적인 문법을 만들어내는 프레임워크들은 언어가 성숙해질 때 뒤처지곤 합니다. fluo는 TC39 데코레이터나 Web Streams API 같은 표준에 맞춤으로써 여러분의 지식이 다른 곳에서도 쓰일 수 있도록 보장합니다.

fluo를 배우는 것은 단순한 프레임워크 학습을 넘어, 더 나은 JavaScript/TypeScript 개발자가 되는 과정입니다. 여기서 배우는 모듈성, 캡슐화, 제어의 역전(IoC) 같은 패턴들은 소프트웨어 산업 전체에서 최고의 엔지니어들이 사용하는 것과 동일한 패턴입니다.

### The Community and Ecosystem

fluo는 단순한 패키지 모음이 아니라, 깨끗한 코드와 성능을 가치 있게 여기는 개발자들의 커뮤니티입니다.

이 책을 진행하면서 프레임워크의 모듈성이 커뮤니티로까지 확장되는 것을 보게 될 것입니다. GraphQL 통합부터 고급 로깅까지, 기여자들이 만든 다양한 전문 모듈이 준비되어 있습니다.

이러한 커뮤니티 중심의 접근 방식은 프레임워크가 실제 필요에 따라 진화하도록 만듭니다. 만약 필요한 기능이 없거나 기존 모듈을 개선할 방법이 있다면 여러분도 직접 기여할 수 있습니다. 이러한 개방적이고 협력적인 문화가 fluo 생태계를 더욱 활기차고 견고하게 만듭니다.

## 1.2 Explicit DI: Dependency Injection Without Magic

의존성 주입은 객체가 필요한 협력자를 스스로 생성하는 대신 외부에서 받는 패턴입니다.

많은 TypeScript 프레임워크에서는 생성자 타입을 읽어 DI가 자동처럼 느껴집니다. fluo는 그 과정을 의도적으로 더 눈에 보이게 만듭니다.

```typescript
import { Injectable, Inject } from '@fluojs/di';
import { DatabaseService } from './db.service';

@Injectable()
export class UsersService {
  constructor(
    @Inject(DatabaseService) private readonly db: DatabaseService,
  ) {}
}
```

### 표준 데코레이터 시그니처 이해하기

레거시 데코레이터와 달리, 표준 TC39 데코레이터는 데코레이트되는 멤버에 대한 정보를 제공하는 컨텍스트 객체를 받습니다. fluo는 이러한 복잡성을 대부분 숨겨주지만, 이것이 "마법 없는" 동작을 가능하게 한다는 점을 아는 것은 유용합니다.

```typescript
// 표준 데코레이터 호출의 개념적 예시
function MyDecorator(value, context) {
  console.log(`${context.kind} 타입의 ${context.name}을(를) 데코레이트하는 중`);
  // ... 프레임워크 로직
}
```

이 코드는 즉시 두 가지 사실을 전달합니다.

1. `UsersService`는 프레임워크가 관리하는 의존성입니다.
2. 이 서비스가 동작하려면 `DatabaseService`가 필요합니다.

추측 단계가 없습니다.

### Why Constructor Injection?

생성자 주입은 필수 의존성을 무시할 수 없게 만들어 주기 때문에 초보자에게 가장 친절한 DI 형태입니다.

생성자 주입으로 만들어지는 객체는 유효하거나, 아니면 아예 생성되지 않습니다.

이 방식은 실질적인 장점을 줍니다.

- 필요한 협력자가 클래스의 맨 위에 드러납니다.
- 테스트에서 대체 구현을 직접 넣을 수 있습니다.
- 숨겨진 필드 초기화 코드를 뒤질 필요가 없습니다.
- 객체 생명주기가 예측 가능합니다.

### Explicit Tokens and Concrete Classes

`@Inject()`는 구체 클래스를 참조할 수 있지만, 이 아이디어는 클래스보다 더 넓게 확장됩니다.

실제 프로젝트에서는 다음도 주입할 수 있습니다.

- 설정 토큰,
- 인터페이스 뒤에 있는 리포지토리,
- 외부 클라이언트를 만드는 팩토리,
- 플랫폼별 어댑터.

초보자 단계에서 기억할 점은 단순합니다. 의존성은 코드에 이름으로 드러나야 합니다.

### A Small Testing Payoff

블로그 글을 생성하는 서비스가 리포지토리에 의존한다고 상상해 봅시다.

명시적 DI를 사용하면 테스트 방식이 분명해집니다.

1. 가짜 리포지토리를 만듭니다.
2. 실제 리포지토리가 들어갈 자리에 그 가짜를 넣습니다.
3. 예측 가능한 데이터로 서비스를 검증합니다.

서비스가 스스로 리포지토리를 만들지 않기 때문에 테스트는 프레임워크 내부보다 동작 자체에 집중할 수 있습니다.

### What fluo Is Trying to Teach

fluo는 단지 기술적인 문제만 해결하려는 것이 아닙니다. 하나의 아키텍처 습관도 함께 가르치고 있습니다.

그 습관은 **중요한 연결을 읽을 수 있게 만들라**는 것입니다.

이 습관을 받아들이면 이후 주제가 더 쉬워집니다.

- 모듈 분리가 쉬워집니다.
- 공유 프로바이더 내보내기가 쉬워집니다.
- 런타임별 코드를 분리하기 쉬워집니다.
- 운영 장애를 진단하기 쉬워집니다.

### Inversion of Control (IoC)

의존성 주입은 제어의 역전(Inversion of Control)이라는 더 넓은 원칙을 구현하는 구체적인 방법입니다.

전통적인 프로그램에서는 상위 컴포넌트가 하위 의존성을 직접 생성하고 제어합니다. 하지만 IoC에서는 그 제어가 "역전"됩니다. 컴포넌트는 자신이 무엇이 필요한지만 정의하고, 외부의 "컨테이너"(여기서는 fluo)가 그 의존성을 제공합니다.

이러한 제어의 이동이 코드를 모듈식으로 만듭니다. `UsersService`는 `DatabaseService`가 어떻게 생성되거나 설정되는지 더 이상 신경 쓰지 않습니다. 실행될 때 그저 존재하기만 하면 됩니다. 이러한 결합 해제(decoupling)가 시스템이 망가지지 않고 성장할 수 있게 만드는 비결입니다.

### The Lifecycle of a Dependency

의존성이 언제 생성되고 파괴되는지 이해하는 것은 메모리나 데이터베이스 연결 같은 자원을 관리하는 데 매우 중요합니다.

fluo에서는 DI 컨테이너가 이 생명주기를 대신 관리합니다. 애플리케이션이 시작될 때 컨테이너는 의존성 그래프를 분석하고 올바른 순서로 인스턴스를 생성합니다.

대부분의 경우 의존성은 싱글톤(singleton)으로 관리되어 한 번 생성된 후 공유됩니다. 하지만 fluo의 명시적인 특성 덕분에 필요한 경우 다양한 생명주기를 정의할 수 있으며, 이는 복잡도가 높아져도 애플리케이션이 효율적으로 동작하게 해 줍니다.

### Debugging with Explicit DI

명시적 DI의 가장 큰 장점 중 하나는 디버깅이 훨씬 쉬워진다는 것입니다.

의존성이 누락되어 서비스가 동작하지 않을 때, fluo는 어떤 토큰이 요청되었고 어떤 모듈이 이를 제공할 책임이 있는지 명확한 에러 메시지를 보여줍니다.

모든 연결이 `@Module()`과 `@Inject()` 호출에 드러나 있기 때문에 프레임워크의 자동 발견 기능이 어디서 잘못되었는지 추측할 필요가 없습니다. 코드를 보고, 임포트를 따라가고, 빠진 조각을 찾으면 됩니다. 이러한 직접적인 방식은 수많은 삽질 시간을 줄여주고 여러분을 더 유능한 개발자로 만들어 줍니다.

## 1.3 Runtime Neutrality: One Codebase, Any Platform

현대적인 백엔드 애플리케이션은 하나의 환경에만 영원히 머무르지 않습니다.

팀은 로컬에서 Node.js로 시작하고, Bun으로 성능을 비교하고, 컨테이너에 배포하고, 일부 워크로드를 엣지 플랫폼으로 옮기기도 합니다. 애플리케이션 로직이 특정 런타임에 강하게 묶인 프레임워크는 이런 미래 이동을 모두 더 비싸게 만듭니다.

fluo는 프레임워크 계약과 런타임 어댑터를 분리함으로써 이 문제를 다룹니다.

### Runtime Boundary vs Business Logic

비즈니스 로직은 다음과 같은 질문에 답합니다.

- 이 사용자가 이 글을 수정할 수 있는가?
- 임시 저장 글은 어떤 규칙으로 검증해야 하는가?
- API는 어떤 응답 형태를 반환해야 하는가?

런타임 코드는 다른 질문에 답합니다.

- HTTP 요청은 어떤 방식으로 도착하는가?
- 서버는 어떻게 시작되는가?
- 어떤 스트리밍 또는 fetch 원시 기능을 사용할 수 있는가?

fluo는 이 관심사를 분리해 도메인 코드가 호스팅 환경 변화에 덜 흔들리게 합니다.

### What Changes per Platform?

서로 다른 런타임은 서로 다른 운영 원시 기능을 제공합니다.

- **Node.js**는 Fastify와 성숙한 툴링을 자주 함께 사용합니다.
- **Bun**은 빠른 시작 속도와 내장 API를 강조합니다.
- **Deno**는 자체 보안 모델과 표준 라이브러리 관례를 가집니다.
- **Cloudflare Workers**는 엣지 중심 제약을 가진 isolate 환경에서 실행됩니다.

이 차이는 분명하지만, 주로 어댑터와 부트스트랩 코드에 머물러야 하며 게시물 서비스나 사용자 서비스까지 흔들어서는 안 됩니다.

### What Stays the Same?

런타임이 달라도 동일하게 유지되어야 하는 구조가 있습니다.

- 컨트롤러는 라우트를 설명해야 합니다.
- 프로바이더는 재사용 가능한 로직을 설명해야 합니다.
- 검증 규칙은 동일하게 동작해야 합니다.
- 직렬화 규칙은 같은 API 계약을 만들어야 합니다.

런타임 중립성은 바로 이 일관성을 지키기 위한 개념입니다. 환경 변화로부터 애플리케이션 모델을 보호합니다.

### A Beginner-Friendly Example

FluoBlog가 `GET /posts`를 제공한다고 가정해 봅시다.

글 목록을 불러와 JSON으로 반환하는 코드는 하나의 런타임 어댑터에서 다른 어댑터로 바뀐다고 해서 달라지면 안 됩니다.

런타임이 바뀔 때마다 동작이 바뀐다면 프레임워크 경계가 너무 많은 일을 하고 있는 것입니다.

### Why This Helps Teams

런타임 중립성은 단지 이식성을 위한 개념만은 아닙니다. 책임 분리에도 도움이 됩니다.

- 플랫폼 담당자는 어댑터를 개선할 수 있습니다.
- 기능 팀은 비즈니스 규칙에 집중할 수 있습니다.
- 테스트는 애플리케이션 계약에 더 가까워집니다.
- 마이그레이션 작업이 더 기계적이고 덜 위험해집니다.

## 1.4 TC39 Standard Decorators: Moving Beyond experimentalDecorators

데코레이터는 fluo 개발 경험의 핵심이지만, 이 프레임워크는 오래된 TypeScript 전용 동작이 아니라 현대 JavaScript 표준 위에 기반을 둡니다.

오랫동안 개발자들은 `experimentalDecorators` 옵션을 통해 레거시 데코레이터를 사용해 왔습니다. 편리하긴 했지만, 그것이 JavaScript의 최종 방향은 아니었습니다.

fluo는 그 대신 표준화된 버전에 맞춥니다.

### Legacy Decorators vs Standard Decorators

초보자가 기억해야 할 핵심은 모든 저수준 시그니처가 아니라 철학의 차이입니다.

- 레거시 데코레이터는 TypeScript 시대의 실험이었습니다.
- 표준 데코레이터는 JavaScript 언어 자체의 방향을 반영합니다.
- 표준 동작은 비공식 컴파일러 요령에 대한 의존을 줄입니다.
- 표준화된 의미론은 프레임워크 동작을 장기적으로 더 이해하기 쉽게 만듭니다.

fluo가 “standard-first”를 말할 때 가장 잘 드러나는 지점 중 하나가 바로 이것입니다.

### A New Era for Metadata

표준 데코레이터는 구조화된 컨텍스트 객체와 메타데이터 지향 패턴을 위한 언어 레벨 지원을 제공합니다.

이는 프레임워크 작성자가 모든 것을 해결하기 위해 더 이상 구식 `reflect-metadata` 스타일을 기본 답으로 삼지 않아도 된다는 뜻입니다.

학습자 입장에서는 장점이 분명합니다.

- `tsconfig.json`을 현재 JavaScript 현실에 더 가깝게 유지할 수 있습니다.
- 프레임워크 어휘가 언어의 미래 방향과 일치합니다.
- 애플리케이션이 오래된 호환성 가정을 덜 끌고 갑니다.

### Why fluo Cares So Much

fluo가 이 지점에서 강한 의견을 갖는 이유는 프레임워크의 토대가 그 위 모든 계층에 영향을 주기 때문입니다.

데코레이터 모델이 불안정하거나 지나치게 마법적이면 DI, 라우팅, 검증, 툴링 모두 그 불안정성을 상속받습니다.

fluo는 표준 모델 위에 일찍 올라탐으로써 단기 편의보다 장기 일관성을 선택합니다.

### A Practical Checklist

현대적인 TypeScript에 맞는 프레임워크인지 평가할 때 다음을 물어보세요.

1. 레거시 컴파일러 플래그가 필요한가?
2. 정상 동작을 위해 출력 메타데이터에 의존하는가?
3. 데코레이터가 의존성 등록에 어떻게 참여하는지 설명하는가?
4. JavaScript 언어의 방향과 가까운가?

fluo는 이 질문에 대한 답을 코드와 문서에서 쉽게 확인할 수 있기를 원합니다.

## 1.5 The fluo Ecosystem Map

fluo는 설계상 모듈형입니다. 하나의 거대한 모놀리식 패키지로 모든 것을 묶기보다, 필요한 스택을 조합할 수 있는 집중된 패키지들을 제공합니다.

| Category | Primary Packages |
| :--- | :--- |
| **Core** | `@fluojs/core`, `@fluojs/di`, `@fluojs/runtime` |
| **HTTP** | `@fluojs/http`, `@fluojs/platform-fastify`, `@fluojs/openapi` |
| **Data** | `@fluojs/prisma`, `@fluojs/drizzle`, `@fluojs/redis` |
| **Logic** | `@fluojs/validation`, `@fluojs/serialization`, `@fluojs/cqrs` |
| **Ops** | `@fluojs/metrics`, `@fluojs/terminus`, `@fluojs/queue` |

### Package Categories at a Glance

각 카테고리는 백엔드 작업의 서로 다른 층을 해결합니다.

- **Core** 패키지는 애플리케이션 모델과 DI 시스템을 정의합니다.
- **HTTP** 패키지는 컨트롤러와 라우팅을 실제 전송 계층에 연결합니다.
- **Data** 패키지는 영속성 또는 캐시 계층과의 통신을 돕습니다.
- **Logic** 패키지는 검증, 직렬화, 아키텍처 패턴을 담당합니다.
- **Ops** 패키지는 신뢰성과 런타임 가시성을 지원합니다.

### Picking Only What You Need

이 모듈형 구조는 초보자에게 인지 부담을 줄여 준다는 점에서 중요합니다.

처음부터 모든 패키지를 마스터할 필요는 없습니다.

Part 0에서 중요한 것은 주로 다음입니다.

- 코어 프레임워크 모델,
- CLI,
- HTTP 표면,
- 데코레이터와 DI의 정신 모델.

그 외 패키지는 프로젝트가 실제로 필요로 할 때 등장하면 됩니다.

### How This Helps Learning

모듈성은 이 책이 깔끔한 학습 순서를 갖게도 해줍니다.

1. 애플리케이션 구조를 배웁니다.
2. 실제 프로젝트를 스캐폴딩합니다.
3. 모듈과 프로바이더를 이해합니다.
4. 데코레이터가 왜 중요한지 이해합니다.
5. 사용 사례가 생길 때 고급 패키지를 추가합니다.

이 흐름은 건강한 실무 시스템이 자라는 방식과도 닮아 있습니다. 한 번에 하나의 분명한 필요를 해결하는 것입니다.

## 1.6 Meet FluoBlog

이 책 전체에서 우리는 **FluoBlog**라는 블로그 API를 함께 만들어 갑니다. 이 프로젝트는 장마다 조금씩 성장합니다.

이 예제를 선택한 이유는 익숙하기 때문입니다.

블로그에는 글, 카테고리, 작성자, 인증, 검증 규칙, 캐싱이나 관측성과 같은 운영 이슈가 모두 들어갑니다. 낯선 비즈니스 도메인을 먼저 학습하지 않아도 충분히 현실적인 구조를 보여 줄 수 있습니다.

### Why a Blog?

블로그 애플리케이션은 등장하는 명사를 쉽게 떠올릴 수 있기 때문에 초보자에게 친숙합니다.

- posts,
- comments,
- users,
- drafts,
- categories,
- permissions.

동시에 기능이 쌓일수록 아키텍처는 충분히 현실적으로 발전합니다.

### What We Build First

초반 장들에서 FluoBlog는 기초에 집중합니다.

1. CLI로 프로젝트를 생성합니다.
2. 생성된 구조를 이해합니다.
3. 첫 번째 도메인 모듈을 추가합니다.
4. 프레임워크가 사용하는 데코레이터 모델을 이해합니다.

이 단계를 통해 검증, 영속성, 인증, 운영 기능을 넣기 전에 안정적인 기반을 먼저 만듭니다.

### What We Delay Until Later

좋은 입문서는 너무 이른 시점에 다루지 않아야 할 것도 분명히 선택합니다.

다음과 같은 심화 주제는 뒤로 미룹니다.

- 데이터베이스 트랜잭션,
- 고급 권한 부여 흐름,
- 캐싱 전략,
- 메트릭과 헬스 체크,
- 프로덕션 롤아웃 문제.

이 지연은 의도적입니다. 초보자는 아키텍처가 의미 있는 층으로 확장될 때 더 빠르게 배웁니다.

### The Project-State Comment

각 장의 상단에는 `project-state` 주석이 있습니다.

이 주석은 작은 내비게이션 표지판 역할을 합니다.

- 장이 어떤 FluoBlog 버전을 전제로 하는지 알려 줍니다.
- 이 책이 누적식 구성이라는 점을 상기시킵니다.
- 예제가 장의 시간선과 맞도록 유지보수하는 데 도움을 줍니다.

### Building for the Long Term

프레임워크를 선택하는 것은 단순한 도구를 선택하는 것이 아니라, 미래 작업의 토대를 선택하는 것입니다.

표준과 명시성에 대한 fluo의 약속은 여러분의 프로젝트가 장기적으로 건강하게 유지되도록 하겠다는 약속이기도 합니다. JavaScript 생태계가 계속 진화하더라도 fluo는 안정적이고 예측 가능한 파트너로 남을 것입니다.

지금 이 기초적인 개념들을 익히는 것은 단순히 이 프로젝트뿐만 아니라 여러분이 앞으로 맡게 될 모든 프로젝트에서의 성공을 위한 준비 과정입니다. 깨끗한 아키텍처, 모듈식 설계, 그리고 명시적 설정이라는 훈련은 여러분의 경력 전체에 걸쳐 큰 힘이 될 것입니다.

### A Mindset for Success

fluo와 함께하는 여정을 시작하면서, 모든 전문가도 처음에는 초보자였다는 사실을 기억하세요.

어떤 개념들은 처음에는 다소 추상적으로 느껴질 수도 있지만 걱정하지 마세요. FluoBlog 프로젝트는 이러한 아이디어들을 실제로 구현하며 보여주도록 설계되었습니다. 실제 애플리케이션에서 이들이 어떻게 작동하는지 곧 눈으로 확인하게 될 것입니다.

이 책을 읽으며 가장 중요한 것은 호기심 어린 마음가짐입니다. 왜 이렇게 설계되었는지 질문을 던져 보세요. 코드를 직접 실험해 보세요. 무언가를 망가뜨려 보기도 하고 다시 고쳐 보기도 하세요. 그런 과정을 통해 진정한 이해가 만들어집니다. 이제 함께 놀라운 것을 만들러 가볼까요?

## Summary
- fluo는 메타데이터 비대화, 숨겨진 연결, 플랫폼 종속성 문제를 줄이려 합니다.
- 명시적 DI는 의존성 그래프를 읽고 테스트하기 쉽게 만듭니다.
- 런타임 중립성은 비즈니스 로직과 환경별 어댑터를 분리합니다.
- TC39 표준 데코레이터는 fluo의 standard-first 철학의 핵심입니다.
- 패키지 생태계는 모듈형이므로 초보자는 한 층씩 배울 수 있습니다.
- FluoBlog는 모든 장을 이어 주는 중심 예제입니다.
