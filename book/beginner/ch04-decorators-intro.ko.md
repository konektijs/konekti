## 4.1 What is a Decorator?

시작은 가장 단순한 질문에서 출발하는 편이 좋습니다.

처음에는 데코레이터를 클래스나 클래스 멤버에 동작 또는 메타데이터 관련 의도를 부여하는 함수 기반 메커니즘으로 이해하면 됩니다.

여기에서 가장 중요한 단어는 **의도(intent)**입니다.

데코레이터를 사용하면 다음과 같은 선언을 할 수 있습니다.

- 이 클래스는 컨트롤러다.
- 이 클래스는 주입 가능하다.
- 이 메서드는 GET 라우트를 처리한다.
- 이 속성은 특정 의존성을 받아야 한다.

이 선언적 스타일은 fluo 같은 프레임워크가 표현력이 좋아 보이는 이유 중 하나입니다.

### Meta-Programming Made Easy

데코레이터는 메타 프로그래밍의 한 형태입니다. 메서드 본문의 순수 비즈니스 로직을 직접 바꾸지 않으면서도 코드가 해석되거나 등록되는 방식에 영향을 주기 때문입니다.

예를 들어 `@Controller('/posts')`는 JavaScript 클래스의 일반 규칙을 바꾸지 않습니다. 대신 프레임워크가 HTTP 계층을 구성할 때 그 클래스를 어떻게 취급해야 하는지 알려 줍니다.

### Why Beginners Should Care

데코레이터가 단지 “멋져 보이는 문법”에 불과했다면 이렇게까지 중요하지 않았을 것입니다.

중요한 이유는 데코레이터가 프레임워크의 어휘가 되기 때문입니다.

- 모듈은 데코레이터로 정의되고,
- 컨트롤러는 데코레이터로 등록되며,
- 라우트는 데코레이터로 매핑되고,
- 의존성 연결도 명시적인 위치에서 데코레이터를 사용합니다.

데코레이터를 이해하면 fluo의 나머지 부분도 훨씬 더 직접적으로 읽힙니다.

### Declarative Code vs Imperative Registration

데코레이터가 없어도 프레임워크 API를 만들 수는 있습니다. 하지만 그 경우 보통 더 많은 수동 등록 코드가 필요해집니다.

데코레이터는 그 등록 의도를 더 짧고 훑어보기 쉬운 문법으로 압축합니다. 그래서 백엔드 프레임워크에서 특히 유용합니다.

### Code Organization Benefits

데코레이터는 시각적인 표식 역할도 합니다. 500줄짜리 파일을 훑어볼 때 메서드 위에 있는 `@Get()`을 발견하면, 그 메서드의 외부적 목적을 바로 알 수 있습니다. 이런 조직화는 새로운 코드베이스를 읽을 때의 "인지 부하(cognitive load)"를 줄이고, 팀이 대규모 프로젝트에서 같은 기준으로 코드를 보게 합니다.

### Separation of Concerns

데코레이터는 라우팅, 유효성 검사 또는 권한 부여와 같은 횡단 관심사(cross-cutting concerns)를 애플리케이션의 핵심 로직에서 분리합니다. 모든 메서드 안에 상용구 코드를 넣는 대신 필요한 의도를 선언하므로, 시간이 지나도 비즈니스 로직과 프레임워크 동작을 구분하기 쉽습니다.

## 4.2 Legacy vs. Standard Decorators: Why the Shift?

데코레이터의 기본 역할이 보였다면, 이제 fluo가 왜 표준 버전을 그렇게 중요하게 여기는지로 자연스럽게 넘어갈 수 있습니다.

많은 TypeScript 개발자는 먼저 레거시 `experimentalDecorators` 모델을 통해 데코레이터를 접했습니다.

그 오래된 방식은 널리 쓰였지만 JavaScript의 최종 목적지는 아니었습니다. 언어는 결국 TC39 Stage 3 표준 데코레이터 설계 쪽으로 이동했습니다.

fluo에게 이 변화는 사소한 구현 세부사항이 아닙니다. 프레임워크 철학의 일부입니다.

### The Legacy Shape

레거시 데코레이터는 종종 TypeScript의 역사적 구현 선택에 강하게 묶여 있는 시그니처를 사용했습니다.

또한 의존성 주입을 편하게 만들기 위해 `reflect-metadata`, `emitDecoratorMetadata`에 자주 의존했습니다.

그 조합은 익숙한 생태계를 만들었지만 동시에 기술 부채도 만들었습니다.

### The Standard Shape

표준 데코레이터는 명확한 컨텍스트 객체와 잘 정의된 실행 의미론을 강조합니다.

이 장에서 모든 API 세부사항을 외울 필요는 없습니다. 지금 가져가야 할 핵심은 더 단순합니다.

- 표준 데코레이터는 JavaScript의 미래 방향에 더 가깝고,
- 레거시 데코레이터는 과도기적 시대의 산물이었으며,
- fluo는 의도적으로 표준 경로를 선택합니다.

### Why the Shift Matters in Practice

이 변화는 문법 이상의 영향을 줍니다.

- 컴파일러 설정이 더 깔끔해지고,
- 프레임워크 동작을 표준 용어로 설명하기 쉬워지며,
- 애플리케이션이 오래된 호환성 가정에 덜 묶이게 됩니다.

### Performance Gains

표준 데코레이터의 자주 간과되는 이점 중 하나는 성능입니다. 현대 JavaScript 엔진에서 더 자연스럽게 처리되고 무거운 런타임 리플렉션 라이브러리가 필요하지 않기 때문에, 애플리케이션의 부팅 속도와 메모리 사용량에 유리합니다. fluo는 이 효율성을 바탕으로 작고 빠른 구조를 유지합니다.

### Improved Debugging

표준 데코레이터는 더 예측 가능한 실행 흐름을 제공합니다. 문제가 발생했을 때, 프레임워크가 숨겨진 리플렉션 계층을 거치지 않기 때문에 스택 트레이스가 더 명확하게 나타나는 경우가 많습니다. 그만큼 데코레이터가 왜 실행되지 않는지 추적하는 시간이 줄고, 실제 기능 구현에 더 집중할 수 있습니다.

### A Good Beginner Question

현대적인 프레임워크를 평가할 때는 이렇게 물어보세요.

“이 API는 언어의 방향 위에 세워졌는가, 아니면 생태계가 익숙해진 레거시 우회책 위에 세워졌는가?”

fluo는 첫 번째 답을 원합니다.

## 4.3 Class Decorators

그다음에는 실제 fluo 코드에서 만나는 순서대로 데코레이터 유형을 살펴보는 편이 이해하기 쉽습니다.

클래스 데코레이터는 클래스 전체에 적용됩니다. fluo에서는 가장 눈에 잘 들어오는 프레임워크 개념 중 상당수가 여기에서 시작됩니다.

- `@Module()`
- `@Global()`
- `@Inject(...)`
- `@Scope(...)`
- `@Controller()`

이 데코레이터들은 각 클래스가 애플리케이션 안에서 어떤 역할을 맡는지 프레임워크에 알려 줍니다.

### What a Class Decorator Communicates

클래스 데코레이터는 고수준 정체성 질문에 답합니다.

- 이 클래스는 애플리케이션 구성의 일부인가?
- 전역으로 노출되는 모듈인가?
- 생성자 의존성 토큰을 명시하는가?
- HTTP 진입점인가?

이 역할 정보는 이후 프레임워크 시스템이 그 위에 쌓이기 때문에 매우 기초적입니다.

프로바이더 자체는 별도 `@Injectable()` 표식이 아니라 모듈의 `providers` 메타데이터로 등록된다는 점도 함께 기억해 두세요.

### Why fluo Uses Them Heavily

fluo가 클래스 데코레이터를 많이 사용하는 이유는 의도를 짧고 읽기 좋게 선언할 수 있기 때문입니다.

```typescript
import { Inject, Module } from '@fluojs/core';
import { Controller } from '@fluojs/http';

export class MyService {}

@Module({ providers: [MyService] })
export class MyModule {}

@Inject(MyService)
@Controller('/api')
export class MyController {
  constructor(private readonly service: MyService) {}
}
```

클래스 첫 줄만 읽어도 그 파일이 무엇인지 알 수 있다면 파일 탐색이 훨씬 쉬워집니다.

이 가독성은 파일 수가 많은 모듈형 코드베이스에서 특히 유용합니다.

### Metadata Ownership

레거시 모델에서 메타데이터는 종종 프로토타입 전체에 걸쳐 섞였습니다. 표준 모델에서는 프레임워크가 클래스에 부착하는 메타데이터에 대해 더 명확한 소유권을 가집니다. 이는 서로 다른 라이브러리가 실수로 서로의 설정을 덮어쓸 위험을 줄이고, 더 안정적인 애플리케이션을 가능하게 합니다.

### Class Identity vs. Instance State

클래스 데코레이터는 개별 인스턴스뿐만 아니라 *클래스 자체*에 영향을 미친다는 점을 기억하세요. 이 구분은 fluo의 의존성 주입 컨테이너가 싱글톤과 팩토리 패턴을 관리하는 방식을 이해하는 데 중요합니다. 클래스를 데코레이트하면 프레임워크가 해당 클래스의 인스턴스를 어떻게 만들고 더 큰 애플리케이션에 어떻게 연결해야 하는지 정의하게 됩니다.

### addInitializer and Framework Setup

표준 데코레이터 모델에서는 클래스 데코레이터가 제멋대로 프로토타입을 조작하는 대신, 잘 정의된 훅을 통해 설정 로직을 연결할 수 있습니다.

여기서 얻어야 할 교훈은 정확한 API 이름이 아니라 아키텍처 개선입니다.

표준 모델은 프레임워크 작성자에게 더 깨끗한 빌딩 블록을 주고, 깨끗한 빌딩 블록은 대개 더 깨끗한 프레임워크 동작으로 이어집니다.

### What to Notice in Real Code

fluo 소스나 예제를 읽을 때는 그 클래스 데코레이터가 해당 파일의 역할에 대해 무엇을 선언하는지 잠깐 생각해 보세요. 이 습관은 낯선 코드를 빠르게 읽는 데 큰 도움이 됩니다.

## 4.4 Method Decorators

클래스 수준의 정체성을 봤다면, 다음은 메서드 수준의 동작입니다.

메서드 데코레이터는 클래스 전체가 아니라 개별 메서드를 대상으로 합니다.

HTTP 코드에서는 라우트 동작을 선언하는 데 자주 사용됩니다.

```typescript
@Get('/')
findAll() {
  return this.postsService.findAll();
}
```

### Why Method Decorators Matter

메서드 데코레이터는 두 가지 아이디어를 연결합니다.

1. 클래스는 컨트롤러를 나타낼 수 있다.
2. 특정 메서드는 라우트 핸들러를 나타낼 수 있다.

메서드 데코레이터가 없다면 라우트 등록을 위해 별도의 매핑 코드가 더 많이 필요해지는 경우가 많습니다.

### Route Intent Becomes Local

메서드 데코레이터의 큰 장점 중 하나는 지역성(locality)입니다.

핸들러의 목적을 알기 위해 멀리 떨어진 설정 파일을 찾을 필요가 없습니다. 라우트 선언이 메서드 본문 바로 옆에 있기 때문입니다.

이 작은 설계 선택이 생각보다 큰 가독성 차이를 만듭니다.

### Method Decorators and Safety

표준 모델에서 데코레이터 컨텍스트는 어떤 종류의 멤버가 데코레이트되는지, 그것을 외부에 노출하는 것이 타당한지에 대해 프레임워크가 더 분명하게 판단하도록 도와줍니다.

즉, 모호한 관례에만 기대지 않고 더 명확한 규칙을 만들 수 있습니다.

### Composition of Behavior

표준 데코레이터는 조합 가능하도록 설계되었습니다. 하나의 메서드에 여러 데코레이터를 쌓을 수 있습니다. 예를 들어 하나는 라우트를 정의하고, 다른 하나는 접근 권한을 정의할 수 있습니다. 표준이 실행 순서를 명확하게 정의하므로, 이런 동작들이 어떤 순서로 상호작용하는지 추적하기 쉽습니다.

### Decorator Ordering Rules

`@Get('/')`나 `@Roles('admin')`과 같은 데코레이터를 여러 개 쌓을 때, 이들은 아래에서 위로 적용되지만 예측 가능한 순서에 따라 초기화됩니다. 이 레이어링을 이해하면 fluo에서 데코레이터를 더 정확하게 읽고 설계할 수 있습니다.

### A Good Review Habit

컨트롤러 메서드를 리뷰할 때는 본문과 데코레이터 줄을 함께 보세요.

메서드 본문은 코드가 무엇을 하는지 보여 주고, 데코레이터 줄은 그 코드가 애플리케이션 표면에서 어디에 참여하는지 알려 줍니다.

## 4.5 Accessor and Field Decorators

표준 모델은 클래스와 메서드에서 멈추지 않으므로, 장을 닫기 전에 더 넓은 그림도 한 번 보고 가는 편이 좋습니다.

표준 데코레이터는 접근자 중심 패턴도 지원하며, 이는 과거의 필드 중심 우회 방식보다 더 구조화된 모델을 제공합니다.

이 점이 fluo와 관련 있는 이유는 유용한 프레임워크 동작이 항상 클래스 또는 메서드 수준에서만 일어나지는 않기 때문입니다.

```typescript
class MyController {
  @Inject(MyService)
  accessor service: MyService;
}
```

### Why This Is Interesting

접근자는 속성 관련 동작에 대해 언어 차원의 더 명확한 훅을 제공합니다.

그래서 다음과 같은 패턴에 유용합니다.

- 속성 기반 주입,
- 지연 초기화,
- 구조화된 메타데이터 부착.

### Why Beginners Should Stay Calm

첫 fluo 프로젝트에서 접근자 데코레이터를 많이 사용하지 않아도 됩니다.

중요한 것은 “이런 도구도 존재한다”는 인식입니다.

표준 데코레이터 모델은 클래스와 메서드만이 아니라 더 넓고 더 깨끗한 설계 공간을 프레임워크 작성자에게 제공합니다.

### Encapsulation and Getters/Setters

`accessor` 키워드를 데코레이터와 결합하면 fluo가 표준 JavaScript 캡슐화를 존중하면서 속성 접근을 가로챌 수 있습니다. 코드는 관용적(idiomatic)인 상태를 유지하고, 프레임워크는 의존성 해결이나 상태 관리 같은 작업을 정해진 지점에서 처리합니다.

### Reactivity and State Management

기본적인 FluoBlog 예제에서는 덜 흔하지만, 접근자 데코레이터는 fluo의 미래 반응성(reactivity) 기능의 토대입니다. 이를 통해 프레임워크는 속성이 언제 변경되는지 감지하고 필요한 부수 효과를 트리거할 수 있습니다. 현대적인 프런트엔드 프레임워크가 상태 변경을 다루는 방식과 비슷한 맥락입니다.

### Performance and Predictability

오래된 필드 데코레이터 패턴은 언어 한계를 우회해야 했기 때문에 어색하게 느껴지는 경우가 많았습니다. 표준화된 접근자 동작은 프레임워크에 더 예측 가능한 기반을 주며, 이는 성능과 유지보수성 모두에 도움이 되는 경향이 있습니다.

### A Practical Takeaway

지금 단계에서는 다음 규칙만 기억하면 충분합니다.

클래스 데코레이터는 클래스가 무엇인지 정의하고, 메서드 데코레이터는 메서드가 앱 표면에서 무엇을 하는지 정의하며, 접근자 데코레이터는 속성이 프레임워크 동작에 어떻게 참여하는지 정의하는 데 도움을 줍니다.

## 4.6 Verification: tsconfig.json Settings

개념을 이해했다면, 그 개념이 실제 프로젝트 설정에서 어떻게 지켜지는지도 연결해 보는 것이 좋습니다.

코드 스타일과 프레임워크 아키텍처는 TypeScript 설정이 의도한 데코레이터 모델과 맞아야만 깔끔하게 동작합니다.

초기 프로젝트에 적합한 기준선은 다음과 같습니다.

```json
{
  "compilerOptions": {
    "experimentalDecorators": false,
    "emitDecoratorMetadata": false,
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler"
  }
}
```

### Why `experimentalDecorators: false` matters

이 설정은 프로젝트가 오래된 레거시 데코레이터 경로를 선택하지 않았다는 가장 분명한 신호입니다.

fluo에게 이 정렬은 의도적이며 토대가 되는 선택입니다.

### Why `emitDecoratorMetadata: false` matters

이 옵션이 중요한 이유는 fluo가 정상적인 프레임워크 동작을 컴파일러가 내보낸 타입 메타데이터에 의존하게 만들고 싶지 않기 때문입니다.

그 덕분에 아키텍처가 명시적 등록과 표준 동작에 더 가까워집니다.

### Why modern targets matter

현대적인 `target` 설정은 도구 체인이 새 JavaScript 기능을 어색한 우회 없이 더 자연스럽게 유지하도록 도와줍니다.

즉, 언어 설정이 프레임워크 철학을 거스르지 않고 지지해야 한다는 뜻입니다.

### Bundler and Module Resolution

`"moduleResolution": "bundler"`와 같은 설정은 개발 환경이 fluo가 사용하는 현대적인 ESM(ECMAScript Modules) 패키지를 해석하는 방법을 이해하게 합니다. 이는 "임포트 오류"를 줄이고, 애플리케이션을 만들 때 IDE가 더 정확한 자동 완성과 타입 체크를 제공하게 합니다.

### Strictly Enforced Standards

이러한 플래그들을 `false`로 설정하면 TypeScript 컴파일러에 "임시적인 실험 기능이 아니라 공식 언어 표준을 사용한다"고 알리는 셈입니다. 이는 TypeScript가 결국 레거시 구현을 제거하더라도 데코레이터를 다시 작성할 가능성을 낮추고, 코드베이스를 더 오래 유지 가능한 상태로 둡니다.

### What to Check in a Real Project

생성된 프로젝트나 스타터가 이상하게 동작한다면 다음을 확인하세요.

1. 올바른 `tsconfig.json`이 실제로 사용되는가?
2. 레거시 데코레이터 플래그가 다시 들어오지는 않았는가?
3. 툴링 플러그인이 오래된 동작을 전제하지는 않는가?

프로젝트가 더 커지고 커스터마이징될수록 이런 설정 인식은 더 중요해집니다.

## 4.7 Looking Ahead: Advanced Decorators

이 시점은 지금 꼭 알아야 할 것과 나중에 배워도 될 것을 구분하기에도 좋습니다.

이 장은 입문이지, 주제의 끝이 아닙니다.

실력이 올라갈수록 데코레이터는 더 깊은 주제로 이어집니다.

- 커스텀 데코레이터 작성,
- 메타데이터 중심 프레임워크 기능 조합,
- 성능 트레이드오프 이해,
- 표준 모델 위에 더 높은 수준의 추상화 만들기.

### Custom Decorator Logic (Preview)

fluo는 일반적인 작업에 필요한 데코레이터를 제공하지만, 나중에는 복잡한 로직을 자신만의 커스텀 데코레이터로 감싸는 방법도 배우게 됩니다. 인증 확인, 역할 검증, 감사 로깅(audit logging)을 하나의 재사용 가능한 `@UserOnly()` 데코레이터로 결합할 수 있습니다. 이런 지점에서 표준 모델의 장점이 더 분명해집니다.

### Meta-Framework Patterns

고급 개발자들은 데코레이터를 사용하여 fluo에 새로운 기능을 확장하는 "메타 프레임워크" 라이브러리를 구축합니다. 예를 들어, 커뮤니티 패키지는 클래스 메서드가 매시간 자동으로 실행되도록 예약하는 `@Cron()` 데코레이터를 제공할 수 있습니다. fluo는 표준 위에 구축되었기 때문에 이런 확장을 만들고 적용하기가 더 단순합니다.

### Why Advanced Topics Can Wait

바로 커스텀 데코레이터 설계로 뛰어들고 싶을 수 있지만, 처음에는 보통 최선이 아닙니다.

먼저 프레임워크가 제공하는 기본 데코레이터를 맥락 속에서 읽는 것이 더 빠른 학습으로 이어집니다.

그 정신 모델이 안정되면 커스텀 추상화도 훨씬 더 자연스럽게 이해됩니다.

### What You Need Right Now

Part 0가 끝나는 시점에서 여러분에게 필요한 것은 자신만의 데코레이터 시스템 구현이 아닙니다.

지금 필요한 것은 신뢰할 수 있는 기반입니다.

- 데코레이터는 프레임워크 의도를 표현하고,
- 표준 모델은 미래 지향적이며,
- fluo는 그 선택 위에 아키텍처를 세웁니다.

이 정도면 다음 장들을 안정적으로 따라갈 수 있습니다.

- 데코레이터는 클래스와 클래스 멤버에 대한 의도를 표현합니다.
