# Chapter 17. fluo Contributing Guide

이 장은 fluo 저장소에 실제로 기여할 때 따라야 할 로컬 개발 환경, 검증 루프, 리뷰 문화, 거버넌스 흐름을 설명합니다. Chapter 16에서 확장 패키지를 설계했다면, 이제 그 결과를 저장소 규칙에 맞춰 제안하고 유지보수 가능한 형태로 기여하는 단계로 넘어갑니다.

## Learning Objectives
- fluo 저장소 구조와 기여 과정에서 중요한 작업 경계를 이해합니다.
- 이슈 템플릿, 라벨 체계, 토론 채널이 어떻게 협업 흐름을 만드는지 배웁니다.
- `pnpm verify` 중심의 검증 게이트와 리뷰 기대치를 익힙니다.
- 행동 계약 정책과 표준 우선 원칙이 PR 판단 기준에 어떻게 반영되는지 살펴봅니다.
- 로컬 개발 환경 준비, 워크트리 사용, 패키지 단위 개발 루프를 정리합니다.
- RFC와 릴리스 거버넌스 흐름 속에서 기여를 제안하고 마무리하는 방법을 확인합니다.

## Prerequisites
- Chapter 16 완료.
- Node.js 18 이상, pnpm, git이 설치된 로컬 개발 환경.
- 모노레포 작업 흐름과 패키지 단위 테스트 실행에 대한 기초 이해.
- 검증 실패를 읽고 수정하는 기본적인 테스트 및 타입체크 경험.

## Repository Structure and Philosophy

fluo 저장소는 `pnpm`으로 관리되는 고성능 모노레포입니다. 핵심 철학은 **행동 계약(Behavioral Contracts)**입니다. 모든 변경 사항은 기능 추가 여부만이 아니라, 다양한 런타임(Node.js, Bun, Workers)에서 프레임워크의 예측 가능성에 어떤 영향을 주는지로 평가됩니다.

fluo는 "평면적이지만 계층화된(Flat but Layered)" 전략을 따릅니다. 모든 패키지는 가능한 한 얇게 유지하고, 복잡한 로직은 전문화된 모듈로 위임하되 통일된 인터페이스를 유지해야 합니다. 이 방식은 fluo의 핵심을 작고 빠르게 유지하면서 생태계가 확장될 여지를 남깁니다. 너무 많은 일을 하려는 "만능 패키지(God package)"를 피하고, 단일 책임 원칙(Single Responsibility Principle)을 따르는 작고 조합 가능한 단위를 선호합니다.

### Workspace Organization

- `packages/`: 프레임워크의 모듈식 구성 요소를 포함합니다. 각 패키지(예: `@fluojs/core`, `@fluojs/di`)는 모노레포의 전역 표준을 준수하면서 독립적인 수명 주기를 유지해야 합니다.
- `docs/`: 운영 정책, RFC, 아키텍처 결정 기록(ADR)을 포함한 중앙 집중식 문서입니다.
- `examples/`: 검증을 위한 표준 애플리케이션 설정들입니다. 이는 학습 리소스이자 전체 생태계에 대한 스모크 테스트(smoke test) 역할을 합니다.
- `.github/`: 기여 품질 게이트를 강제하는 워크플로우 정의 및 이슈/PR 템플릿입니다.
- `scripts/`: 워크스페이스 관리, 릴리스 준비 및 벤치마킹을 위한 내부 도구입니다.
- `tools/`: 모노레포 전반의 일관성을 보장하는 재사용 가능한 빌드 도구 및 공유 설정(ESLint, Vitest)입니다.

`packages/` 디렉토리의 모든 패키지는 자체 테스트 스위트와 문서를 가진 독립된 단위로 취급되지만, 코드 스타일, 라이선스 및 행동 계약에 관한 전역 저장소 정책을 준수합니다.

## Issue and Label Workflow

메인테이너의 시간이 영향력 있는 작업에 집중될 수 있도록 fluo는 구조화된 이슈 접수 프로세스를 사용합니다. 이 구조는 "유지보수 피로"를 줄이고 개발 속도를 안정적으로 유지합니다. 목표는 초기 제출부터 최종 해결까지 모든 보고에 명확한 경로를 제공하는 것입니다.

### Issue Templates

fluo 저장소에서는 빈 이슈(blank issue)가 비활성화되어 있습니다. 모든 이슈는 시작부터 필요한 컨텍스트가 제공되도록 다음 템플릿 중 하나를 따라야 합니다. 이 방식은 불필요한 질의응답을 줄이고 메인테이너가 이슈를 더 정확히 분류하게 합니다:
- **Bug Report**: 최소 재현 사례(stackblitz 또는 저장소)와 예상 동작 및 실제 동작에 대한 명확한 설명이 필요합니다. 환경 상세 정보(Node 버전, OS 등)도 포함해야 합니다.
- **Feature Request**: 상세한 "이유(Why)"와 "방법(How)" 제안이 필요하며, 행동 계약 정책(Behavioral Contract Policy)과 어떻게 일치하는지도 포함해야 합니다. 해결하려는 문제와 기존 API로 처리할 수 없는 이유를 설명해야 합니다.
- **Documentation Issue**: 가이드의 누락, 오타 또는 개념적 오류를 수정하기 위한 것입니다. 명확한 문서는 코드만큼이나 중요합니다.
- **DX/Maintainability**: 개발자가 더 빠르고 안정적으로 작업할 수 있도록 돕는 내부 개선 사항(예: CI 최적화, 빌드 도구 업데이트)을 위한 것입니다.

작업 목록을 실행 가능한 상태로 유지하고 커뮤니티에서 검색 가능하게 하기 위해 질문은 이슈 트래커가 아닌 **GitHub Discussions**로 유도되어야 합니다.

### Labeling System

이슈는 사용된 템플릿에 따라 자동으로 라벨이 지정되지만, 메인테이너는 종종 분류를 구체화하고 상태를 알리기 위해 보조 라벨을 추가합니다:
- `bug`: 계약을 위반하는 확인된 회귀(regression) 또는 예기치 않은 동작.
- `enhancement`: 프레임워크의 기능을 확장하는 새로운 기능 또는 개선 사항.
- `type:maintainability`: 내부 정리, 도구 개선 또는 기술 부채 감소.
- `priority:p0` ~ `p2`: 이슈의 심각도. P0 이슈는 일반적으로 다음 릴리스를 차단합니다.
- `status:needs-repro`: 제보자가 최소 재현 사례를 제공할 때까지 이슈 작업이 차단됩니다.
- `good first issue`: 코드베이스에 익숙해지려는 새로운 기여자에게 적합합니다.
- `help wanted`: 핵심 팀이 즉시 처리할 여력이 없는 중요한 작업들입니다.

## Review Culture

fluo에서 Pull Request 리뷰는 엄격한 검증 절차입니다. 단순히 "LGTM"만 남기지 않고, 변경이 시스템의 성능이나 신뢰성을 낮추지 않는지 확인합니다.

### Verification Gate

모든 PR은 다음과 같은 복합 작업을 실행하는 `pnpm verify` 명령을 통과해야 합니다:
- **린팅 및 포맷팅**: Prettier 및 특수 ESLint 규칙을 통해 일관성을 확인합니다. 코드베이스를 읽기 쉽고 유지 관리하기 쉬운 상태로 유지하기 위해 엄격한 규칙 세트를 적용합니다.
- **단위 및 통합 테스트**: 모든 패키지에 대해 전체 Vitest 스위트를 실행합니다. 핵심 패키지에 대해서는 100% 커버리지를 목표로 하며, 예외 상황을 다루는 고품질 테스트를 기대합니다.
- **타입 체크**: 전체 워크스페이스 그래프에서 타입 안전성을 확인하기 위해 `tsc --noEmit`을 실행합니다. 여기에는 암시적인 `any` 사용이나 안전하지 않은 타입 단언(assertion)에 대한 확인도 포함됩니다.
- **빌드 검증**: 모든 패키지 타겟(CJS, ESM, 때로는 UMD)에 대해 `pnpm build`가 오류 없이 완료되는지 확인합니다. 또한 번들 크기 저하 여부도 확인합니다.
- **의존성 감사**: 새로운 의존성을 확인하고 라이선스가 올바른지, 보안 취약점을 도입하지 않는지 확인합니다.

### Behavioral Contract Review

고급 기여자의 리뷰는 변경 사항이 기존 계약을 보존하는지에 집중해야 합니다. `@fluojs/di`의 최적화가 `@fluojs/platform-cloudflare-workers`의 스코핑 규칙을 깨뜨리지 않는지, `@fluojs/core`의 새로운 데코레이터가 TC39 표준을 준수하는지 확인해야 합니다.

fluo는 **편의성보다 사양 준수(spec-compliance)**를 우선합니다. 제안된 기능이 ECMAScript 또는 TypeScript 표준에서 벗어나야 한다면, 순수하게 선택 사항인 외부 패키지로 구현할 수 없는 한 거부될 가능성이 높습니다. 추가되는 모든 코드 라인은 프레임워크의 핵심 원칙에 따라 설명 가능해야 합니다. 영리하지만 모호한 구현보다 명시성과 명확성을 더 중요하게 봅니다.

### Documentation First

PR이 공개 API를 추가한다면 인라인 문서(JSDoc)와 `docs/` 또는 `packages/*/README.md`의 관련 마크다운 업데이트를 **반드시** 포함해야 합니다. 문서화되지 않은 기능은 실질적으로 존재하지 않는 것으로 봅니다. 리뷰어는 추가 문서의 명확성, 정확성, 어조도 확인합니다. 문서는 "어떻게"와 "왜"를 모두 이해해야 하는 사용자의 관점에서 작성되어야 하며, 새 기능을 현실적인 맥락에서 보여주는 예제도 필요합니다.

## Release Process and Governance

fluo는 모듈식 생태계 전반에서 높은 안정성을 유지하기 위해 감독된 릴리스 모델을 따릅니다.

### Package Tiers

패키지는 사용자에게 안정성을 알리고 명확한 기대치를 설정하기 위해 세 가지 계층으로 분류됩니다:
- **Official**: 프로덕션 준비 완료, 엄격한 유의적 버전(semver)을 따르며 100% 테스트 커버리지를 가집니다. fluo 생태계의 기준 패키지입니다.
- **Preview**: 조기 채택자를 위한 준비 완료, 실제 피드백에 따라 변경될 수 있습니다. 중요하지 않은 환경에서 테스트하는 것을 권장합니다.
- **Experimental**: 인큐베이션 단계, 예고 없이 제거되거나 대폭 변경될 수 있습니다. 새로운 설계를 검증하는 공간입니다.

### SEMVER and Migration Notes

0.x 버전에서도 파괴적 변경(breaking changes)은 신중하게 다룹니다. 모든 파괴적 변경은 해당 패키지의 `CHANGELOG.md`에 상세한 마이그레이션 노트를 작성해야 합니다. fluo는 `changesets`를 사용해 버전 관리와 변경 이력 생성을 관리하며, 중요한 커밋과 기여자가 정확히 반영되도록 합니다. 이 투명성은 사용자 신뢰를 유지하는 핵심입니다.

### Release Operations

릴리스 운영은 반복 가능성과 보안을 위해 GitHub Actions로 관리됩니다. fluo는 "감독된 자동(supervised-auto)" 모델을 사용합니다:
1. 메인테이너가 로컬에서 `pnpm verify:release-readiness`를 실행하여 워크스페이스 일관성과 보류 중인 변경 사항을 확인합니다.
2. Changeset이 `main` 브랜치에 병합되면 새 버전을 계산하는 "Version Packages" PR이 트리거됩니다.
3. 버전 관리 PR이 병합되면 GitHub Action이 자동으로 패키지를 npm에 게시하고 GitHub Release를 생성합니다.
이 프로세스는 불완전한 코드가 실수로 게시되는 일을 막고, 게시된 아티팩트가 저장소의 소스 코드와 정확히 일치하도록 합니다.

## Governance and RFC Workflow

작은 수정이나 문서 업데이트는 직접 PR할 수 있지만, 중요한 아키텍처 변경은 커뮤니티 합의와 기술적 타당성을 확인하기 위해 RFC(Request for Comments) 프로세스를 거쳐야 합니다. 이 프로세스는 아키텍처의 변질을 막고 모든 주요 추가 사항이 충분한 근거를 갖도록 설계되었습니다.

### The RFC Path

1. **GitHub Discussions**: 커뮤니티의 관심도를 확인하고 잠재적인 함정을 조기에 찾기 위해 "Ideas" 또는 "RFC" 카테고리에 스레드를 시작합니다. 이곳은 부담 없는 브레인스토밍을 위한 장소입니다.
2. **Formal Proposal**: 복잡한 변경 사항의 경우, 마크다운 제안서(`packages/graphql/field-resolver-rfc.md`의 템플릿 참조)를 작성하고 `docs/proposals` 디렉토리에 PR을 엽니다. 제안서는 동기, 상세 설계, 단점 및 대안을 다루어야 합니다.
3. **Review and Consensus**: 핵심 메인테이너와 커뮤니티가 RFC를 리뷰합니다. 구현을 시작하기 전에 최소 두 명의 핵심 메인테이너 승인이 필요합니다. 기술적 견고함과 fluo의 핵심 철학에 맞는지 확인합니다.
4. **Implementation**: RFC가 병합("Accepted")되면, 작업은 실행 가능한 이슈로 분할되어 기여자에게 할당됩니다. 원본 RFC 작성자가 이 단계의 리더가 되어 연속성과 방향성을 제공하는 경우가 많습니다.
5. **Finalization**: 구현과 문서화가 완료되면 RFC는 "Implemented" 상태로 이동하여 설계 결정의 영구적인 기록으로 남습니다.

### Behavioral Contract Policy

모든 기여자는 `docs/contracts/behavioral-contract-policy.md`를 준수해야 합니다. 이 정책은 fluo의 "헌법"에 가깝습니다. JavaScript 언어 경로에서 벗어나는 비표준 TypeScript 기능 사용을 금지해 fluo가 "표준 우선" 프레임워크로 남게 합니다. 표준에 대한 이 약속이 fluo의 장기 호환성을 지탱합니다.

이 정책은 다음과 같은 몇 가지 핵심 영역을 다룹니다:
- **데코레이터 표준**: TC39 표준 데코레이터(TypeScript 5.0+에서 지원)만 허용됩니다. 레거시 실험적 데코레이터는 엄격히 금지됩니다.
- **리플렉션 및 메타데이터**: `reflect-metadata`의 사용은 지양되며, 명시적 레지스트리 및 표준 메타데이터 제안을 선호합니다.
- **런타임 추상화**: 모든 I/O 및 환경 특정 로직은 Node.js, Bun, Deno 및 Workers 간의 이식성을 보장하기 위해 `Platform` 추상화 레이어 뒤에 배치되어야 합니다.
- **오류 처리**: 확인되지 않은 예외보다는 도메인 수준의 오류에 대해 `Result` 패턴을 사용하는 것을 권장합니다.
- **명시적 제어의 역전(IoC)**: DI 시스템은 가능한 한 의존성 그래프의 정적 분석이 가능한 방식으로 사용되어야 합니다.

이 규칙을 따르면 코드베이스가 TypeScript 생태계의 특정 관성에만 묶이지 않고, JavaScript 개발자도 접근할 수 있는 상태로 유지됩니다.

또한 fluo는 "공개 내보내기 기준(Public Export Baseline)"을 엄격하게 유지합니다. 패키지에서 내보내는 모든 함수, 클래스, 인터페이스는 완전한 TSDoc 주석을 가져야 합니다. 여기에는 `@param`, `@returns`, 최소 하나 이상의 `@example` 블록이 포함됩니다. 이 기준은 문서가 이 가이드 같은 마크다운에만 머물지 않고, 개발자가 프레임워크를 사용할 때 IDE 안에서도 바로 읽히도록 합니다. 이런 세부 사항은 전문적인 프레임워크에 필요하며, 새로운 기여자와 사용자의 학습 곡선을 낮추는 데 도움이 됩니다.

표준 우선 접근 외에도 fluo는 "최소 의존성" 정책을 시행합니다. 패키지에 추가되는 모든 새 의존성은 PR에서 근거를 제시해야 하며, 가능하면 의존성이 없거나 매우 적은 모듈이어야 합니다. 이는 공급망을 안전하게 유지하고 번들 크기를 예측 가능하게 만듭니다. 가능할 때마다 특수 라이브러리보다 네이티브 Node.js 및 웹 API를 선호해 장기적인 유지보수성과 플랫폼 이식성 목표를 강화합니다.

코드 품질을 높이기 위해 "동형적 테스트(Isomorphic Testing)"도 장려합니다. 이는 동일한 비즈니스 로직이 Node.js 환경뿐 아니라 브라우저 환경에서도 같은 방식으로 동작하는지 확인하는 과정입니다. Vitest 같은 도구로 가상 환경 검증을 병행하며, 이를 통해 클라우드플레어 워커(Cloudflare Workers)나 번(Bun) 같은 다양한 런타임에서도 fluo가 일관되게 동작하도록 확인합니다.

PR 설명에는 이러한 정책이 어떻게 적용되었는지 적는 편이 좋습니다. 리뷰어는 기술 구현뿐 아니라 그 이면의 철학적 일관성도 확인하기 때문입니다. fluo의 코드는 이유 없이 추가되지 않아야 하며, 모든 결정은 표준, 성능, 명시성이라는 핵심 가치를 향해야 합니다.

매주 금요일, 핵심 팀과 기여자들은 "Maintenance Friday"를 위해 시간을 할애합니다. 새 기능 개발보다 다음 항목에 집중하는 세션입니다:
- **의존성 업그레이드**: 최신 보안 패치 및 라이브러리 버전으로 워크스페이스를 최신 상태로 유지합니다.
- **리팩토링**: 코드 명확성을 개선하고 오래된 모듈의 복잡성을 줄입니다.
- **테스트 스위트 확장**: 커버리지를 높이고 최근 수정된 버그에 대한 회귀 테스트를 추가합니다.
- **CI/CD 최적화**: 더 빠른 피드백 루프를 위해 빌드 및 검증 파이프라인을 개선합니다.

이 전용 시간은 기술 부채가 쌓이는 것을 막고 fluo 생태계를 민첩하고 고성능으로 유지하는 데 필요합니다. 유지보수 기여도 기능 기여만큼 중요하게 다룹니다.

## Local Development Workflow

fluo 저장소를 로컬에 설정하고 기여를 시작하려면, 로컬 환경이 CI 기대치와 일치하도록 다음 단계를 따르세요:

1. **사전 요구 사항**: Node.js(LTS), `pnpm`(최신 버전), 그리고 `git`이 설치되어 있는지 확인하세요. fluo는 효율적인 워크스페이스 관리와 엄격한 의존성 해석을 위해 `pnpm`을 사용합니다.
2. **복제 및 설치**:
```bash
# Clone the repository
git clone https://github.com/fluojs/fluo.git
cd fluo

# Install dependencies
pnpm install
```
3. **설치 확인**: 로컬 설정이 정상이고 저장소의 기본 상태와 일치하는지 확인하기 위해 전체 검증 스위트를 실행합니다.
```bash
# Run verification
pnpm verify
```
4. **개발 루프**: 특정 패키지에서 작업할 때 필터 명령을 사용해 감시 프로세스를 가볍게 유지하세요. 빠른 피드백 루프를 위해 테스트를 watch 모드로 실행할 수도 있습니다.
```bash
pnpm --filter @fluojs/core dev

pnpm --filter @fluojs/core test:watch
```

메인테이너는 격리된 이슈 작업에 **git worktrees**를 사용하는 것이 권장됩니다. 별도 디렉토리에서 장기 기능을 작업하는 동안 `main` 브랜치를 깨끗하게 유지하고 긴급 핫픽스에 대비할 수 있습니다. 이 관행은 컨텍스트 전환 비용을 줄이고 잘못된 브랜치에 실수로 커밋할 위험도 낮춥니다. 또한 린팅 및 포맷팅 자동화를 돕기 위해 `.vscode/extensions.json`에 VS Code 권장 확장 도구 세트를 제공합니다. 일관된 에디터 설정은 스타일 관련 PR 리뷰 코멘트를 줄이는 데 도움이 됩니다.

## Final Words

fluo의 강점은 커뮤니티와 표준에 대한 일관된 약속에 있습니다. 프레임워크에 기여한다는 것은 TypeScript 백엔드가 더 명시적이고, 표준을 준수하며, 플랫폼에 덜 묶인 방향으로 발전하도록 돕는 일입니다. 단순한 오타 수정부터 주요 아키텍처 개선 또는 새로운 런타임 통합까지 모든 기여는 같은 검증 기준 아래에서 다룹니다.

기여자는 소프트웨어의 품질과 예측 가능성을 진지하게 다루는 개발자 그룹에 합류합니다. 각 기여는 fluo를 개선할 뿐 아니라 더 넓은 TypeScript 생태계가 더 나은 관행을 검토하도록 영향을 줄 수 있습니다. 첫 PR이든 토론의 의견이든, 중요한 것은 변경의 이유와 검증 결과를 명확히 남기는 것입니다. 이 가이드의 어떤 부분이든 혼란스럽다면 기여자 경험을 개선할 수 있도록 이슈를 열어 주세요.

기여자 간 지식 공유를 위해 매달 "fluo Tech Talk"도 개최합니다. 이 세션에서는 최근 도입된 주요 아키텍처 변경 사항이나 특정 성능 최적화 기법을 깊게 논의합니다. 기여자는 발표자로 참여할 수 있으며, 자신의 작업을 설계와 검증 관점에서 공유할 수 있습니다. fluo는 도구이면서 동시에 학습 가능한 기술 커뮤니티로 운영됩니다.

기여 과정의 모든 소통은 서로를 존중하는 어조로 이루어져야 합니다. 건설적인 비판은 환영하지만, 인격적인 비하로 이어지는 표현은 엄격히 금지합니다. 안전하고 포용적인 환경이 더 나은 기술적 판단을 가능하게 하기 때문입니다. 좋은 기여 문화는 코드 품질을 지키는 기반입니다.

문서화의 중요성도 다시 강조할 필요가 있습니다. 코드는 언젠가 변경되거나 삭제될 수 있지만, 그 뒤에 담긴 의도와 설계 결정은 문서로 남아 후속 기여자에게 중요한 자산이 됩니다. 따라서 모든 주요 기여에는 배경을 설명하는 문서 업데이트가 따라야 합니다. 이는 fluo가 지속 가능한 프레임워크로 남기 위한 필수 조건입니다.

기여자의 전문성은 fluo의 품질을 높이는 직접적인 힘입니다. 작은 수정도 명확한 근거와 검증을 갖추면 저장소의 장기 안정성에 보탬이 됩니다. 첫 기여는 작게 시작해도 충분합니다. 중요한 것은 변경 범위를 분명히 하고, 검증 결과를 함께 남기는 습관입니다.

fluo의 미래 로드맵이 궁금하다면 `docs/roadmap/` 디렉토리를 참고하세요. 프로젝트가 어디로 가고 있는지, 어떤 영역에서 도움이 필요한지 확인할 수 있습니다. 로드맵을 먼저 읽으면 제안하려는 변경이 현재 방향과 맞는지도 판단하기 쉽습니다.

기여를 시작하기 전에 `docs/CONTRIBUTING_GUIDELINES_EXTENDED.md`도 확인해야 합니다. 이 문서에는 코드 스타일 가이드, 커밋 메시지 컨벤션, 브랜치 전략, Pull Request 템플릿 작성법에 대한 세부 지침이 포함되어 있습니다. 이 지침을 따르면 리뷰어의 부담이 줄고, 변경이 더 빠르게 검토될 수 있습니다.

프레임워크의 내부 동작을 더 잘 이해할 수 있도록 "Deep Dive" 기술 블로그 시리즈도 운영합니다. 이 블로그에서는 의존성 주입 엔진의 구현 세부 사항이나 데코레이터 메타데이터가 런타임에 처리되는 방식 등을 다룹니다. 이런 자료는 변경의 영향을 더 정확히 예측하는 데 도움이 됩니다.

fluo 생태계는 코어 프레임워크에만 머물지 않습니다. 커뮤니티 어댑터, 플러그인, 스타터 템플릿을 개발해 생태계 확장에 기여할 수도 있습니다. 이런 외부 기여 역시 공식 문서의 "Ecosystem" 섹션에 소개될 수 있으며, 사용자에게 실질적인 선택지를 제공합니다.

기여 과정에서 발생하는 피드백은 저장소를 개선하는 중요한 입력입니다. 리뷰 과정에서 이해되지 않는 부분이 있거나 다른 의견이 있다면 근거와 함께 의견을 제시하세요. 건강한 기술적 토론은 더 나은 설계를 만드는 핵심 동력입니다. 의견은 사람보다 설계와 증거를 향해야 합니다.

기여를 통해 얻는 큰 자산 중 하나는 다양한 개발자와 교류하며 얻는 통찰입니다. fluo는 오픈소스 프로젝트이자 기술적 우수성을 추구하는 엔지니어들의 네트워크입니다. 이 네트워크 안에서 중요한 것은 영향력 있는 변경을 작게 쪼개고, 검증 가능한 근거로 논의하는 태도입니다.

모든 기여는 기록되고 릴리스 노트와 변경 이력의 일부가 됩니다. 매년 연말에는 우수 기여자를 선정해 fluo 에코시스템의 특별한 굿즈와 감사 이벤트도 진행합니다. 다만 기여의 핵심은 보상보다 저장소 품질을 함께 책임지는 데 있습니다. 변경의 범위와 검증을 명확히 남기는 것이 가장 좋은 출발점입니다.

함께 작업할 때 fluo는 더 안정적인 프레임워크가 됩니다. 참여는 코드, 문서, 테스트, 이슈 재현 등 여러 형태를 가질 수 있습니다. 이 가이드를 기준으로 변경 범위를 정하고, 필요한 검증을 수행한 뒤, PR 설명에 결과를 남기세요.

새로운 기여자를 위한 "멘토링 프로그램"도 운영합니다. 처음 기여를 시작할 때 어디서부터 손을 대야 할지 어렵다면 `mentor-needed` 라벨이 붙은 이슈를 찾으세요. 핵심 팀 멤버 중 한 명이 첫 기여 과정을 1:1로 안내할 수 있습니다. 이 프로그램은 fluo의 개발 문화와 검증 기준에 빠르게 적응하도록 돕기 위해 설계되었습니다.

또한 매 분기마다 "Contributor Day"를 온라인으로 개최합니다. 이 날은 전 세계의 기여자들이 모여 실시간으로 협업하고, 새로운 아이디어를 나누며, 복잡한 버그를 함께 해결하는 해커톤 형식의 행사입니다. 참여하려면 뉴스레터를 구독하거나 공식 슬랙 채널에 가입하면 됩니다.

테스트 철학도 짚고 넘어가야 합니다. fluo는 "Test-Driven Development(TDD)"를 강력히 권장합니다. 새 기능을 추가할 때는 해당 기능의 명세를 정의하는 테스트 코드를 먼저 작성하세요. 이는 버그를 사전에 줄일 뿐 아니라 설계 결함을 조기에 발견하는 데 도움이 됩니다. 견고한 테스트는 fluo의 안정성을 지탱하는 핵심 기둥입니다.

fluo의 보안 정책도 반드시 확인해야 합니다. 심각한 보안 취약점을 발견했다면 공개 이슈로 등록하지 말고 `security@fluo.js`로 직접 이메일을 보내야 합니다. 제보된 취약점은 신속하게 분석하고 수정하며, 제보자의 공로를 공식적으로 인정합니다. 보안은 공동 책임이며, 세심한 관찰이 fluo를 더 안전한 프레임워크로 만듭니다.

기여 여정은 작은 재현 사례나 문서 수정에서 시작해도 충분합니다. 중요한 것은 저장소 규칙을 따르고, 변경의 이유와 검증 결과를 분명히 남기는 것입니다.

기여자가 프레임워크 성능을 직접 측정하고 개선할 수 있도록 "Benchmark Suite"도 제공합니다. `scripts/bench/` 디렉토리에는 다양한 시나리오에 대한 성능 측정 스크립트가 포함되어 있습니다. 최적화를 제안한다면 실제 효과를 데이터로 증명하세요. 성능은 fluo가 포기할 수 없는 핵심 가치 중 하나입니다.

기여 과정에서 문서화가 필요하다면 `docs/style-guide.md`를 따르는 것을 권장합니다. 기술 문서는 명확하고 간결해야 하며, 독자가 바로 실행해 볼 수 있는 예제를 포함해야 합니다. 좋은 문서는 코드의 가치를 높이고, 다른 개발자가 변경의 의도를 빠르게 이해하게 합니다.

지역별 커뮤니티 활동도 지원합니다. 지역에서 fluo 밋업이나 스터디를 운영하고 싶다면 핵심 팀에 연락하세요. 로고 사용 권한, 스피커 지원, 홍보 등 가능한 지원 범위를 안내할 수 있습니다. fluo는 전 세계 어디에서나 사용할 수 있는 글로벌 표준을 지향합니다.

기여를 통해 배우게 되는 또 다른 가치는 협업의 방식입니다. 여러 사람의 생각이 하나의 프레임워크에 반영되려면 명확한 제안, 검증 가능한 근거, 차분한 합의 과정이 필요합니다. 그 과정에서 생기는 의견 차이를 설계와 데이터 중심으로 조율하는 경험은 엔지니어링 판단력을 키웁니다.

fluo를 사용하는 기업을 위한 "Enterprise Support" 기여 채널도 열어두고 있습니다. 실제 비즈니스 현장에서 발생하는 복잡한 요구사항을 프레임워크에 반영하는 작업은 어렵지만 가치가 큽니다. 기업 사용자의 피드백은 프레임워크의 경계 조건과 운영 요구를 확인하는 데 도움이 됩니다.

모든 코드 라인과 문장은 fluo의 변경 이력에 남습니다. 그래서 작은 수정도 책임 있게 작성해야 합니다. `CONTRIBUTING.md`를 읽고, 범위가 명확한 첫 이슈부터 시작하세요. 좋은 기여는 작아도 검증 가능하고 유지보수 가능해야 합니다.

기여자 편의를 위해 전용 개발 컨테이너 설정(`devcontainer.json`)도 제공합니다. 도커를 사용한다면 복잡한 로컬 설정 없이 개발을 시작할 수 있습니다. 일관된 개발 환경은 환경 차이에서 오는 버그를 줄이고 기여 속도를 높이는 데 도움이 됩니다.

fluo의 라이선스 정책도 확인해야 합니다. 모든 코드는 MIT 라이선스를 따르며, 이는 기여가 전 세계 사용자에게 자유롭게 공유되고 사용될 수 있음을 의미합니다. 오픈소스 라이선스의 의미를 이해하고 기여하는 것이 중요합니다.

프로젝트 투명성을 높이기 위해 매 분기마다 "Transparency Report"를 발행합니다. 이 리포트에는 해당 분기 동안 처리된 보안 취약점의 개수, 주요 아키텍처 결정 사항 요약, 기여자가 제안한 RFC의 승인 현황 등이 공개됩니다. 이런 정보 공유는 커뮤니티 신뢰를 바탕으로 프로젝트를 운영하는 데 필요합니다. 누구나 이 리포트를 통해 fluo가 어떤 방향으로 나아가는지, 어떤 도전 과제를 다루는지 확인할 수 있습니다.

fluo의 기술 스택을 더 깊이 이해하려는 개발자를 위해 "Advanced Learning Path"도 별도로 구축하고 있습니다. 이 경로는 단순한 API 사용법을 넘어 컴파일러 플러그인 작성법이나 커스텀 DI 스코프 설계 같은 고난도 주제를 단계별로 다룹니다. 이 경로는 코어 설계에 가까운 기여를 준비하는 데 유용합니다.

기여자가 자신의 작업을 공유할 기회도 제공합니다. fluo에 기여한 내용이나 fluo를 사용해 구축한 프로젝트에 대해 기술 블로그를 작성하면, 공식 소셜 미디어 채널을 통해 소개될 수 있습니다. 이는 개인의 작업을 알리는 동시에 fluo 생태계의 다양한 사용 사례를 보여주는 자료가 됩니다.

fluo 커뮤니티에 참여할 때는 작은 변경도 같은 책임감으로 다루면 됩니다. 저장소에서 Pull Request를 열기 전에는 관련 테스트와 문서 업데이트를 확인하세요. 좋은 기여는 화려한 규모보다 명확한 목적, 보존된 계약, 재현 가능한 검증으로 판단됩니다.
