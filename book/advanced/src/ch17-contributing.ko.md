# Chapter 17. fluo Contributing Guide

fluo 시리즈의 마지막 장에 오신 것을 축하드립니다. 여기까지 오셨다면 표준 데코레이터의 복잡성, 의존성 주입, 그리고 고급 런타임 아키텍처를 완전히 마스터하셨다는 뜻입니다. 고급 fluo 개발자로서 논리적인 다음 단계는 프레임워크 자체를 형성하는 데 도움을 주는 것입니다.

fluo에 기여하는 것은 단순히 코드를 작성하는 것 이상입니다. 이는 엄격한 행동 계약(behavioral contracts), 명시적 설계, 그리고 플랫폼에 무관한 신뢰성이라는 문화에 참여하는 것을 의미합니다. 이 가이드는 fluo 저장소 구조, 기여 워크플로우, 그리고 생태계를 안정적으로 유지하는 거버넌스 모델에 대해 깊이 있게 다룹니다.

## Repository Structure and Philosophy

fluo 저장소는 `pnpm`으로 관리되는 고성능 모노레포입니다. 우리의 철학은 **행동 계약(Behavioral Contracts)**을 중심으로 합니다. 이는 모든 변경 사항이 단순히 기능뿐만 아니라, 다양한 런타임(Node.js, Bun, Workers)에서 프레임워크의 예측 가능성에 미치는 영향에 의해 평가됨을 의미합니다.

우리는 "평면적이지만 계층화된(Flat but Layered)" 전략을 고수합니다. 모든 패키지는 가능한 한 얇게 유지되어야 하며, 복잡한 로직은 전문화된 모듈로 위임하는 동시에 통일된 인터페이스를 유지해야 합니다. 이를 통해 fluo의 핵심은 작고 빠르게 유지되면서도 생태계는 무한히 확장될 수 있습니다. 우리는 너무 많은 일을 하려는 "만능 패키지(God package)"를 피하고, 단일 책임 원칙(Single Responsibility Principle)을 따르는 작고 조합 가능한 단위를 선호합니다.

### Workspace Organization

- `packages/`: 프레임워크의 모듈식 구성 요소를 포함합니다. 각 패키지(예: `@fluojs/core`, `@fluojs/di`)는 모노레포의 전역 표준을 준수하면서 독립적인 수명 주기를 유지해야 합니다.
- `docs/`: 운영 정책, RFC, 아키텍처 결정 기록(ADR)을 포함한 중앙 집중식 문서입니다.
- `examples/`: 검증을 위한 표준 애플리케이션 설정들입니다. 이는 학습 리소스이자 전체 생태계에 대한 스모크 테스트(smoke test) 역할을 합니다.
- `.github/`: 기여 품질 게이트를 강제하는 워크플로우 정의 및 이슈/PR 템플릿입니다.
- `scripts/`: 워크스페이스 관리, 릴리스 준비 및 벤치마킹을 위한 내부 도구입니다.
- `tools/`: 모노레포 전반의 일관성을 보장하는 재사용 가능한 빌드 도구 및 공유 설정(ESLint, Vitest)입니다.

`packages/` 디렉토리의 모든 패키지는 자체 테스트 스위트와 문서를 가진 독립된 단위로 취급되지만, 코드 스타일, 라이선스 및 행동 계약에 관한 전역 저장소 정책을 준수합니다.

## Issue and Label Workflow

우리는 메인테이너의 시간이 영향력 있는 작업에 집중될 수 있도록 매우 구조화된 이슈 접수 프로세스를 사용합니다. 이러한 구조는 "유지보수 피로"를 방지하고 개발 속도를 높게 유지합니다. 우리의 목표는 초기 제출부터 최종 해결까지 모든 보고에 대해 명확한 경로를 제공하는 것입니다.

### Issue Templates

fluo 저장소에서는 빈 이슈(blank issue)가 비활성화되어 있습니다. 모든 이슈는 시작부터 필요한 모든 컨텍스트가 제공되도록 다음 템플릿 중 하나를 따라야 합니다. 이는 불필요한 질의응답을 줄이고 메인테이너가 더 효과적으로 이슈를 분류할 수 있게 합니다:
- **Bug Report**: 최소 재현 사례(stackblitz 또는 저장소)와 예상 동작과 실제 동작에 대한 명확한 설명이 필요합니다. 환경 상세 정보(Node 버전, OS 등)를 포함해 주세요.
- **Feature Request**: 상세한 "이유(Why)"와 "방법(How)" 제안이 필요하며, 이것이 어떻게 행동 계약 정책(Behavioral Contract Policy)과 일치하는지 포함해야 합니다. 해결하려는 문제와 기존 API로 처리할 수 없는 이유를 설명해 주세요.
- **Documentation Issue**: 가이드의 누락, 오타 또는 개념적 오류를 수정하기 위한 것입니다. 명확한 문서는 코드만큼이나 중요합니다.
- **DX/Maintainability**: 개발자가 더 빠르고 안정적으로 작업할 수 있도록 돕는 내부 개선 사항(예: CI 최적화, 빌드 도구 업데이트)을 위한 것입니다.

작업 목록을 실행 가능한 상태로 유지하고 커뮤니티에서 검색 가능하게 하기 위해 질문은 이슈 트래커가 아닌 **GitHub Discussions**로 유도되어야 합니다.

### Labeling System

이슈는 사용된 템플릿에 따라 자동으로 라벨이 지정되지만, 메인테이너는 종종 분류를 구체화하고 상태를 알리기 위해 보조 라벨을 추가합니다:
- `bug`: 계약을 위반하는 확인된 회귀(regression) 또는 예기치 않은 동작.
- `enhancement`: 프레임워크의 기능을 확장하는 새로운 기능 또는 개선 사항.
- `type:maintainability`: 내부 정리, 도구 개선 또는 기술 부채 감소.
- `priority:p0` ~ `p2`: 이슈의 심각도. P0 이슈는 일반적으로 다음 릴리스를 차단합니다.
- `status:needs-repro`: 제보자가 최소 재현 사례를 제공할 때까지 이슈 작업이 차단됩니다.
- `good first issue`: 코드베이스에 익숙해지려는 새로운 기여자에게 적합합니다.
- `help wanted`: 핵심 팀이 즉시 처리할 여력이 없는 중요한 작업들입니다.

## Review Culture

fluo에서 Pull Request를 리뷰하는 것은 엄격한 프로세스입니다. 우리는 단순히 "LGTM"만 하지 않고 검증합니다. 우리의 목표는 어떤 PR도 시스템의 성능이나 신뢰성을 저하시키지 않도록 보장하는 것입니다.

### Verification Gate

모든 PR은 다음과 같은 복합 작업을 실행하는 `pnpm verify` 명령을 통과해야 합니다:
- **린팅 및 포맷팅**: Prettier 및 특수 ESLint 규칙을 통해 일관성을 보장합니다. 우리는 코드베이스를 읽기 쉽고 유지 관리하기 쉬운 상태로 유지하기 위해 엄격한 규칙 세트를 적용합니다.
- **단위 및 통합 테스트**: 모든 패키지에 대해 전체 Vitest 스위트를 실행합니다. 우리는 핵심 패키지에 대해 100% 커버리지를 목표로 하며, 예외 상황을 다루는 고품질 테스트를 기대합니다.
- **타입 체크**: 전체 워크스페이스 그래프에서 타입 안전성을 보장하기 위해 `tsc --noEmit`을 실행합니다. 여기에는 암시적인 `any` 사용이나 안전하지 않은 타입 단언(assertion)에 대한 확인도 포함됩니다.
- **빌드 검증**: 모든 패키지 타겟(CJS, ESM, 때로는 UMD)에 대해 `pnpm build`가 오류 없이 완료되는지 확인합니다. 또한 번들 크기 저하 여부도 확인합니다.
- **의존성 감사**: 새로운 의존성을 확인하고 라이선스가 올바른지, 보안 취약점을 도입하지 않는지 확인합니다.

### Behavioral Contract Review

고급 기여자로서 여러분의 리뷰는 변경 사항이 기존 계약을 보존하는지에 집중해야 합니다. `@fluojs/di`의 최적화가 `@fluojs/platform-cloudflare-workers`의 스코핑 규칙을 깨뜨리지는 않는지? `@fluojs/core`의 새로운 데코레이터가 TC39 표준을 준수하는지 확인해야 합니다.

우리는 **편의성보다 사양 준수(spec-compliance)**를 우선시합니다. 제안된 기능이 ECMAScript 또는 TypeScript 표준에서 벗어나야 하는 경우, 순수하게 선택 사항인 외부 패키지로 구현될 수 없는 한 거부될 가능성이 높습니다. 추가되는 모든 코드 라인은 프레임워크의 핵심 원칙에 따라 방어 가능해야 합니다. 우리는 영리하지만 모호한 구현보다 명시성과 명확성을 가치 있게 여깁니다.

### Documentation First

PR이 공개 API를 추가하는 경우, 인라인 문서(JSDoc)와 `docs/` 또는 `packages/*/README.md`의 관련 마크다운 파일 업데이트를 **반드시** 포함해야 합니다. 우리는 문서화되지 않은 기능은 존재하지 않는다고 믿습니다. 리뷰어의 업무에는 이러한 추가 사항의 명확성, 정확성 및 어조를 확인하는 것도 포함됩니다. 문서는 "어떻게"와 "왜"를 모두 이해해야 하는 사용자의 관점에서 작성되어야 합니다. 또한 새로운 기능을 현실적인 맥락에서 보여주는 예제도 찾습니다.

## Release Process and Governance

fluo는 모듈식 생태계 전반에서 높은 안정성을 유지하기 위해 감독된 릴리스 모델을 따릅니다.

### Package Tiers

패키지는 사용자에게 안정성을 알리고 명확한 기대치를 설정하기 위해 세 가지 계층으로 분류됩니다:
- **Official**: 프로덕션 준비 완료, 엄격한 유의적 버전(semver)을 따르며 100% 테스트 커버리지를 가집니다. 이들은 fluo 생태계의 기둥입니다.
- **Preview**: 조기 채택자를 위한 준비 완료, 실제 피드백에 따라 변경될 수 있습니다. 중요하지 않은 환경에서 테스트하는 것을 권장합니다.
- **Experimental**: 인큐베이션 단계, 예고 없이 제거되거나 대폭 변경될 수 있습니다. 이곳은 우리가 혁신하고 위험을 감수하는 곳입니다.

### SEMVER and Migration Notes

0.x 버전에서도 우리는 파괴적 변경(breaking changes)을 매우 신중하게 다룹니다. 모든 파괴적 변경은 해당 패키지의 `CHANGELOG.md`에 상세한 마이그레이션 노트를 작성해야 합니다. 우리는 `changesets`를 사용하여 버전 관리 및 변경 이력 생성을 관리하며, 모든 중요한 커밋이 정확하게 반영되고 기여자가 명시되도록 보장합니다. 이러한 투명성은 사용자와의 신뢰를 구축하는 핵심입니다.

### Release Operations

릴리스 운영은 반복 가능성과 보안을 보장하기 위해 GitHub Actions를 통해 관리됩니다. 우리는 "감독된 자동(supervised-auto)" 모델을 사용합니다:
1. 메인테이너가 로컬에서 `pnpm verify:release-readiness`를 실행하여 워크스페이스 일관성과 보류 중인 변경 사항을 확인합니다.
2. Changeset이 `main` 브랜치에 병합되면 새 버전을 계산하는 "Version Packages" PR이 트리거됩니다.
3. 버전 관리 PR이 병합되면 GitHub Action이 자동으로 패키지를 npm에 게시하고 GitHub Release를 생성합니다.
이 프로세스는 불완전한 코드의 실수로 인한 게시를 방지하고, 게시된 아티팩트가 저장소의 소스 코드와 정확히 일치하도록 보장합니다.

## Governance and RFC Workflow

작은 수정이나 문서 업데이트는 직접 PR할 수 있지만, 중요한 아키텍처 변경은 커뮤니티의 합의와 기술적 우수성을 보장하기 위해 RFC(Request for Comments) 프로세스를 거쳐야 합니다. 이 프로세스는 아키텍처의 변질을 방지하고 모든 주요 추가 사항이 충분한 근거를 갖도록 설계되었습니다.

### The RFC Path

1. **GitHub Discussions**: 커뮤니티의 관심도를 측정하고 잠재적인 함정을 조기에 식별하기 위해 "Ideas" 또는 "RFC" 카테고리에 스레드를 시작합니다. 이곳은 부담 없는 브레인스토밍을 위한 장소입니다.
2. **Formal Proposal**: 복잡한 변경 사항의 경우, 마크다운 제안서(`packages/graphql/field-resolver-rfc.md`의 템플릿 참조)를 작성하고 `docs/proposals` 디렉토리에 PR을 엽니다. 제안서는 동기, 상세 설계, 단점 및 대안을 다루어야 합니다.
3. **Review and Consensus**: 핵심 메인테이너와 커뮤니티가 RFC를 리뷰합니다. 구현을 시작하기 전에 최소 두 명의 핵심 메인테이너의 승인이 필요합니다. 우리는 기술적 견고함과 fluo의 핵심 철학과의 일치 여부를 확인합니다.
4. **Implementation**: RFC가 병합("Accepted")되면, 작업은 실행 가능한 이슈로 분할되어 기여자에게 할당됩니다. 원본 RFC 작성자가 종종 이 단계의 리더가 되어 연속성과 비전을 제공합니다.
5. **Finalization**: 구현과 문서화가 완료되면 RFC는 "Implemented" 상태로 이동하여 설계 결정의 영구적인 기록으로 남습니다.

### Behavioral Contract Policy

모든 기여자는 `docs/operations/behavioral-contract-policy.md`를 준수해야 합니다. 이 정책은 fluo의 "헌법"과 같습니다. 이는 JavaScript 언어 경로에서 벗어나는 비표준 TypeScript 기능의 사용을 금지함으로써 fluo가 "표준 우선" 프레임워크로 남을 수 있도록 보장합니다. 표준에 대한 이러한 약속이 fluo를 미래지향적으로 만듭니다.

이 정책은 다음과 같은 몇 가지 핵심 영역을 다룹니다:
- **데코레이터 표준**: TC39 표준 데코레이터(TypeScript 5.0+에서 지원)만 허용됩니다. 레거시 실험적 데코레이터는 엄격히 금지됩니다.
- **리플렉션 및 메타데이터**: `reflect-metadata`의 사용은 지양되며, 명시적 레지스트리 및 표준 메타데이터 제안을 선호합니다.
- **런타임 추상화**: 모든 I/O 및 환경 특정 로직은 Node.js, Bun, Deno 및 Workers 간의 이식성을 보장하기 위해 `Platform` 추상화 레이어 뒤에 배치되어야 합니다.
- **오류 처리**: 확인되지 않은 예외보다는 도메인 수준의 오류에 대해 `Result` 패턴을 사용하는 것을 권장합니다.
- **명시적 제어의 역전(IoC)**: DI 시스템은 가능한 한 의존성 그래프의 정적 분석이 가능한 방식으로 사용되어야 합니다.

이러한 규칙을 따름으로써 우리는 코드베이스가 TypeScript 생태계의 특정 특성에 깊이 빠져 있는 사람뿐만 아니라 모든 JavaScript 개발자가 접근할 수 있는 상태로 유지되도록 보장합니다.

또한, 우리는 "공개 내보내기 기준(Public Export Baseline)"을 엄격하게 유지합니다. 이는 패키지에서 내보내는 모든 함수, 클래스 및 인터페이스가 완전한 TSDoc 주석을 가져야 함을 의미합니다. 여기에는 `@param`, `@returns`, 그리고 최소 하나 이상의 `@example` 블록이 포함되어야 합니다. 이 기준은 우리의 문서가 이 가이드와 같은 문서에만 존재하는 것이 아니라, 개발자가 프레임워크를 사용할 때 IDE 내에서 직접 사용 가능하도록 보장합니다. 우리는 이러한 세부 사항이 전문적인 프레임워크에 필수적이며 새로운 기여자와 사용자의 학습 곡선을 최소화하는 데 도움이 된다고 믿습니다.

표준 우선 접근 방식 외에도 우리는 "최소 의존성" 정책을 시행합니다. 패키지에 추가되는 모든 새로운 의존성은 PR에서 정당화되어야 하며, 이상적으로는 의존성이 없거나 매우 적은 모듈이어야 합니다. 이는 우리의 공급망을 안전하게 유지하고 번들 크기를 예측 가능하게 만듭니다. 또한 가능할 때마다 특수 라이브러리보다 네이티브 Node.js 및 웹 API를 선호하여 장기적인 유지보수성 및 플랫폼 이식성이라는 목표를 더욱 강화합니다.

우리는 또한 코드의 품질을 높이기 위해 "동형적 테스트(Isomorphic Testing)"를 장려합니다. 이는 동일한 비즈니스 로직이 Node.js 환경뿐만 아니라 브라우저 환경에서도 동일하게 작동하는지 확인하는 과정입니다. 이를 위해 Vitest와 같은 최신 도구를 사용하여 가상 환경에서의 검증을 병행하며, 이를 통해 클라우드플레어 워커(Cloudflare Workers)나 번(Bun)과 같은 다양한 런타임에서도 fluo가 문제없이 동작할 수 있도록 보장합니다.

기여자 여러분이 PR을 제출할 때, 이러한 정책들이 어떻게 적용되었는지 설명하는 것은 리뷰 과정을 훨씬 더 원활하게 만듭니다. 우리는 기술적인 구현뿐만 아니라 그 이면에 있는 철학적 일관성도 중요하게 생각하기 때문입니다. fluo의 모든 코드는 이유 없이 추가되지 않으며, 모든 결정은 표준과 성능, 그리고 명시성이라는 핵심 가치를 지향합니다.

#### Maintenance Friday and Technical Debt

매주 금요일, 핵심 팀과 기여자들은 "Maintenance Friday"를 위해 시간을 할애합니다. 이는 새로운 기능을 개발하는 대신 다음에 집중하는 집중 세션입니다:
- **의존성 업그레이드**: 최신 보안 패치 및 라이브러리 버전으로 워크스페이스를 최신 상태로 유지합니다.
- **리팩토링**: 코드 명확성을 개선하고 오래된 모듈의 복잡성을 줄입니다.
- **테스트 스위트 확장**: 커버리지를 높이고 최근 수정된 버그에 대한 회귀 테스트를 추가합니다.
- **CI/CD 최적화**: 더 빠른 피드백 루프를 위해 빌드 및 검증 파이프라인을 개선합니다.

우리는 이 전용 시간이 기술 부채의 축적을 방지하고 fluo 생태계가 민첩하고 고성능으로 유지되도록 하는 데 필수적이라는 것을 알고 있습니다. 모든 기여자가 이러한 유지보수 노력에 동참하는 것을 환영합니다!

## Local Development Workflow

fluo 저장소를 로컬에 설정하고 기여를 시작하려면, 로컬 환경이 CI 기대치와 일치하도록 다음 단계를 정확히 따르세요:

1. **사전 요구 사항**: Node.js(LTS), `pnpm`(최신 버전), 그리고 `git`이 설치되어 있는지 확인하세요. 우리는 효율적인 워크스페이스 관리와 엄격한 의존성 해결을 위해 특히 `pnpm`을 사용합니다.
2. **복제 및 설치**:
```bash
# Clone the repository
git clone https://github.com/fluojs/fluo.git
cd fluo

# Install dependencies
pnpm install
```
3. **설치 확인**: 로컬 설정이 정상이고 저장소의 기본 상태와 일치하는지 확인하기 위해 전체 검증 스위트를 실행합니다.
```bash
# Run verification
pnpm verify
```
4. **개발 루프**: 특정 패키지에서 작업할 때 필터 명령을 사용하여 감시 프로세스를 가볍게 유지하세요. 빠른 피드백 루프를 위해 테스트를 watch 모드로 실행할 수도 있습니다.
```bash
# 특정 패키지의 변경 사항 감시
pnpm --filter @fluojs/core dev

# 패키지에 대해 테스트를 watch 모드로 실행
pnpm --filter @fluojs/core test:watch
```

메인테이너는 격리된 이슈 작업을 위해 **git worktrees**를 사용하는 것이 권장됩니다. 이를 통해 별도의 디렉토리에서 장기적인 기능을 작업하는 동안 `main` 브랜치를 깨끗하게 유지하고 긴급한 핫픽스에 대비할 수 있습니다. 이 관행은 컨텍스트 전환 오버헤드를 방지하고 잘못된 브랜치에 실수로 커밋할 위험을 줄여줍니다. 또한, 린팅 및 포맷팅 자동화를 돕기 위해 `.vscode/extensions.json`에 VS Code 권장 확장 도구 세트를 제공합니다. 일관된 에디터 설정을 사용하면 스타일에 관한 사소한 PR 리뷰 코멘트를 줄이는 데 도움이 됩니다.

마지막으로, fluo의 보안 정책에 대해 유의해 주시기 바랍니다. 만약 심각한 보안 취약점을 발견하셨다면, 공개적인 이슈로 등록하지 마시고 `security@fluo.js`로 직접 이메일을 보내 주십시오. 우리는 제보된 취약점을 신속하게 분석하고 수정하며, 제보자의 공로를 공식적으로 인정할 것입니다. 보안은 우리 모두의 책임이며, 여러분의 세심한 관찰이 fluo를 더욱 안전한 프레임워크로 만듭니다.

여러분의 여정이 지금 시작됩니다. fluo와 함께 더 나은 개발 세상을 만들어 갑시다. 행운을 빕니다!

우리는 또한 기여자들이 프레임워크의 성능을 직접 측정하고 개선할 수 있도록 "Benchmark Suite"를 제공합니다. `scripts/bench/` 디렉토리에는 다양한 시나리오에 대한 성능 측정 스크립트가 포함되어 있습니다. 여러분이 제안하는 최적화가 실제로 얼마나 효과적인지 데이터로 증명해 보세요. 성능은 fluo가 포기할 수 없는 최고의 가치 중 하나입니다.

기여 과정에서 문서화가 필요한 경우, 우리는 `docs/style-guide.md`를 따를 것을 권장합니다. 기술 문서는 명확하고, 간결하며, 독자가 바로 실행해 볼 수 있는 예제를 포함해야 합니다. 좋은 문서는 코드의 가치를 두 배로 만듭니다. 여러분의 설명이 다른 개발자들에게 큰 영감이 될 것입니다.

우리는 또한 지역별 커뮤니티 활동을 지원합니다. 만약 여러분의 지역에서 fluo 밋업이나 스터디를 운영하고 싶으시다면, 핵심 팀에 연락해 주세요. 로고 사용 권한부터 시작하여 스피커 지원, 홍보 등 가능한 모든 도움을 드릴 준비가 되어 있습니다. fluo는 전 세계 어디에서나 사용될 수 있는 글로벌 표준을 지향합니다.

기여를 통해 배우게 될 또 다른 중요한 가치는 "협업의 미학"입니다. 수많은 사람들의 생각이 모여 하나의 프레임워크를 완성해 나가는 과정은 그 자체로 거대한 예술 작품과 같습니다. 그 과정에서 발생하는 갈등을 조율하고 최선의 합의점을 찾아가는 경험은 여러분을 더 훌륭한 엔지니어로 성장시킬 것입니다.

우리는 또한 fluo를 사용하는 기업들을 위해 "Enterprise Support" 기여 채널도 열어두고 있습니다. 실제 비즈니스 현장에서 발생하는 복잡한 요구사항들을 프레임워크에 녹여내는 작업은 매우 도전적이고 가치 있는 일입니다. 기업 사용자의 목소리를 프레임워크의 발전에 반영해 주세요.
