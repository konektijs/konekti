# Introduction: Scaling Beyond the Monolith

fluo 시리즈의 중급편에 오신 것을 환영합니다.

초급편에서 우리는 단일 인스턴스 애플리케이션을 견고하게 구축하는 데 집중했습니다. fluo의 핵심인 표준 우선(Standard-First) 데코레이터 모델, 의존성 주입(DI), 그리고 기초적인 HTTP 레이어를 살펴보았습니다. 우리는 서비스 경계를 명확히 하고 엄격한 타입 안정성을 강조하며, 공유 프로세스 내에서 FluoShop의 초기 버전을 모듈형 모놀리스(Modular Monolith)로 구축했습니다.

중급편은 여러분의 전문적인 성장에 있어 중요한 이정표가 될 것입니다. 웹 서버 구축의 기초를 넘어, 이제 시스템 설계의 복잡성과 씨름하기 시작할 것이기 때문입니다. 여기에는 데이터 로컬리티(Data Locality), 네트워크 파티션, 서비스 디스커버리(Service Discovery)에 대한 고민이 포함됩니다. 모놀리스에서는 데이터 일관성이 단일 데이터베이스와 로컬 트랜잭션에 의해 보장되곤 하지만, 분산 시스템에서는 로컬 함수 호출에서 네트워크 바인딩된 메시지 패턴으로 사고를 전환하여 최종 일관성(Eventual Consistency)과 함께 살아가는 법을 배워야 합니다.

## The Intermediate Journey

이 책의 목표는 여러분이 fluo를 사용하여 모듈형 모놀리스에서 확장 가능하고 회복력이 있는 마이크로서비스 아키텍처로 전환하도록 돕는 것입니다. 우리는 단순히 이론만 이야기하지 않습니다. FluoShop을 여러 서비스, 프로토콜, 런타임에 걸친 분산 시스템으로 실제로 리팩토링하고 발전시켜 나갈 것입니다.

우리는 네 가지 주요 테마에 집중합니다.

1. **마이크로서비스 인프라**: 트랜스포트 프로토콜, 메시지 패턴, 그리고 fluo가 어떻게 네트워크 복잡성을 추상화하는지 이해합니다 (TCP, Redis, Kafka, NATS, gRPC).
2. **이벤트 기반 아키텍처**: 직접적인 요청 방식에서 벗어나 CQRS와 메시지 브로커를 활용한 디커플링된 이벤트 흐름으로 이동합니다.
3. **실시간 통신**: WebSocket 게이트웨이와 특화된 알림 시스템을 통해 요청-응답 모델을 넘어 확장합니다.
4. **플랫폼 이식성**: fluo의 통합 파사드(Facade)를 활용하여 Node.js, Bun, Deno, 그리고 에지(Edge) 환경(Cloudflare Workers)에서 로직을 실행합니다.

## The Evolution of FluoShop

이 책에서 FluoShop은 단순한 상점에서 분산된 생태계로 진화합니다. 우리는 이를 다음과 같은 전문 서비스들로 분리할 것입니다.

- **API 게이트웨이**: 데이터 집계와 횡단 관심사를 처리하는 진입점.
- **카탈로그 서비스**: 고성능 읽기 작업을 통해 제품과 재고를 관리.
- **주문 서비스**: 복잡한 비즈니스 워크플로우와 상태 변화를 오케스트레이션.
- **결제 서비스**: 엄격한 실패 처리와 외부 연동이 필요한 고위험 도메인.
- **알림 서비스**: 멀티 채널(Email, Slack, Discord) 알림을 처리하는 리액티브 컨슈머.

우리는 이 토폴로지를 사용하여 단순한 TCP부터 Kafka, NATS와 같은 강력한 브로커에 이르기까지 다양한 트랜스포트 프로토콜을 탐구할 것입니다. 통일된 프로그래밍 모델 덕분에 비즈니스 로직의 수정 없이 이러한 인프라를 교체할 수 있음을 직접 확인하게 될 것입니다.

## Philosophy: Explicit Over Implicit

이 여정 전반에 걸쳐 우리는 fluo의 핵심 철학인 **명시성이 암시성보다 우선한다(Explicit Over Implicit)**는 원칙을 고수할 것입니다.

분산 시스템은 본질적으로 복잡합니다. 세 개의 서비스와 메시지 브로커에 걸친 오류를 디버깅하는 것은 매우 어렵습니다. 숨겨진 리플렉션(Reflection), 자동 탐색, 불투명한 메타데이터와 같은 프레임워크의 마법은 데이터와 의존성의 실제 흐름을 가려 이 과정을 더 힘들게 만듭니다.

fluo에서 모든 것은 명시적인 프로바이더입니다. 의존성 그래프는 감사(Audit) 가능하며, 트랜스포트 설정은 명확합니다. 우리는 명시성을 선택함으로써 약간의 보일러플레이트를 감수하는 대신, 예측 가능성과 유지보수성이라는 거대한 이점을 얻습니다. 또한 이벤트의 안정적인 발행을 위한 아웃박스(Outbox) 패턴과 네트워크 재시도로 인한 중복 메시지 처리를 위한 멱등성 키 활용법을 배울 것입니다.

## How to Navigate This Book

이 책은 분산 시스템 구축의 자연스러운 흐름을 따르는 논리적인 파트들로 구성되어 있습니다.

- **Part 0. 마이크로서비스 준비**: 전략을 정의하고 첫 번째 트랜스포트로 TCP를 학습합니다.
- **Part 1. 메시지 브로커**: Redis, RabbitMQ, Kafka, NATS, gRPC를 통합하고 각 트레이드오프를 평가합니다.
- **Part 2. 이벤트 기반 아키텍처**: 이벤트 버스, CQRS, Saga 패턴, 분산 잠금을 구현합니다.
- **Part 3. 실시간 통신**: 멀티 서비스 환경에서 WebSocket 게이트웨이와 Socket.IO를 확장합니다.
- **Part 4. 알림 시스템**: 이메일, 슬랙, 디스코드를 통해 알림을 오케스트레이션합니다.
- **Part 5. API 확장**: GraphQL과 Drizzle, Mongoose와 같은 현대적인 ORM을 탐구합니다.
- **Part 6. 플랫폼 이식성**: 동일한 비즈니스 로직을 Bun, Deno, Cloudflare Workers에 배포합니다.

## Setting Expectations

마이크로서비스 구축은 만병통치약이 아닙니다. 네트워크 지연, 부분적 실패, 데이터 일관성, 운영 오버헤드와 같은 새로운 과제를 불러옵니다. 이 책은 이러한 도전 과제들을 피하지 않습니다. 마이크로서비스를 사용하지 *말아야* 할 때와 "분산된 모놀리스(Distributed Monolith)"의 함정, 그리고 아키텍처가 어떻게 의도적인 트레이드오프의 연속인지 논의할 것입니다.

fluo는 이 전환을 위해 만들어졌습니다. 이제 1장으로 넘어가 모놀리스를 깨뜨려 봅시다.

## 용어 사전

- **트랜스포트(Transport)**: 통신에 사용되는 기저 프로토콜(TCP, Kafka 등).
- **메시지 패턴(Message Pattern)**: 요청-응답 상호작용을 위한 고유 식별자.
- **이벤트 패턴(Event Pattern)**: 화이어-앤-포겟(Fire-and-forget) 브로드캐스트를 위한 고유 식별자.
- **클라이언트 프록시(Client Proxy)**: 원격 서비스로 메시지나 이벤트를 보내기 위해 사용하는 fluo 컴포넌트.
- **파사드(Facade)**: 서로 다른 런타임(Node, Bun, Edge)의 복잡성을 숨기는 추상화 레이어.

이 용어들은 fluo 마이크로서비스 생태계의 언어이므로 잘 익혀 두시기 바랍니다.

이제 아키텍처로 넘어가 보겠습니다.
