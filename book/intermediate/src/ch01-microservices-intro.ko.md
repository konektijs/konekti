# 1. Microservice Architecture and fluo Strategy

마이크로서비스는 크고 복잡한 애플리케이션을 네트워크를 통해 통신하는 작고 독립적인 서비스들로 분해합니다.

fluo는 비즈니스 로직을 한 번 작성하면 다양한 트랜스포트 프로토콜에 걸쳐 배포할 수 있는 통일된 프로그래밍 모델을 제공합니다.

이 장에서는 fluo 마이크로서비스 전략의 핵심 철학을 소개하고, 이 책 전반에 걸쳐 구축할 **FluoShop** 프로젝트의 기반을 마련합니다.

이 첫 장의 목표는 마이크로서비스를 과장되게 포장하는 것이 아닙니다.

어디에서 도움이 되고, 어디에서 비용이 커지며, fluo가 모듈형 모놀리스에서 분산 시스템으로 이동하는 부담을 어떻게 낮추는지 정의하는 것입니다.

장을 마치면 FluoShop의 서비스 이름과 책임을 설명할 수 있어야 하고, 어떤 브로커를 고르기 전에 왜 트랜스포트 독립성이 중요한지도 말할 수 있어야 합니다.

## 1.1 The FluoShop Topology

이 책을 통해 우리는 실무 마이크로서비스 패턴을 보여주기 위해 설계된 단계적 이커머스 프로젝트인 **FluoShop**을 구축할 것입니다.

우리의 아키텍처는 각기 특정 도메인을 담당하는 다섯 개의 핵심 서비스로 구성됩니다.

1. **API Gateway**: 모든 클라이언트 요청의 진입점입니다. 라우팅, 인증, 요청 집계를 담당합니다.
2. **Catalog Service**: 상품 정보, 카테고리, 재고 수준을 관리합니다.
3. **Order Service**: 주문 생성, 상태 전환, 다른 서비스 간 조율을 담당합니다.
4. **Payment Service**: 결제 트랜잭션과 외부 결제 제공업체 연동을 관리합니다.
5. **Notification Service**: 시스템 이벤트에 따라 이메일, SMS, 각종 알림을 발송합니다.

이 토폴로지는 빠르게 이해할 수 있을 만큼 작게 잡았습니다.

동시에 실제 시스템에서 중요한 경계를 드러낼 만큼은 충분히 풍부합니다.

게이트웨이는 클라이언트 지향 프로토콜 문제를 맡습니다.

카탈로그 서비스는 빠른 조회를 강조합니다.

주문 서비스는 비즈니스 워크플로의 조정 중심이 됩니다.

결제 서비스는 엄격한 실패 처리 규칙이 필요한 외부 위험 도메인을 대표합니다.

알림 서비스는 요청 경로와 느슨하게 결합되어야 하는 다운스트림 반응을 보여줍니다.

### 1.1.1 Architecture Diagram

시스템은 저지연과 높은 신뢰성 사이의 균형을 맞추기 위해 하이브리드 통신 모델을 따릅니다.

- **Requests (Synchronous)**: API Gateway는 요청-응답 패턴을 사용해 Catalog Service 및 Order Service와 통신합니다. Order Service는 현재 트랜스포트가 허용하는 방식으로 결제 승인을 위해 Payment Service를 호출합니다.
- **Events (Asynchronous)**: 서비스는 상태 변화를 알리기 위해 이벤트를 발행합니다. Catalog Service는 재고 갱신을 브로드캐스트할 수 있고, Payment Service는 Notification Service가 소비할 결제 성공 이벤트를 발행할 수 있습니다.

FluoShop의 첫 단계를 간단히 그리면 다음과 같습니다.

```text
Client
  -> API Gateway
      -> Catalog Service
      -> Order Service
          -> Payment Service
              -> Notification Service
```

이것이 모든 요청에 대한 엄격한 호출 체인은 아닙니다.

이것은 도메인 관계를 보여주는 지도입니다.

어떤 상호작용은 직접 요청입니다.

어떤 상호작용은 fire-and-forget 이벤트입니다.

또 어떤 흐름은 나중에 실패 복구가 더 중요해질 때 지속성 있는 브로커 기반 흐름으로 바뀝니다.

바로 그 진화 가능성 때문에 이 토폴로지를 1장에서 먼저 제시합니다.

이후 트랜스포트 장들이 채워 넣을 정신적 모델을 지금 준비하는 것입니다.

## 1.2 Unified Programming Model

fluo에서 마이크로서비스의 비즈니스 로직은 기본 네트워크 프로토콜과 분리되어 있습니다.

핸들러 내부에 트랜스포트 전용 코드를 작성하지 않습니다.

대신 데코레이터로 메시지 패턴을 정의하고, 트랜스포트 어댑터가 프레이밍, 직렬화, 전달 메커니즘을 처리하게 합니다.

이 분리는 트랜스포트 변경이 흔하기 때문에 중요합니다.

팀은 종종 직접 서비스 간 네트워킹으로 시작합니다.

그 뒤 재시도, 지속성, fan-out이 필요해졌다는 사실을 알게 됩니다.

애플리케이션 코드가 이미 특정 트랜스포트 클라이언트에 의존하고 있다면 마이그레이션 비용은 크게 증가합니다.

트랜스포트가 설정에만 머물러 있다면 마이그레이션의 중심은 운영 쪽으로 이동합니다.

### 1.2.1 Pattern-Based Routing

`@MessagePattern` 및 `@EventPattern`과 같은 데코레이터를 사용하면 fluo가 문자열 또는 정규식 패턴을 기반으로 들어오는 패킷을 올바른 핸들러로 라우팅할 수 있습니다.

```typescript
import { MessagePattern, EventPattern } from '@fluojs/microservices';

export class OrderHandler {
  @MessagePattern('orders.create')
  async createOrder(data: CreateOrderDto) {
    // 요청-응답 흐름을 처리합니다.
    // fluo가 직렬화와 프레이밍을 자동으로 처리합니다.
    return { id: 'order-123', status: 'pending' };
  }

  @EventPattern('orders.completed')
  async handleOrderCompleted(data: OrderCompletedEvent) {
    // fire-and-forget 이벤트 브로드캐스트를 처리합니다.
    // 발행자에게 응답을 보내지 않습니다.
  }
}
```

패턴 이름이 계약입니다.

트랜스포트는 전달 수단입니다.

덕분에 라우팅 결정이 코드베이스 안에서 읽기 쉬운 형태로 남습니다.

또한 테스트는 소켓 절차보다 핸들러 계약에 집중할 수 있습니다.

FluoShop에서는 같은 주문 생명주기가 여러 장에 걸쳐 반복되기 때문에 이 점이 특히 중요합니다.

인프라를 바꿀 때마다 주문 도메인을 다시 설명하고 싶지는 않습니다.

우리가 원하는 것은 트랜스포트가 변해도 도메인은 안정적으로 유지되는 것입니다.

### 1.2.2 Protocol Independence

이 추상화 덕분에 모듈의 트랜스포트 설정만 바꿔 TCP에서 Kafka, NATS, gRPC로 전환할 수 있습니다.

비즈니스 로직은 그대로 유지되므로 시스템을 다양한 인프라 요구 사항에 유연하게 적응시킬 수 있습니다.

프로토콜 독립성이 모든 트랜스포트가 완전히 동일하게 동작한다는 뜻은 아닙니다.

핸들러의 표면적이 충분히 안정적이어서, 트랜스포트 차이가 설정과 운영 정책, 엣지 케이스 의미론에 격리된다는 뜻입니다.

예를 들어 다음과 같습니다.

- TCP는 낮은 지연과 단순한 배포를 제공합니다.
- Pub/Sub 브로커는 fan-out 동작을 제공합니다.
- Durable stream은 복구 수단을 제공합니다.
- RPC 성향의 트랜스포트는 요청 계약을 강조합니다.

fluo의 가치는 이 모든 것을 똑같다고 가장하는 데 있지 않습니다.

오히려 비즈니스 핸들러, DTO, DI 구조를 일관되게 유지한 채 서비스 연결마다 적절한 트랜스포트를 고를 수 있게 해준다는 점에 있습니다.

## 1.3 Strategic Advantages

fluo의 마이크로서비스 모듈을 사용하면 몇 가지 전략적 이점을 얻을 수 있습니다.

- **Developer Velocity**: 소켓 관리나 브로커 전용 API에 신경 쓰지 않고 비즈니스 로직에 집중할 수 있습니다.
- **Operational Flexibility**: 개발 단계에서는 단순한 TCP로 시작하고, 운영 단계에서는 핸들러를 다시 쓰지 않고 durable broker로 이동할 수 있습니다.
- **Safety Defaults**: fluo는 거대 패킷, 위험한 정리 패턴, 전달 혼동처럼 즉흥적인 트랜스포트 코드에서 자주 나타나는 문제를 줄여줍니다.

여기에 책이 진행될수록 더 분명해지는 네 번째 이점도 있습니다.

공통된 프로그래밍 관례는 팀 간 조율 비용을 낮춥니다.

모든 서비스가 같은 핸들러 스타일로 패턴을 노출하면, 새로운 유지보수 담당자는 프레임워크 규칙을 다시 배우지 않고도 여러 도메인을 오갈 수 있습니다.

이는 하나의 서비스를 여러 개로 나눌 때의 부담을 낮춥니다.

반대로 어떤 마이크로서비스 경계가 성급한 판단이었다는 사실을 알게 되었을 때 다시 합치는 부담도 낮춥니다.

## 1.4 Deep Dive into the Microservice Module

`MicroservicesModule`은 fluo의 분산 기능의 핵심입니다.

이 모듈을 등록하면 fluo는 들어오는 패킷을 처리하고 프로바이더에 디스패치하기 위한 인프라를 구성합니다.

```typescript
import { Module } from '@fluojs/core';
import { MicroservicesModule, TcpMicroserviceTransport } from '@fluojs/microservices';

@Module({
  imports: [
    MicroservicesModule.forRoot({
      transport: new TcpMicroserviceTransport({ port: 4000 })
    })
  ]
})
export class AppModule {}
```

이 구성은 애플리케이션을 특정 트랜스포트에 바인딩합니다.

내부적으로 fluo는 `@MessagePattern` 또는 `@EventPattern`이 붙은 메서드를 찾아 트랜스포트 리스너에 등록합니다.

이 등록 파이프라인은 아키텍처 논의에서 중요합니다.

프레임워크가 숨겨진 리플렉션 마법으로 동작을 발견하는 것이 아니기 때문입니다.

명시적인 프로바이더들을 런타임 그래프로 조합하는 방식입니다.

이 설계는 마이크로서비스 기능을 fluo 생태계의 다른 구성 요소와도 같은 철학 위에 올려놓습니다.

FluoShop 관점에서 보면 실질적인 결과는 단순합니다.

이 파트의 모든 서비스 장은 애플리케이션 구조를 매번 처음부터 다시 설명하지 않고도 새로운 트랜스포트를 소개할 수 있습니다.

모듈은 트랜스포트 배선의 집입니다.

핸들러는 비즈니스 반응의 집입니다.

보조 프로바이더는 도메인 로직의 집입니다.

## 1.5 The Philosophy of "No Magic"

fluo는 많은 편의 기능을 제공하지만, "no magic" 철학을 고수합니다.

모든 컴포넌트는 DI 컨테이너 안의 명시적인 프로바이더입니다.

예를 들어 `MICROSERVICE` 토큰은 서비스에 클라이언트 프록시를 주입할 때 사용됩니다.

이 방식은 코드를 테스트 가능하고 예측 가능하게 유지합니다.

이 철학은 일반적인 웹 애플리케이션보다 분산 시스템에서 더 중요합니다.

요청이 네트워크를 건너가는 순간 모호함의 비용이 커지기 때문입니다.

재시도 동작, 직렬화 경계, 의존성 생명주기가 분명하지 않으면 디버깅은 여러 프로세스로 퍼집니다.

명시적 설정은 그런 혼란을 줄입니다.

패턴 해석이 멈추거나 클라이언트가 타임아웃을 내기 시작했을 때 어디를 봐야 하는지 알려줍니다.

## 1.6 Why Microservices with fluo?

전통적인 마이크로서비스 프레임워크는 프로토콜 세부 사항을 비즈니스 로직에 노출하는 경우가 많습니다.

REST 기반 접근으로 시작했다가 나중에 확장성을 위해 메시지 브로커가 필요해지면, 서비스의 상당 부분을 다시 작성해야 하는 일이 흔합니다.

fluo는 트랜스포트를 교체 가능한 구현 세부 사항으로 다루어 이런 마찰을 줄입니다.

물론 이것이 모든 마이그레이션 작업을 없애 주는 것은 아닙니다.

여전히 타임아웃 예산, 멱등성 보장, 전달 의미론, 관측 가능성을 다시 검토해야 합니다.

하지만 프레임워크 모델이 바뀌었다는 이유만으로 모든 핸들러 시그니처를 다시 설계할 필요는 없어집니다.

FluoShop에서는 이 덕분에 학습 경로가 누적형이 됩니다.

2장은 TCP의 트레이드오프에 집중할 수 있습니다.

3장은 Redis의 트레이드오프에 집중할 수 있습니다.

이후 장들은 더 무거운 브로커를 도입할 수 있습니다.

프로젝트는 매번 리셋되지 않고 계속 앞으로 진행됩니다.

## 1.7 Summary

- **Scalability**: 마이크로서비스는 독립적인 확장을 제공하지만 강력한 통신 전략이 필요합니다.
- **FluoShop**: 다섯 서비스 토폴로지는 고급 패턴을 실습할 현실적인 환경을 제공합니다.
- **Abstraction**: fluo의 통일된 모델은 트랜스포트를 교체 가능한 드라이버로 다룹니다.
- **Patterns**: 요청에는 `@MessagePattern`, 이벤트에는 `@EventPattern`을 사용합니다.
- **Progression**: Part 0의 나머지 장들은 이 추상 아키텍처를 구체적인 트랜스포트 선택으로 바꿉니다.

이 요약은 의도적으로 아키텍처 중심입니다.

우리는 아직 어떤 것도 최적화하지 않았습니다.

다만 이후 장에서 그 최적화를 평가할 수 있도록 서비스 경계와 통신 방식을 먼저 고정했습니다.

그 순서가 맞습니다.

안정적인 서비스 지도가 없는 상태에서 인프라 세부 사항으로 먼저 들어가면 우발적인 복잡성만 커지기 쉽습니다.
