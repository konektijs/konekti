# dev reload architecture

Konekti supports two different development-time update paths, and they should not be confused with each other.

## the two supported paths

### 1. source-code changes use process restart

The starter application generated by `@konekti/cli` uses a scaffolded `dev` script based on Node watch mode and `tsx`.

That means the default starter experience for source edits is:

- Node detects a file change
- the process restarts
- runtime bootstraps again from scratch

This is a watch-driven restart model, not in-process hot module replacement.

### 2. config changes can use targeted in-process reload

`@konekti/config` exposes `createConfigReloader()` for explicit env-file reload.

`@konekti/runtime` can now consume that reloader in dev mode when the caller opts in with `watch: true`.

That path is intentionally narrow:

- it is limited to validated config snapshots
- it preserves the existing `ConfigService` reference instead of rebuilding the application shell
- it notifies runtime-managed reload participants after the new snapshot is applied
- it restores the previous snapshot if runtime-side reload handling fails

This is not a general-purpose code hot reload API.

## ownership boundaries

### `@konekti/cli`

Owns development runner orchestration for the starter workflow.

- starter `pnpm dev` behavior
- process restart for source changes
- future runner-level decisions such as alternate builders

### `@konekti/config`

Owns config snapshot production.

- env file resolution
- config merge and validation
- file watching for config sources
- explicit reload subscriptions and error delivery

### `@konekti/runtime`

Owns runtime reaction to accepted config reloads.

- keeping `ConfigService` identity stable
- applying validated config snapshots in dev mode
- notifying runtime-managed reload participants
- restoring the previous snapshot when runtime-side reload handling fails

## current guarantees

- Konekti does not ship a public `@konekti/hot-reload` package.
- Generated starter apps still use process restart for code changes.
- Runtime config reload is opt-in and dev-only.
- Invalid watched config does not replace the last known good runtime snapshot.
- Closing the application closes the runtime-owned config watch path.

## why there is no `@konekti/hot-reload` package yet

Konekti does not currently have a broad, stable reload abstraction that justifies a public package boundary.

Today the stable split is:

- CLI owns restart orchestration
- config owns snapshot production
- runtime owns reload application semantics

If a broader shared contract emerges later, package extraction can happen from that proven boundary instead of from a premature package name.
