# 트랜잭션 (transactions)

<p><a href="./transactions.md"><kbd>English</kbd></a> <strong><kbd>한국어</kbd></strong></p>


이 가이드는 런타임 및 공식 ORM 통합 전반에 걸친 현재 트랜잭션 의미론을 설명합니다.

함께 보기:

- `../../packages/prisma/README.ko.md`
- `../../packages/drizzle/README.ko.md`
- `./http-runtime.ko.md`

## 기본 규칙

- 권장되는 트랜잭션 경계는 서비스 레이어(service layer)입니다.
- 자동 요청 트랜잭션은 인터셉터 통합을 통해서만 선택적으로 사용 가능합니다.
- 전역적인 암시적 트랜잭션은 기본값이 아닙니다.

## 리포지토리 및 서비스 경계

- 서비스는 `transaction(...)`을 시작할 수 있습니다.
- 리포지토리는 트랜잭션을 시작하지 않습니다.
- 리포지토리는 통합 패키지에서 노출하는 현재 트랜잭션 인식 핸들을 확인하여 동작합니다.
- 생성된 리포지토리들도 동일한 규칙을 따릅니다.

## 전파 방식 (propagation stance)

현재 기본값은 `required` 전파 방식에 가장 가깝습니다:

- 트랜잭션이 없으면 새로 시작합니다.
- 이미 활성화된 트랜잭션이 있으면 참여합니다.

현재 기본 규약에 포함되지 않는 사항:

- `requires_new`
- 세이브포인트 (savepoints)
- 분산 트랜잭션

## 요청 범위 트랜잭션 경로 (request-scoped transaction path)

- 요청 범위의 트랜잭션 동작은 인터셉터가 소유합니다.
- 가드(guard) 성공 후 인터셉터 경계에서 시작됩니다.
- 트랜잭션 핸들 확인은 패키지/통합 라이브러리에 따라 다릅니다.
- 인증 거부는 롤백 경로가 아니라 트랜잭션이 아예 시작되지 않는 경로입니다.

## 커밋 및 롤백 타이밍

- 커밋은 래핑된 요청 경로가 성공한 후에 발생합니다.
- 롤백은 컨트롤러/인터셉터 실패, 바인딩/검증 실패, 그리고 커밋 전 요청 중단을 포함합니다.
- 중단된 요청이 트랜잭션을 열어둔 상태로 두어서는 안 됩니다.

## 스트리밍 및 긴 응답 (streaming and long-lived responses)

- 스트리밍, 파일, 또는 긴 응답에 대해서는 자동 요청 트랜잭션이 기본값이 아닙니다.
- 이러한 경우 명시적인 비트랜잭션 처리가 선호됩니다.

## 테스트 시 시사점

- 유닛 테스트는 트랜잭션 인터셉터를 필요로 하지 않아야 합니다.
- 통합 테스트는 전파, 참여, 롤백, 그리고 어댑터 상호작용 경로를 명시적으로 다루어야 합니다.
