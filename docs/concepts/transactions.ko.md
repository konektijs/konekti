# 트랜잭션 (transactions)

<p><strong><kbd>English</kbd></strong> <a href="./transactions.ko.md"><kbd>한국어</kbd></a></p>

이 가이드는 런타임 및 공식 ORM 통합(특히 `@konekti/prisma`, `@konekti/drizzle`)에서 사용되는 트랜잭션 의미론을 설명합니다.

### 관련 문서

- `../../packages/prisma/README.md`
- `../../packages/drizzle/README.md`
- `./http-runtime.ko.md`

## 표준 원칙

- **서비스 레이어 소유권**: 서비스 레이어는 트랜잭션 경계를 정의하기 위한 권장 위치입니다.
- **선택적 트랜잭션**: 자동 요청 레벨 트랜잭션은 인터셉터 통합을 통해서만 사용할 수 있습니다.
- **암시보다 명시**: 전역적이고 암시적인 트랜잭션은 기본적으로 활성화되지 않습니다.

## 리포지토리 및 서비스 경계

- **서비스**: 트랜잭션을 열고 관리할 책임이 있습니다.
- **리포지토리**: 트랜잭션을 시작해서는 안 됩니다. 통합 패키지에서 제공하는 트랜잭션 인식 핸들을 해결하여 사용합니다.
- **생성된 코드**: 동일한 경계 규칙을 따릅니다.

## 트랜잭션 전파

기본 동작은 "REQUIRED" 전파 모델을 따릅니다:

- 기존 트랜잭션이 없으면 새 트랜잭션을 시작합니다.
- 이미 활성화된 트랜잭션이 있으면 해당 트랜잭션에 참여합니다.

다음 전파 유형은 현재 표준 기본값으로 지원되지 않습니다:
- `REQUIRES_NEW`
- 세이브포인트(Savepoints)
- 분산 트랜잭션

## 요청 스코프 트랜잭션

- 인터셉터를 통해 관리됩니다.
- 가드가 성공적으로 실행된 후 인터셉터 경계에서 시작됩니다.
- 트랜잭션 핸들은 패키지별 방식으로 해결됩니다.
- 인증 실패는 트랜잭션 시작을 방지하여 불필요한 롤백 로직을 피합니다.
- Prisma 통합(`requestTransaction`)은 abort-aware 실행을 사용하며, 드라이버가 transaction `signal` 옵션을 거부하면 `signal` 없이 1회 재시도합니다.
- Drizzle 통합(`requestTransaction`)은 abort-aware이며, wrapped handle에 `transaction` runner가 없으면(strict 모드가 아닐 때) abort-aware 직접 실행으로 폴백합니다.

## 커밋 및 롤백

- **커밋**: 래핑된 요청 경로가 성공적으로 종료된 후 발생합니다.
- **롤백**: 컨트롤러 또는 인터셉터 실패, 검증 에러, 또는 완료 전 요청 중단에 의해 트리거됩니다.
- **정리**: 중단된 요청이 고립된 트랜잭션을 남겨두어서는 안 됩니다.
- **셧다운 안전성**: Prisma/Drizzle 통합은 애플리케이션 종료 시 활성 request transaction을 abort하고 settle 완료를 기다린 뒤 disconnect/dispose 정리를 진행합니다.

## 스트리밍 및 장기 실행 요청

- 스트리밍 또는 장기 실행(long-lived) 응답의 경우 자동 요청 레벨 트랜잭션은 권장되지 않습니다.
- 자원 고갈을 방지하기 위해 이러한 시나리오에서는 명시적인 비트랜잭션 처리를 사용하세요.

## 테스트

- **유닛 테스트**: 트랜잭션 인터셉터 없이 실행되도록 설계되어야 합니다.
- **통합 테스트**: 전파, 참여 및 롤백 동작을 명시적으로 검증해야 합니다.
